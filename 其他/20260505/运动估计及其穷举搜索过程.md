### (a) What is motion estimation in video compression? Why is it needed?

**What is Motion Estimation?**
Motion estimation is the process of determining movement inside a video sequence by tracking the motion of objects or pixels from one frame to another. In block-based video coding (like MPEG or H.264), the current frame is divided into blocks (usually called macroblocks). Motion estimation finds a block in a previously encoded reference frame that closely matches the current block being encoded. The relative displacement between the current block and the matched block in the reference frame is represented as a **motion vector** $(dx, dy)$.

**Why is it needed?**
Motion estimation is essential because it is the primary method used to exploit **temporal redundancy** in video compression. 
*   In a typical video, adjacent frames are highly similar; only small parts of the scene (like moving objects or camera panning) change from one frame to the next. 
*   Instead of encoding every single pixel of the new frame from scratch (which takes an enormous amount of data), the encoder only needs to transmit the motion vectors and the small difference (the prediction error or residual) between the predicted block and the actual block. 
*   This drastically reduces the amount of data required to represent the video, resulting in highly efficient compression.

### (b) Procedure of Motion Estimation for a Macroblock using Exhaustive Search

The **Exhaustive Search** (also known as Full Search) is the most straightforward, yet computationally intensive, block-matching algorithm. It evaluates *every* possible candidate block within a predefined search area to find the absolute best match.

Here is the step-by-step procedure:

1.  **Define the Macroblock:** 
    Divide the current frame into non-overlapping macroblocks of size $N \times N$ pixels (commonly $16 \times 16$). Let the top-left coordinate of the current macroblock in the current frame be $(x, y)$.

2.  **Define the Search Window:** 
    In the reference frame (the previously encoded frame), establish a search window around the spatial coordinate $(x, y)$. If the maximum allowed motion displacement is $\pm p$ pixels in both horizontal and vertical directions, the search window will have a size of $(2p + N) \times (2p + N)$ pixels.

3.  **Evaluate Every Candidate Block:** 
    Slide a virtual $N \times N$ block across every single pixel position within the search window in the reference frame. For a search parameter $p$, there are $(2p + 1)^2$ possible candidate blocks.

4.  **Calculate the Matching Cost:** 
    For each candidate block at displacement $(dx, dy)$, compute a cost function to determine how similar it is to the current macroblock. The most common metric is the **Sum of Absolute Differences (SAD)**:
  
    $$SAD(dx, dy) = \sum_{i=0}^{N-1} \sum_{j=0}^{N-1} |C(x+i, y+j) - R(x+dx+i, y+dy+j)|$$
  
    Where:
    *   $C(x+i, y+j)$ is the pixel value in the **C**urrent frame.
    *   $R(x+dx+i, y+dy+j)$ is the pixel value in the **R**eference frame.
    *   $dx \in [-p, p]$ and $dy \in [-p, p]$ represent the search displacement.

5.  **Determine the Motion Vector:** 
    Compare the SAD values for all $(2p + 1)^2$ candidates. The displacement $(dx, dy)$ that yields the **minimum SAD** is chosen as the final motion vector for that macroblock.

<details>
<summary><b>Click to expand: Alternative Cost Functions and Characteristics of Exhaustive Search</b></summary>

*   **Alternative Cost Functions:** While SAD is preferred for its computational simplicity (it only requires addition and absolute values), encoders can also use **Mean Absolute Difference (MAD)** or **Sum of Squared Differences (SSD)**:
    $$SSD(dx, dy) = \sum_{i=0}^{N-1} \sum_{j=0}^{N-1} (C(x+i, y+j) - R(x+dx+i, y+dy+j))^2$$
*   **Advantage:** Exhaustive search guarantees finding the global minimum within the search window, providing the highest possible peak signal-to-noise ratio (PSNR) for block matching.
*   **Disadvantage:** It is extremely computationally expensive. For a search range of $p=15$, the algorithm must perform $31 \times 31 = 961$ block comparisons *per macroblock*. Therefore, modern video codecs typically use fast search algorithms (like Diamond Search or Three-Step Search) that evaluate far fewer points at the cost of a slightly less optimal match.
