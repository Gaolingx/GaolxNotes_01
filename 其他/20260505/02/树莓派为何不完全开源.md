# 树莓派为何不完全开源

### 🧑‍💻 User

为什么树莓派并不是完全开源的，为什么只开源了设备树，而bootrom/闭源固件（start.elf、bootcode.bin）等没有？

---

### 🤖 Assistant

树莓派（Raspberry Pi）虽然在推动开源社区和创客文化方面做出了巨大贡献，但它确实**不是一个完全开源的硬件平台**。其核心原因可以归结为：**树莓派使用的是博通（Broadcom）的商业化系统级芯片（SoC），而博通对其核心底层技术实施了严格的知识产权（IP）保护。**

以下是关于为什么设备树开源，而 `bootrom`、`bootcode.bin`、`start.elf` 等固件闭源的深入解析：

### 1. 核心矛盾：博通（Broadcom）的商业模式与知识产权

树莓派的历代主芯片（如 BCM2835、BCM2711 等）都来自博通。博通是一家传统的芯片巨头，其主要客户是机顶盒、路由器和通信设备的制造商，而不是开源社区。

*   **专有架构 (VideoCore)：** 树莓派 SoC 的核心其实是博通的 **VideoCore GPU**，而 ARM CPU 只是作为它的一个“协处理器”存在（至少在树莓派 1-3 代是典型的这种架构）。`start.elf` 等固件是运行在 VideoCore 上的操作系统（名为 VCOS）。
*   **商业机密：** 开放这些底层固件的源代码，就等于向竞争对手公开了 VideoCore GPU 的硬件工作原理、寄存器定义和内部架构。这触及了博通的核心商业机密。
*   **保密协议 (NDA)：** 树莓派基金会只是博通的客户（尽管关系密切，树莓派创始人 Eben Upton 曾是博通工程师）。基金会在获取这些芯片及其底层代码时，签署了严格的保密协议，**在法律上没有权利**开源这些固件。

<summary><b>展开查看：第三方 IP 与授权限制</b></summary>
除了博通自己的技术，SoC 内部还包含了许多从第三方购买的 IP（例如硬件视频解码器、HDMI 控制器、USB 控制器等）。
博通在购买这些 IP 时，授权协议明确禁止公开相关的驱动和固件代码。也就是说，即使博通自己想开源，他们也必须先去获得所有第三方 IP 供应商的许可，这在商业上是极其昂贵且不现实的。
</details>

### 2. 为什么 `bootcode.bin` 和 `start.elf` 必须闭源？

要理解这一点，需要了解树莓派奇特的**引导过程（Boot Process）**：

在传统的 PC 或 ARM 手机中，上电后是 CPU 先启动，然后去初始化显卡和其他外设。但在树莓派（特别是 Pi 4 之前的型号）上，逻辑是完全相反的：

1.  **`bootrom`（掩膜 ROM）：** 烧录在芯片硅片内部，无法更改。它负责唤醒 **GPU (VideoCore)**。
2.  **`bootcode.bin`：** 由 GPU 运行的二级引导程序。它初始化基本的硬件（如 SDRAM）。
3.  **`start.elf`：** 这是重头戏。它是一个庞大的闭源操作系统，运行在 GPU 上。它负责分配内存、配置系统时钟、读取 `config.txt`，最后将 Linux 内核加载到内存中，并**唤醒 ARM CPU** 让其开始执行 Linux。

因为这一整套引导流程完全依赖于博通专有的 VideoCore 架构，且包含了大量的底层硬件操作和第三方授权代码，所以树莓派基金会只能以**二进制二进制 blob（Binary Blob）**的形式提供它们。

### 3. 为什么只有“设备树（Device Tree）”和内核开源了？

这就是**法律约束（GPL 协议）**和**技术规范**的作用了。

*   **Linux 内核的 GPL 限制：** 树莓派运行的主要是 Linux 操作系统。Linux 内核采用了 GNU 通用公共许可证（GPL）。GPL 规定，任何修改和分发 Linux 内核代码的行为，都必须开源这些修改。因此，树莓派基金会必须开源他们为 ARM CPU 编写的 Linux 驱动程序。
*   **设备树（Device Tree）的作用：** 在 ARM Linux 的生态中，设备树（`.dtb` 文件）的作用是向通用的 Linux 内核“描述”当前的硬件长什么样（比如 GPIO 挂在哪、I2C 地址是多少）。它只是一个**数据结构**，不包含博通的专有执行代码。开源设备树是让 Linux 能够在其硬件上运行的行业标准做法，不涉及泄露芯片底层的机密。

简单来说：**在 ARM CPU 启动之后的世界（Linux 内核、驱动、应用层），GPL 强制要求开源；而在 ARM CPU 启动之前的世界（GPU 固件、Bootrom），商业法则和 NDA 强制要求闭源。**

### 4. 现状与未来的改变

虽然核心 SoC 固件依然闭源，但树莓派基金会一直在尽力推动开源的边界：

<li><b>开源图形驱动 (VC4/V3D)：</b> 过去，不仅启动固件闭源，连 3D 图形加速也要依赖闭源的二进制库。后来，博通和树莓派雇佣了开发者（如 Eric Anholt），编写了完全符合开源标准的 Mesa 3D 驱动程序。现在大部分的图形渲染已经实现了开源化，尽管底层的系统管理（电源、时钟）依然由闭源的 <code>start.elf</code> 掌控。</li>
<li><b>Raspberry Pi Pico (RP2040 / RP2350)：</b> 这是树莓派基金会的转折点。由于 RP2040 及其后续芯片是<b>树莓派基金会完全自主设计</b>的硅片（不再依赖博通），因此它的数据手册极其详尽，甚至连 Bootrom 的源代码都对外公开了。这证明了：只要 IP 掌握在树莓派自己手里，他们是非常愿意完全开源的。</li>
</ul>
</details>

### 总结

树莓派的“部分开源”是一种**商业妥协**。为了实现“极低的售价”和“强大的多媒体性能”，他们选择了博通现成的商业芯片。代价就是必须接受博通的闭源固件（`start.elf` 等）。而开源的设备树和 Linux 内核，则是为了遵守软件开源协议并融入现有的 Linux 生态。

### 🧑‍💻 User

既然bootrom不开源，也没有采用ampere那套标准的uefi+acpi，为什么它却可以运行绝大多数发行版arm linux？

---

### 🤖 Assistant

这是一个非常专业且切中要害的问题。在 ARM 服务器领域（比如 Ampere 等），确实是以 UEFI + ACPI 作为行业标准（SBSA/SBBR 规范），以此来实现像 PC 那样“一个系统镜像随处安装”。

树莓派虽然底层是闭源的“黑盒”，且默认不支持硬件级别的 UEFI + ACPI，但它之所以能运行绝大多数主流 ARM Linux 发行版（如 Ubuntu, Debian, Fedora, Arch 等），主要依赖于以下几个核心机制：

### 1. 采用嵌入式 ARM 的标准：设备树 (Device Tree)

在 ARM 生态中，除了服务器级别的 ACPI，还有另一套广泛应用于嵌入式和单板计算机（SBC）的标准：**Device Tree (DTB)**。

*   **ACPI 的做法：** 固件（UEFI）不仅初始化硬件，还提供一系列运行时接口，操作系统通过调用这些接口来发现硬件。
*   **设备树的做法：** 将硬件的拓扑结构（有多少内存、有哪些外设、中断号是多少）写成一个静态的配置文件（`.dtb` 文件）。

树莓派的闭源固件 `start.elf` 扮演了非常关键的“黑盒 BIOS”角色。在它将控制权交给 ARM CPU 之前，它会读取用户配置的 `config.txt`，动态修改和组装设备树，然后**将这个 `.dtb` 文件所在的内存地址传递给 Linux 内核**。Linux 内核读取设备树后，就知道该如何驱动这些硬件了。这是嵌入式 Linux 最标准的启动方式。

### 2. 主线内核 (Mainline Kernel) 的强大支持

绝大多数发行版（如 Ubuntu 或 Fedora）并不会专门为树莓派去魔改内核，它们使用的是**主线 Linux 内核（Mainline Kernel）**。

树莓派基金会以及社区开发者花费了数年时间，将树莓派底层硬件（如 BCM283x/2711 SoC）的驱动程序**全部提交并合并到了 Linus Torvalds 维护的 Linux 主线代码库中**。

这意味着，当 Ubuntu 或 Fedora 编译他们官方的 ARM64 内核时，树莓派所需的时钟、GPIO、USB、网卡、甚至是开源的显示驱动（VC4）就已经默认包含在内了。只要闭源固件能把内核拉起来并喂给它正确的设备树，这些标准的发行版内核就能顺畅运行。

### 3. “桥接”技术：利用 U-Boot 模拟标准启动环境

闭源的 `start.elf` 默认只能直接引导 Linux 内核。但是，像 Fedora、openSUSE 等发行版高度依赖通用的启动管理器（如 GRUB）和 EFI 环境。为了解决这个问题，社区和发行版广泛采用了一种“桥接”方案：

<details>
<summary><b>展开查看：U-Boot 和 EFI 桥接技术</b></summary>
<ul>
<li><b>启动链的转变：</b> 闭源固件 <code>start.elf</code> 不再直接加载 Linux 内核，而是去加载一个开源的引导加载程序：<b>U-Boot</b>。</li>
<li><b>U-Boot 提供 EFI 接口：</b> U-Boot 在 ARM 生态中非常强大，它内部实现了一个 <b>EFI Payload (EFI 子系统)</b>。U-Boot 运行后，会在内存中向发行版暴露一套兼容 UEFI 规范的接口。</li>
<li><b>标准 GRUB 介入：</b> 发行版的 <code>grubaa64.efi</code> 被 U-Boot 唤醒。从 GRUB 和操作系统的视角来看，它以为自己正运行在一个标准的 UEFI 主板上，于是顺利地加载通用的 ARM Linux 镜像。</li>
</ul>
</details>

### 4. 社区甚至为树莓派做出了真正的 UEFI (EDK2)

虽然博通的 Bootrom 和 `start.elf` 都不支持标准 UEFI，但开源社区（如 Pi Firmware Task Force）利用逆向和底层开发，成功为树莓派移植了 **TianoCore EDK2**（PC和服务器上使用的标准 UEFI 固件）。

你可以将这个开源的 UEFI 固件放在 SD 卡里。闭源的 `start.elf` 加载这个 UEFI 固件后，树莓派就真的变成了一台符合服务器标准的机器。你可以像给电脑装系统一样，插上装有 Windows on ARM 或原生 Ubuntu 安装镜像的 U 盘，进入经典的 BIOS 界面进行安装，甚至可以使用 ACPI 而不是设备树来引导系统。

### 总结

树莓派虽然底层启动过程是封闭和非标准的，但它非常聪明地**在移交控制权的那一刻，做到了与开源标准的完美对接**：

1.  **底层黑盒（闭源）：** Bootrom -> `bootcode.bin` -> `start.elf` 负责把复杂的硬件初始化搞定。
2.  **交接点（标准）：** `start.elf` 按照 Linux 的规范，把内核放进内存，准备好标准的 Device Tree，或者加载 U-Boot/EDK2 提供 UEFI 接口。
3.  **上层系统（开源主线）：** 主流发行版的 Linux 内核因为已经包含了树莓派的开源驱动，顺理成章地接管并正常运行。
