# 安卓手机内核为何老旧

### 🧑‍💻 User

对比发行版aarch64 linux，为什么很多新的arm架构soc安卓手机的linux kernel版本非常旧？

---

### 🤖 Assistant

这是一个非常经典且触及安卓生态核心痛点的问题。

对比标准的 AArch64 Linux 发行版（如 Ubuntu、Debian、Arch Linux ARM），最新的安卓手机虽然使用了最先进的 ARM SoC（如骁龙 8 Gen 3、天玑 9300），但其底层的 Linux Kernel 版本往往停留在 5.15、5.10 甚至更老的版本，且在手机的整个生命周期内**几乎不会进行内核的大版本升级**。

导致这一现象的根本原因在于**极其漫长且封闭的内核供应链**、**高度定制化的专有硬件驱动**以及**商业利益的考量**。

以下是详细的原因剖析：

### 1. 漫长且碎片化的“安卓内核供应链”

标准的 Linux 发行版通常直接基于 Mainline（主线）内核进行少量的补丁修改。而安卓手机的内核从 Linux 主流版本到最终搭载在手机上，需要经历一个极其漫长且层层加码的“套娃”过程：

*   **第一步：LTS (Long Term Support) 内核**：Linux 社区发布一个长期支持版内核（例如 5.15）。
*   **第二步：ACK (Android Common Kernel)**：Google 拿到这个 LTS 内核，加入安卓系统特有的核心功能（如 Binder、Ashmem 等），形成安卓通用内核。
*   **第三步：BSP (Board Support Package) 内核**：SoC 厂商（高通、联发科）拿到 ACK，针对其最新的芯片架构加入大量的硬件驱动和电源管理代码。
*   **第四步：OEM 设备内核**：手机厂商（三星、小米、OPPO 等）拿到 SoC 厂商的 BSP，再加入自家定制的硬件驱动（屏幕、摄像头传感器、指纹识别、快充芯片等）。

**时间滞后效应**：当高通或联发科开始研发一款新的 SoC 时，他们通常会选择当时最稳定、最新的 Android LTS 内核作为基础。由于芯片研发周期长达 1-3 年，当这款芯片最终量产并搭载在手机上发布时，当初选择的“新”内核早就变成了“老”内核。

### 2. 庞大的“树外驱动”（Out-of-Tree Drivers）与专有闭源硬件

这是安卓内核无法像常规 Linux 发行版那样随时升级的**最大技术障碍**。

*   **常规 AArch64 Linux**：在服务器或开源硬件上，硬件厂商通常会将驱动代码提交到 Linux Mainline（主线）中（即“树内驱动”）。只要升级内核，驱动通常会自动兼容。
*   **移动端 SoC**：智能手机 SoC 内部集成了极其复杂的专有模块（GPU、NPU、ISP、基带 Modem、DSP）。为了保密商业核心技术，高通和联发科等厂商**拒绝将这些驱动的源代码提交到 Linux 主线**。
*   他们采用的做法是将驱动写成只适配某一个特定内核版本的模块（Out-of-Tree）。如果强行升级 Linux 内核版本，由于内核内部 API/ABI 的改变，成百上千万行的专有驱动代码将直接报错无法编译，导致手机的相机、基带、GPU 直接瘫痪。

### 3. 商业考量：高昂的适配成本与极低的 ROI

对于手机厂商（OEM）来说，升级 Linux 大版本内核是一项**成本极高且吃力不讨好**的工作：

*   **研发成本极高**：要升级内核，手机厂商需要等待 SoC 厂商提供新内核的驱动支持。但 SoC 厂商更愿意把精力放在卖新芯片上，而不是为老芯片更新底层代码。
*   **用户感知度为零**：普通消费者买手机看重的是 UI 流畅度、拍照画质、续航和游戏帧率。将内核从 5.10 升级到 6.6，绝大多数用户根本感知不到任何区别。
*   **稳定性风险**：老内核经过了成千上万次测试，已经非常稳定（“只要代码能跑，就不要去动它”）。更换新内核意味着需要重新进行极其严苛的 GMS 认证、运营商认证以及各种 Bug 修复。

<details>
<summary><b>点击展开：Google 的破局之道（GKI 机制）</b></summary>

为了解决安卓内核碎片化和无法升级的问题，Google 从 Android 11/12 开始强制推行了 **GKI (Generic Kernel Image，通用内核映像)** 架构。

1.  **内核与驱动分离**：Google 强制要求 SoC 厂商和手机厂商不能再直接修改核心内核代码。所有的硬件驱动必须以**可加载内核模块（Kernel Modules，`.ko` 文件）**的形式存在。
2.  **统一的 API/ABI 边界**：Google 维护一个标准的 `boot.img`（包含纯净的 Linux 内核），并保证这个内核对外的接口（ABI）在同一个内核大版本内绝对稳定。
3.  **独立升级**：通过这种设计，理论上 Google 可以通过 Play 商店单独下发更新来修补核心 Linux 内核的安全漏洞，而不会破坏高通或小米编写的底层驱动。

尽管有了 GKI，**跨大版本升级（例如从 5.15 升级到 6.1）依然极其困难**，因为跨大版本的内核 ABI 必定会改变。因此，新手机出厂时仍然会绑定某一个特定的 LTS 版本直到其退役。
</details>

### 总结

标准的 AArch64 Linux 发行版追求的是**通用性、开源和紧跟社区步伐**，硬件驱动大多在主线内；
而基于 ARM SoC 的安卓手机，其本质是一个**高度定制化、高度商业化且包含大量闭源专有硬件的嵌入式系统**。老旧的 Linux 内核是 SoC 研发周期、闭源驱动生态以及商业利益博弈下必然的产物。

### 🧑‍💻 User

为什么ampere altra这种arm cpu可以非常自由的使用最新发行版linux甚至win on arm？

---

### 🤖 Assistant

这是一个非常深刻且直击 ARM 生态核心差异的问题。

虽然 Ampere Altra 和高通骁龙、联发科天玑一样，底层指令集都是 ARM 架构（AArch64），但它们的**系统架构设计规范**、**固件标准**以及**商业驱动模式**截然不同。

简而言之：**Ampere Altra 是按照传统的 x86 PC/服务器标准来设计的 ARM 芯片，而手机 SoC 是高度封闭的定制化嵌入式系统。**

以下是 Ampere Altra 能够自由安装最新 Linux 发行版甚至 Windows on ARM 的核心原因：

### 1. 遵循统一的服务器/PC 硬件标准 (SBSA & SBBR)

这是最根本的技术差异。为了打入原本被 x86 垄断的服务器市场，ARM 联合各大厂商制定了严格的硬件和启动规范：

*   **SBSA (Server Base System Architecture)**：规定了服务器级 ARM 芯片必须具备的基础硬件特性，例如标准的内存映射、中断控制器（GIC）、串口等。
*   **SBBR (Server Base Boot Requirements)**：规定了系统必须如何启动。Ampere Altra 严格遵循这一规范，这意味着它使用了与 x86 电脑完全一样的固件接口：**`UEFI` 和 `ACPI`**。

**对比手机 SoC：**
手机 SoC 采用的是 `Device Tree`（设备树）和私有 Bootloader（如 `U-Boot`、`lk`）。设备树将硬件信息硬编码在一个特定的文件中，并与特定版本的内核高度绑定。内核稍有改动，设备树可能就会失效。而 `ACPI` 则是由主板固件动态提供硬件信息，对操作系统做了一层完美的抽象隔离。

### 2. 标准化的设备发现机制 (PCIe 体系)

在 Ampere Altra 这样的服务器芯片上，除了核心的 CPU 和内存控制器，几乎所有的外设扩展（网卡、显卡、NVMe 硬盘）都是通过 **`PCIe` (Peripheral Component Interconnect Express)** 总线连接的。

*   `PCIe` 具有**标准的硬件枚举（发现）机制**。最新版的标准 Linux 内核自带通用的 PCIe 驱动。当 Linux 或 Windows 在 Ampere Altra 上启动时，它会自动扫描 PCIe 总线，发现插入的硬件，然后加载对应的标准驱动。
*   **对比手机 SoC：** 手机 SoC 内部集成了大量专有模块（NPU、ISP、Modem），这些模块通过各家厂商魔改的片内总线互联，没有统一的枚举标准。操作系统根本不知道 SoC 里面藏了什么硬件，必须依靠上千万行的专有驱动代码去手动唤醒和通信。

### 3. 积极拥抱 Linux Mainline（主线）

Ampere 等服务器芯片厂商的商业模式要求他们必须支持标准的操作系统生态（如 RHEL, Ubuntu, SUSE）。

*   **“树内”（In-Tree）驱动**：Ampere 的软件团队会积极地将他们芯片的核心驱动代码（如 CPU 电源管理、性能监控、中断控制器等）提交给 Linus Torvalds 维护的 **Linux Mainline（主线内核）**。
*   一旦代码被合并入主线，这意味着**任何基于该主线版本打包的最新 Linux 发行版（如 Ubuntu 24.04，搭载 6.8 内核）天然就支持这颗芯片**。用户插上 U 盘，按照 x86 的习惯一路“下一步”就能安装成功。

<details>
<summary><b>点击展开：为什么它能运行 Windows on ARM？</b></summary>

Ampere Altra 能运行 Windows on ARM，核心密码在于对 **`UEFI` 和 `ACPI`** 的完美支持。

微软的 Windows 操作系统在设计之初，就深度依赖 `ACPI`（高级配置与电源接口）表来获取硬件拓扑结构。Windows **不支持**开源界常用的 `Device Tree`。

由于 Ampere Altra 的主板固件（如 TianoCore EDK II）提供了标准的 `UEFI` 引导和完整的 `ACPI` 表，Windows 安装程序（Windows PE）在启动时，会像在 Intel/AMD 机器上一样：
1. 通过 `UEFI` 加载 Bootloader。
2. 读取固件提供的 `ACPI` 表，了解 CPU 核心数、内存分布和 PCIe 根节点。
3. 利用 Windows 自带的标准通用驱动（Inbox drivers）初始化基础硬件。

只要 Ampere 提供了少数必须的底座驱动（通常会打包成标准的 `.inf` 和 `.sys` 文件注入安装镜像），Windows 就能顺利安装并运行。
</details>

### 4. 商业模式与终端用户的诉求

*   **服务器/桌面用户**：购买 Ampere 裸机服务器或开发板的用户，期望的是买到硬件后，能够**自己决定**安装什么操作系统、什么版本的内核，以便运行特定的软件容器（如 Docker, Kubernetes）。如果芯片做不到开箱即用，根本卖不出去。
*   **手机用户**：普通消费者购买的是“一台手机”（包含完整的软硬件生态），而不是“一块 SoC”。高通和手机厂商没有任何商业动力去花费巨资支持用户自由刷入最新的原生 Linux。

### 总结比较

我们可以用一个简单的比喻：

*   **Ampere Altra (标准化 ARM)** 就像是用**标准化乐高积木**拼成的底盘。任何标准的车壳（Linux 发行版、Windows）只要尺寸对，就能直接扣上去，因为接口是统一的。
*   **手机 ARM SoC** 就像是**一体成型的手办**。它的硬件和底层软件是熔铸在一起的，你想换一件衣服（升级主线内核），必须把原有的外壳打碎重新开模（重写成百上千个闭源驱动），成本极其高昂。
