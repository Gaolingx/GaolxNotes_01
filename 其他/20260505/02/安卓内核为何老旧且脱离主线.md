# 安卓内核为何老旧且脱离主线

### 🧑‍💻 User

为什么安卓手机用的linux内核都是bsp的一部分，而且版本很旧，和主线aarch64 linux有什么区别？

---

### 🤖 Assistant

这是一个非常深刻且直击安卓生态痛点的问题。你观察到的现象非常准确：长期以来，安卓手机的 Linux 内核确实是作为 SoC 厂商（如高通、联发科）BSP（Board Support Package，板级支持包）的一部分交付的，而且版本往往落后于主线（Mainline）数年。

要理解这个问题，我们需要从**安卓的供应链生态**以及**主线 Linux 的开发模式**两个维度来看。

以下是针对你的问题的详细解答：

### 核心区别：安卓 BSP 内核 vs 主线 aarch64 Linux

安卓 BSP 内核和主线 `aarch64` Linux 的核心区别在于**驱动的集成方式**、**定制化程度**以及**闭源组件的妥协**。

| 维度 | 安卓 BSP 内核 (Vendor Kernel) | 主线 aarch64 Linux (Mainline) |
| :--- | :--- | :--- |
| **驱动代码位置** | 大量“树外”（Out-of-tree）代码，直接硬编码在内核源码中。 | 所有驱动必须经过严格审查，合并到“树内”（In-tree）。 |
| **代码质量与规范** | 以“能跑通硬件、快速上市”为目的，往往包含大量丑陋的 Hack 代码。 | 追求极高的代码质量、通用性和抽象层设计。 |
| **专有硬件支持** | 包含大量未开源的私有 API（用于配合用户态的闭源 GPU、NPU、相机 ISP 驱动）。 | 拒绝闭源 API 妥协，要求硬件厂商提供开源驱动方案（如 Freedreno）。 |
| **升级能力** | 几乎为零。内核与特定硬件驱动强绑定，升级内核会导致驱动全部失效。 | 极强。只要硬件代码在主线中，就可以随时无痛升级到最新的内核版本。 |

### 为什么版本总是那么旧？

这主要归咎于传统安卓内核漫长且臃肿的**派生链条（Forking Chain）**。一台安卓手机的内核从 Linux 主线到最终搭载在手机上，需要经历至少 4 个阶段。

<details>
<summary><b>点击展开：了解安卓内核漫长的“四级派生”生命周期</b></summary>

1.  **Linux LTS（长期支持版）：** Linus Torvalds 发布主线版本，内核社区选定一个 LTS 版本（例如 Linux 4.19 或 5.10）。
2.  **ACK (Android Common Kernel)：** Google 将 LTS 版本拉取过来，加入安卓专用的补丁（如之前的 Binder、Ashmem，现在主要是通用框架），形成 ACK。
3.  **SoC BSP Kernel（芯片厂商内核）：** 高通或联发科从 Google 拉取 ACK，加入海量的、针对其新一代 SoC（如骁龙 8 Gen 2）的驱动代码、电源管理（EAS）、核心调度等，形成 BSP 内核。**这一步耗时最长，且代码改动最大。**
4.  **Device Kernel（手机厂商内核）：** 小米、OPPO、三星等手机厂商拿到高通的 BSP 内核后，再加入自己手机特有的硬件驱动（特定的屏幕面板、触摸IC、马达、快充芯片等）。

**时间滞后效应：**
当芯片厂商（如高通）开始研发新一代 SoC 时，他们会选择当时最新的 LTS 内核（假设是 $T_0$ 时刻）。硬件研发和 BSP 编写需要 1-2 年。等芯片交付给手机厂商，手机厂商再开发半年。等手机最终上市（$T_0 + 2$ 年）时，这个内核版本在开源社区里已经是一两年前的“旧古董”了。而且为了稳定性，厂商绝对不会在手机生命周期内去跨大版本升级内核。
</details>

### 主线 Linux 为什么不能直接跑在安卓手机上？

理论上，`aarch64` 主线 Linux 是可以运行在 ARM 架构设备上的，但要在现代安卓手机上跑起来，面临巨大的阻碍：

1.  **硬件极度碎片化与不标准：** 
    不同于 x86 阵营有标准的 UEFI/ACPI 来描述硬件，ARM 嵌入式设备依赖设备树（Device Tree, DTS）。手机内部的硬件集成度极高且是非标准的。主线 Linux 缺乏这些具体手机型号的设备树和驱动。
2.  **闭源的商业壁垒：** 
    手机的卖点在于影像（ISP）、AI（NPU）和游戏（GPU）。高通、ARM（Mali）绝对不愿意开源这些核心组件的驱动，因为里面包含大量商业机密和核心算法。他们通常只提供一个简单的内核态驱动用于内存分配和中断处理，而将核心逻辑放在用户态的闭源二进制文件（Binary Blobs）中。主线 Linux 社区对这种做法极其反感，通常拒绝接收此类配合闭源驱动的内核代码。
3.  **功耗与性能调度：**
    手机对续航极其敏感。芯片厂商在 BSP 中加入了大量极其复杂的、针对特定芯片物理特性的电源管理和大小核调度机制。这些机制如果想合入主线，需要经过社区极其严苛的通用性审查，很多时候根本通不过。

<details>
<summary><b>点击展开：Linux 社区对 ARM 架构的历史态度</b></summary>
早年间，Linus Torvalds 曾对 ARM 架构在 Linux 内核中的代码混乱大发雷霆。因为每个 ARM 芯片厂商都在内核目录里塞满了自己专用的、互不兼容的垃圾代码（板级支持代码）。这促使了后来 Device Tree（设备树）在 ARM Linux 中的强制普及。尽管如此，手机 SoC 依然是定制化重灾区，导致其难以完全回归主线。
</details>

### 时代的转折：Google 的 GKI（通用内核镜像）项目

既然老旧的 BSP 内核导致了严重的安全隐患（无法及时打内核安全补丁）和生态碎片化，作为安卓的掌控者，Google 必然会出手干预。

从 **Android 11 / 12** 开始，Google 强制推行了 **GKI (Generic Kernel Image)** 项目，彻底改变了内核的交付方式：

1.  **统一核心：** Google 编译并锁定核心的 Linux 内核镜像（GKI）。手机厂商**绝对不允许**修改 GKI 的核心代码。
2.  **驱动模块化：** 芯片厂商（高通/联发科）和手机厂商（小米/OV）所有的 BSP 驱动，**必须**写成可加载内核模块（LKM, `.ko` 文件）。
3.  **稳定的 KMI：** Google 保证内核模块接口（KMI）的稳定。这意味着，底层的核心内核（GKI）可以直接由 Google 通过 Play 商店进行升级，而不需要高通或小米的参与，且升级后原有的 `.ko` 驱动依然可以正常工作。

**总结：**
过去，安卓手机的内核完全是 BSP 的“附庸”，是一个高度定制、难以升级的分支。现在，由于 GKI 的推行，安卓内核正在向主线标准靠拢，把 BSP 剥离到了模块层。未来的安卓设备，内核版本与主线 Linux 的差距将会越来越小，更新也会更加频繁。
