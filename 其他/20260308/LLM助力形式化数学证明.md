# LLM助力形式化数学证明

### 🧑‍💻 User

目前的llm对于辅助formalized mathematics proof方面有哪些意义和应用？

---

### 🤖 Assistant

大语言模型（LLMs）与形式化数学（Formalized Mathematics）的结合，是目前人工智能和基础科学领域最激动人心的前沿方向之一。

传统的形式化数学依赖于交互式定理证明器（ITP，如 `Lean`、`Coq`、`Isabelle`），要求人类专家将数学证明转化为极其严谨的计算机代码。这虽然保证了绝对的正确性，但学习曲线陡峭且过程极其繁琐。LLM 的引入正在彻底改变这一现状。

以下是 LLM 在辅助形式化数学证明方面的主要意义和应用场景：

### 一、 核心意义 (Significance)

1. **破解 LLM 幻觉，提供“绝对真理”的反馈环境**
   LLM 在自然语言推理中容易产生幻觉，而形式化数学系统（如 `Lean` 的内核）可以作为绝对客观的“裁判”。LLM 生成证明步骤，ITP 验证其正确性。这种结合为大模型提供了高质量的强化学习（RL）反馈信号，是通向更高阶逻辑推理甚至 AGI 的重要路径。
2. **大幅降低形式化数学的门槛**
   数学家通常使用自然语言和 LaTeX（如 $\forall x \in \mathbb{R}, x^2 \ge 0$）来思考和交流。LLM 充当了“自然语言”与“形式化代码”之间的翻译官，让不懂底层类型论的数学家也能参与到定理的验证中。
3. **加速数学知识的数字化与验证**
   例如陶哲轩（Terence Tao）等顶尖数学家正在积极推动将现代数学（如多项式 Freiman-Ruzsa 猜想）形式化。LLM 可以自动处理大量繁琐的引理验证，极大加速了类似 `Mathlib`（Lean 的数学库）的建设。

---

### 二、 核心应用场景 (Applications)

#### 1. 自动形式化 (Auto-formalization)
这是目前最具挑战也最有价值的应用。LLM 直接将非正式的数学文本（Informal Math）翻译为形式化语言（Formal Math）。
* **命题翻译**：将自然语言的定理描述转化为 `Lean` 或 `Isabelle` 中的定理声明。
* **证明翻译**：将教科书上的证明段落翻译为完整的策略（Tactic）证明脚本。

```lean
-- LLM 可以将 "如果 n 是偶数，则 n^2 也是偶数" 自动翻译为如下 Lean 4 代码：
theorem even_square (n : ℕ) (h : Even n) : Even (n^2) := by
  rcases h with ⟨k, rfl⟩
  use 2 * k^2
  ring
```

#### 2. 证明副驾驶 (Proof Copilot / Tactic Generation)
类似于程序员使用的 GitHub Copilot，在人类编写形式化证明时，LLM 可以根据当前的目标状态（Goal state）预测下一步或接下来的几步策略。
* **单步预测**：推荐最有可能解决当前子目标的 `tactic`（如 `simp`, `rw`, `apply`）。
* **证明补全**：如果剩下的证明逻辑相对机械，LLM 可以一次性生成数十行代码来关闭当前目标（close the goal）。

#### 3. 前提选择 (Premise Selection)
在大型数学库中（如 `Mathlib` 包含数十万个定理），人类很难记住所有引理的名字。当证明进行到某一步时，比如需要用到 $e^{i\pi} + 1 = 0$ 相关的性质，LLM 可以通过语义理解，从庞大的库中检索并推荐最相关的定理（Premises）供当前证明使用。这通常结合了 RAG（检索增强生成）技术。

#### 4. 证明修复与重构 (Proof Repair)
当底层数学库更新导致旧代码报错，或者人类写出的证明有微小类型不匹配时，LLM 可以读取错误信息（Error messages），并自动建议修复方案。

### 三、 代表性系统与进展

* **AlphaProof (Google DeepMind)**
  结合了预训练语言模型和基于强化学习的 AlphaZero 算法。它通过将非正式的数学问题自动形式化为 `Lean` 语言，然后在形式化环境中通过自我博弈（Self-play）寻找证明，成功达到了国际数学奥林匹克（IMO）银牌水平。
* **AlphaGeometry (Google DeepMind)**
  专门针对几何问题。它结合了神经语言模型（负责直觉和辅助线构造）和符号演绎引擎（负责严谨的逻辑推导），解决了大量 IMO 级别的几何难题。
* **Draft, Sketch, and Prove (DSP)**
  一种常见的 LLM 证明范式。先让 LLM 用自然语言写出证明草图（Draft/Sketch），然后再指导形式化系统一步步填补逻辑漏洞。
* **Lean Copilot / LlmStep**
  直接集成在 `Lean 4` 编辑器（VS Code）中的插件。当你在写代码时，它会在后台调用本地或云端的 LLM，实时给出下一步证明的建议。
* **InternLM-Math / DeepSeek-Prover**
  开源社区中专门针对数学和形式化语言（如 Lean）进行微调的强大开源模型，在策略生成和定理翻译上表现优异。

### 四、 当前面临的挑战

尽管前景广阔，但目前 LLM 在形式化数学中仍面临一些瓶颈：

1. **“非正式到正式”的对齐难题**：自然语言往往省略了大量显而易见的细节，而形式化系统要求 $100\%$ 的严谨。LLM 经常在填补这些隐含假设时出错。
2. **长程逻辑规划能力不足**：对于需要构造精巧的中间引理（Intermediate lemmas）或极其复杂的长证明，现有的 LLM 往往会在中途迷失方向，难以像顶尖数学家那样进行高瞻远瞩的规划。
3. **数据稀缺**：相比于 Python 或 C++ 数百亿行的训练数据，高质量的形式化数学代码（如 `Lean 4` 脚本）极其稀缺，这限制了模型的学习上限。

**总结而言**，LLM 正在将形式化数学从“少数逻辑学家的玩具”变成“广大数学家和计算机科学家的实用工具”。随着大模型推理能力（如基于强化学习的搜索）的提升，我们有望在未来几年看到 AI 协助人类验证甚至发现全新的重大数学定理。
