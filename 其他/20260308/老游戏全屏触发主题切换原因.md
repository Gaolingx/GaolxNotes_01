# 老游戏全屏触发主题切换原因

### 🧑‍💻 User

为什么很多老的游戏引擎全屏会触发windows主题切换（aero->basic），而现代的引擎基本不再出现了

---

### 🤖 Assistant

这是一个非常经典的Windows系统与图形渲染历史问题。很多经历过 Windows Vista 和 Windows 7 时代的玩家，对游戏启动时屏幕闪烁一下、任务栏从透明（Aero Glass）变成灰白色不透明（Windows Basic）的现象一定记忆犹新。

简单来说，这源于**早期游戏对显示器控制权的“独占”需求**与**Windows桌面窗口管理器（DWM）**之间的冲突。现代引擎不再出现这种情况，是因为底层的渲染架构和Windows系统本身都发生了翻天覆地的变化。

以下是具体的原理解析：

### 1. 过去的冲突：为什么老游戏会“杀掉” Aero？

在 Windows Vista 和 Windows 7 时代，Aero 主题不仅仅是好看，它代表了 Windows 引入了一项核心技术：**DWM (Desktop Window Manager，桌面窗口管理器)**。

Aero 开启时，所有应用程序的画面不再直接画到屏幕上，而是先画到显存里的一个离屏缓冲区，最后由 DWM 统一加上半透明、阴影等特效后，合并（Composite）输出到屏幕。

老游戏导致 Aero 关闭（退回 Basic）主要有以下三个原因：

*   **独占全屏 (Exclusive Fullscreen, FSE)**
    老游戏为了追求极致的帧率和极低的延迟，通常会向系统申请“独占全屏”。在这种模式下，游戏引擎会直接接管显卡的输出（Display Controller），绕过 Windows 的其他层。既然游戏独占了屏幕，负责合成桌面的 DWM 就成了累赘，甚至会引发画面撕裂或卡顿。因此，系统会自动挂起 DWM，Aero 也就随之失效。
*   **API 兼容性与陈旧的架构 (DirectDraw / DX7 / DX8)**
    很多老游戏使用的是早期 DirectX 甚至 DirectDraw API。这些 API 诞生于 Windows 98/XP 时代（那时根本没有 DWM）。当 Vista/Win7 侦测到这些古老的 API 试图全屏运行时，为了保证兼容性，会自动强制关闭 DWM，退回不支持硬件加速合成的 Basic 主题。
*   **显存与性能焦虑（主动调用 API 关闭）**
    Aero 特效需要占用一部分显存和 GPU 算力（当年主流显存只有 $256\text{MB}$ 到 $512\text{MB}$）。很多老游戏引擎（比如早期的 Source 引擎、虚幻3）为了榨干电脑性能，会在代码中主动调用 Windows API `DwmEnableComposition(DWM_EC_DISABLE)`，强制关掉 Aero 以释放系统资源。

---

### 2. 现代的改变：为什么现在不会发生了？

现代游戏和现代操作系统经过演进，彻底抛弃了旧的模式。

*   **Windows 8 及以后，DWM 变成“强制开启”**
    从 Windows 8 开始，微软改变了底层架构，DWM 成为了系统不可或缺的一部分，**无法再被关闭**。现代 Windows (Win10/Win11) 根本就没有“Basic”这个降级主题了。无论你运行什么，DWM 都在底层运行。
*   **全新的“翻转模型” (Flip Model) 与全屏优化**
    现代 DX11、DX12 和 Vulkan 采用了新的 Presentation Model（呈现模型）。
    在 Windows 10 中，微软引入了**全屏优化 (Fullscreen Optimizations)**。现在的“全屏”实际上是一种**高度优化的无边框窗口模式 (Borderless Windowed)**。当游戏画面填满屏幕时，系统会使用“独立翻转 (Independent Flip)”技术：游戏依然把画面交给系统，但系统发现游戏盖住了整个屏幕后，会自动把显卡的扫描器直接指向游戏的缓冲区，实现类似“独占全屏”的低延迟，同时保留 DWM 在后台存活。这也就是为什么现在的游戏全屏时不仅不切主题，还能瞬间 Alt+Tab 切出，且支持 Xbox Game Bar 等透明覆盖层的原因。
*   **显卡性能过剩，无需再抠资源**
    现在的显卡动辄 $8\text{GB}$、$16\text{GB}$ 甚至 $24\text{GB}$ 显存。DWM 占用的那几十兆显存和微乎其微的 GPU 算力，对现代游戏引擎来说已经完全可以忽略不计，根本没有必要再去为了节省资源而干掉操作系统的 UI 渲染。

<details>
<summary><b>🛠️ 展开查看底层技术细节：Blt 模型 vs Flip 模型</b></summary>

在图形渲染输出（Swap Chain）层面，老游戏和新游戏使用了完全不同的模型：

1.  **BitBlt (Blt) 模型（旧模型）**：
    在 DX9 时代，常用的模式是将后台缓冲区（Back Buffer）的像素数据通过一次庞大的内存拷贝（Bit Block Transfer）复制到前台缓冲区（Front Buffer）或 DWM 重定向表面。这个过程不仅慢，而且在独占全屏时与 DWM 极不兼容。
2.  **Flip 模型（新模型）**：
    从 DX10 开始，特别是 DX11.1 以后，微软强烈推行 Flip Model。在这种模式下，系统不再拷贝像素，而是直接**交换指针（Swap pointers）**。
    当全屏运行且使用 Flip 模型时，DWM 会智能地“让步”，直接将显卡读取地址指向游戏的后台缓冲区。这意味着游戏获得了等同于独占全屏的性能，但从系统架构上看，DWM 并没有被关闭，只是处于一种“直通（Passthrough）”状态。
</details>

### 总结

老游戏触发主题切换，是因为它们用着旧时代的 API，抢占着屏幕的**绝对控制权**，并把操作系统的桌面合成器（DWM/Aero）视为眼中钉；
而现代游戏和现代 Windows 达成了**和解**，通过新的图形 API（DX12/Vulkan）和**翻转模型（Flip Model）**，在不关闭系统 UI 合成器的情况下，实现了同样高效的全屏渲染。

### 🧑‍💻 User

像ue4 unity2021等，为什么现代的游戏引擎很少会用到“独占全屏”，而是一种称为无边框窗口的全屏？

---

### 🤖 Assistant

现代游戏引擎（如 Unreal Engine 4/5、Unity 2021+）不仅很少使用传统的“独占全屏（Exclusive Fullscreen, FSE）”，甚至在底层 API 层面（如 DirectX 12）已经**实质性地淘汰了传统意义上的独占全屏**。

现在的游戏所谓的“全屏”，绝大多数情况下确实是**“无边框窗口化（Borderless Windowed）”**加上了系统级的优化。

这一转变不仅是游戏引擎的选择，更是微软 Windows 团队和现代玩家习惯共同推动的结果。主要原因可以归结为以下几个核心方面：

### 1. 玩家习惯的改变：多显示器、多任务与“秒切”
在过去，玩游戏是一件“沉浸且单一”的事情。但在今天，玩家的习惯发生了巨大变化：
*   **瞬间 Alt+Tab 的需求**：传统独占全屏在按下 Alt+Tab 切出游戏时，显卡需要进行显示模式的上下文切换（Context Switch），屏幕会黑屏 $2 \sim 3$ 秒，甚至导致游戏崩溃。而无边框窗口化因为本身就是桌面上的一个窗口，Alt+Tab 切换是**瞬间完成（$0$ 延迟）**的。
*   **多显示器普及**：现代玩家经常使用双屏甚至三屏，主屏打游戏，副屏看网页、看直播或聊天。在独占全屏下，鼠标很难平滑移动到副屏，或者点一下副屏主屏游戏就会最小化。无边框窗口完美解决了这个问题。

### 2. 直播与覆盖层（Overlays）的爆发
现在的游戏生态极其依赖第三方软件的“覆盖层”和画面捕捉：
*   **覆盖层（Overlays）**：Steam 界面、Discord 语音悬浮窗、Xbox Game Bar、NVIDIA GeForce Experience 性能面板等。
*   **直播软件（OBS）**：主播需要实时抓取游戏画面。
在传统的独占全屏模式下，游戏绕过了 Windows 的窗口管理器（DWM），第三方软件要想把 UI 画在游戏画面上，必须使用**“代码注入（Hooking）”**的暴力手段。这不仅容易引发游戏崩溃，还经常被反作弊系统（如 EAC, BattlEye）误判为外挂。
而无边框窗口化模式下，游戏和覆盖层都是 DWM 管理的层，系统可以极其安全、轻松地将 Discord 界面或直播捕获层叠加在游戏之上。

### 3. 最关键的原因：全屏优化实现了“鱼和熊掌兼得”
如果你觉得无边框窗口会带来输入延迟或性能下降，那已经是过去式了。
从 Windows 10 开始，微软引入了一项杀手锏级别的技术：**全屏优化（Fullscreen Optimizations, FSO）**。

当 Windows 检测到一个游戏以无边框窗口运行，**且画面尺寸正好填满了整个屏幕时**，系统会动态触发一种叫做**“独立翻转（Independent Flip）”**的机制。
*   **它的原理是**：DWM（桌面窗口管理器）会自动“退居幕后”，让显卡直接读取游戏的渲染缓冲区输出到屏幕，绕过桌面合成步骤。
*   **最终效果**：游戏获得了**几乎等同于独占全屏的极低延迟和极高帧率**（差距通常在 $\approx 0\text{-}1\text{ ms}$ 内），但同时**保留了无边框窗口化能瞬间 Alt+Tab 切出、兼容覆盖层的全部优点**。

既然“优化后的无边框窗口”已经拥有了独占全屏的性能，独占全屏也就失去了存在的意义。

<details>
<summary><b>🛠️ 展开查看开发者视角的痛点：为什么引擎极其讨厌“独占全屏”？</b></summary>

对于使用 UE4 或 Unity 的底层图形程序员来说，独占全屏（特别是在 DX9 和 DX11 时代）是一个巨大的“系统稳定性地雷”。

1.  <b>设备丢失（Device Lost）的噩梦</b>：
    在独占全屏模式下，当玩家 Alt+Tab 切出游戏或系统弹出 UAC 警告时，显卡的控制权会被系统强行夺走。此时图形 API 会抛出一个 `Device Lost`（设备丢失）错误。
    游戏引擎此时必须手动释放所有的显存资源（纹理、顶点缓冲区、着色器），等玩家切回游戏时，再重新加载这些几 GB 的数据。如果不小心漏掉了一个指针没有清理，游戏就会直接 Crash（崩溃）。这是以前老游戏切屏容易死机的根本原因。
2.  <b>DX12 的强制进化</b>：
    到了 DirectX 12 时代，微软彻底重构了底层。DX12 的 DXGI（DirectX Graphics Infrastructure）**完全移除了传统的独占全屏（FSE）支持**。
    在 DX12 中，所有的全屏本质上都是使用了 Flip Model 的无边框窗口。即使你在游戏设置里看到了“独占全屏”的选项，那通常也只是引擎提供的一个“伪装选项”（仅仅是为了锁定鼠标或改变显示器分辨率），其底层渲染管线依然走的是现代的翻转模型。
</details>

### 总结
现代游戏引擎（UE4/5, Unity）基本放弃独占全屏，是因为现代操作系统（Win10/11）通过 **Flip Model** 和 **全屏优化**，赋予了“无边框窗口”以独占全屏的性能。
这让玩家既享受了**无延迟的秒切和多屏便利**，又保住了**高帧率和低输入延迟**，同时免去了引擎开发者处理切屏崩溃的痛苦，是现代计算机图形架构发展的一次巨大进步。

### 🧑‍💻 User

为什么独占全屏会涉及到分辨率切换延迟，而Borderless fullscreen不存在？

---

### 🤖 Assistant

要理解这个问题，我们需要分清**“软件（操作系统/GPU）”**和**“硬件（物理显示器）”**两个层面的概念。

独占全屏切换分辨率会产生黑屏延迟，是因为它触发了**物理显示器硬件级别的重新握手**；而无边框窗口全屏不产生延迟，是因为它本质上是**软件级别的画面拉伸**，显示器硬件根本不知道发生了变化。

以下是具体的原理解析：

### 1. 独占全屏 (Exclusive Fullscreen)：硬件级别的“改朝换代”

在独占全屏模式下，游戏引擎绕过了 Windows 桌面窗口管理器（DWM），直接控制了显卡的**显示控制器（Display Controller）**。

*   **物理信号的改变**：当你在独占全屏游戏中把分辨率从 $4\text{K} (3840\times2160)$ 切到 $1080\text{p} (1920\times1080)$ 时，显卡会直接向显示器发送指令：“嘿，我现在要改变输出信号的规格了！”
*   **显示器的硬切换（Mode Set）**：此时，显卡会切断当前的视频信号，导致屏幕变黑。然后，显卡使用新的分辨率和刷新率（比如 $1080\text{p} @ 60\text{Hz}$）发送新的时钟频率和同步信号（Timing信号）。
*   **为什么会慢？**：显示器的内部硬件（控制主板和缩放器）必须检测新的信号源、读取 EDID（显示器支持的规格数据）、重新对齐电子束扫描的频率（V-Sync/H-Sync），并调整物理面板的像素映射。这是一个纯硬件层面的“握手”过程，通常需要花费 $1 \sim 3$ 秒的时间。在这期间，你看到的就是黑屏。

### 2. 无边框窗口 (Borderless Fullscreen)：软件级别的“瞒天过海”

在无边框窗口模式下，游戏并没有绕过系统，它本质上**依然是 Windows 桌面上的一个应用程序窗口**，只是这个窗口恰好去掉了边框（标题栏和关闭按钮），并且尺寸正好等于你的屏幕分辨率。

*   **物理显示器被“锁死”**：在这种模式下，显示器是由 Windows 系统（DWM）控制的。只要你在系统设置里设定的分辨率是 $4\text{K}$，物理显示器就会**永远**接收 $4\text{K}$ 的固定信号，无论你在游戏里怎么折腾，显示器硬件都不需要进行任何“模式切换（Mode Set）”。
*   **GPU 负责缩放（GPU Scaling）**：如果你在这个模式下，将游戏内的分辨率设为 $1080\text{p}$ 会发生什么？
    此时，游戏引擎会在显存里画一张 $1920\times1080$ 大小的画面，然后 GPU 或 Windows DWM 会瞬间使用软件算法（如双线性插值），将这张画面强制拉伸到 $3840\times2160$（填满桌面），最后输出给屏幕。
*   **零延迟的原因**：在这个过程中，物理显示器接收到的信号始终是稳定的 $4\text{K}$ 信号，它根本不知道游戏画面变成了 $1080\text{p}$。所有的缩放工作都在 GPU 内部（显存中）以极高的速度完成了，因此**不存在任何硬件握手的黑屏时间**，切换是瞬间的。

### 3. 现代引擎的“障眼法”：渲染分辨率 (Render Resolution)

正因为现代游戏引擎（如 UE5）大量采用无边框窗口（Borderless），且为了避免任何分辨率切换导致的卡顿，它们引入了一个更加聪明的做法：**分离“输出分辨率”和“渲染分辨率”**。

在很多现代游戏中，当你选择“无边框全屏”时，你会发现**“分辨率”这一选项变成了灰色，不可更改**（锁定为你的桌面原生分辨率）。取而代之的是一个叫**“渲染比例 (Render Scale)”**或**“3D 分辨率”**的滑块（通常在 $50\% \sim 100\%$ 之间）：
*   **输出分辨率**：固定为 $4\text{K}$，保证 UI 极度清晰，切屏零延迟。
*   **渲染分辨率**：你可以调到 $50\%$。此时，游戏里的 3D 模型和光影实际上是以 $1080\text{p}$ 渲染的（节省显卡性能），然后再由引擎通过 FSR/DLSS 或 TAAU 等高级算法拉伸成 $4\text{K}$ 画面。

这种设计彻底消灭了传统改分辨率时的黑屏问题，同时保证了极佳的性能和 UI 清晰度。

<details>
<summary><b>🛠️ 展开查看扩展知识：Alt+Tab 切屏为什么独占全屏也会黑屏？</b></summary>

理解了上述原理，你就明白为什么独占全屏下 `Alt+Tab` 切回桌面会卡半天了。

当你处于独占全屏时，显卡正在输出游戏专用的信号（比如游戏强制要求 $144\text{Hz}$ 或特定伽马值）。
当你按下 `Alt+Tab`，系统需要把屏幕控制权从游戏交还给 Windows DWM。如果你的桌面分辨率/刷新率（比如 $60\text{Hz}$）和游戏内不一致，显卡就必须再次进行一次硬件级别的 **Mode Set（模式切换）**，向显示器发送桌面环境的信号。

显示器硬件被迫再次进行“信号断开 -> 重新握手 -> 亮屏”的流程，这就是你看到黑屏几秒钟的罪魁祸首。而在无边框窗口下，桌面和游戏始终共享同一个显示信号环境，所以 `Alt+Tab` 可以做到毫秒级的无缝切换。
</details>
