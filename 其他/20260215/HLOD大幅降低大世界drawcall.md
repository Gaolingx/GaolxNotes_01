# HLOD大幅降低大世界drawcall

### 🧑‍💻 User

HLOD(Hierarchical Level of Detail)+vt对于大世界drawcall的改善究竟有多显著？

---

### 🤖 Assistant

这是一个非常专业且切中肯綮的问题。在现代大规模开放世界（Open World）游戏的渲染管线中，**HLOD (Hierarchical Level of Detail)** 配合 **VT (Virtual Texture / Runtime Virtual Texture)** 对 Draw Call (DC) 的改善是**数量级（Order of Magnitude）**层面的，可以说是“从不可运行到流畅运行”的基石技术。

简单直接的结论是：**在远景渲染中，这套组合拳可以将原本数千甚至数万的 Draw Call 降低到个位数或两位数。**

以下我们将从原理、协同效应以及量化对比三个维度来详细拆解这一改善究竟有多显著。

---

### 1. 核心原理拆解

要理解为什么提升巨大，首先需要理解两者各自解决的“Draw Call 瓶颈”是什么。

#### HLOD：解决 Mesh 数量瓶颈
传统的 LOD (Level of Detail) 是针对**单个物体**的。
*   **近处：** 渲染精细模型。
*   **远处：** 渲染低模。
*   **瓶颈：** 如果你有一座城市，即便远处的 1000 栋房子都用了最低模（LOD N），CPU 依然需要提交 1000 次 Draw Call（假设没有 Instancing）。

**HLOD (分层细节)** 的逻辑是**空间聚类与合并 (Spatial Clustering & Merging)**：
*   它将空间上相邻的一组静态物体（例如一个街区的所有房子、路灯、长椅）在离线阶段合并成**一个单一的 Proxy Mesh（代理网格）**。
*   当玩家距离足够远时，引擎不再渲染那 1000 个独立的物体，而是只渲染这 1 个 HLOD Mesh。
*   **DC 变化：** $1000 \to 1$。

#### VT：解决 Material/Texture 瓶颈
仅仅合并 Mesh 是不够的。如果那 1000 个物体使用了 50 种不同的材质（木头、水泥、玻璃...），合并后的 HLOD Mesh 为了正确显示，通常需要分为 50 个 Sub-mesh（因为传统渲染管线一个 Draw Call 对应一个 Material ID），或者需要制作一张巨大的 Atlas（图集）。

**传统 Atlas 的问题：**
1.  **显存爆炸：** 一张能容纳 1000 个物体纹理的 Atlas 可能是 $16k \times 16k$，显存吃不消。
2.  **Mipmap 失效：** 采样变得复杂。

**VT (虚拟纹理) 的逻辑：**
*   它允许将超大尺寸的纹理（物理上无法放入显存）切片（Tiles）。
*   在运行时，根据屏幕覆盖率，只加载当前视角可见的那些小切片到物理显存（Physical Page Table）。
*   **对 DC 的关键贡献：** VT 允许我们将成百上千种不同物体的纹理烘焙到同一套巨大的 VT 索引中。这意味着，**无论原来的物体用了多少种材质，合并后的 HLOD Mesh 只需要 1 个 Material（即 VT Material）。**

---

### 2. 协同效应：为何 "1+1 > 2"

没有 VT 的 HLOD 往往是跛脚的，而两者的结合实现了 Draw Call 优化的终极目标：**单次绘制大量几何体与材质。**

| 场景阶段 | 渲染策略 | Draw Call 估算 (假设视野内有 1 个大区域) | 状态切换 (State Change) |
| :--- | :--- | :--- | :--- |
| **原始状态** | 无 LOD | $$N_{objects}$$ (例如 5000) | 极高 (频繁切换 VB/IB/Shader/Texture) |
| **传统 LOD** | 普通 LOD | $$N_{objects}$$ (5000) | 依然极高 (虽然面数少了，但对象数没变) |
| **仅 HLOD** | Mesh 合并，但材质未合并 | $$\sum_{clusters} M_{materials}$$ (例如 10 个 Cluster $\times$ 50 材质 = 500 DC) | 中等 (必须按材质拆分 Sub-mesh) |
| **HLOD + VT** | **Mesh 合并 + 材质统一** | $$\sum_{clusters} 1$$ (例如 10 个 Cluster = **10 DC**) | **极低** (几乎无状态切换) |

**显著性分析：**
*   **几何层面：** HLOD 将成百上千个物体变为 1 个。
*   **材质层面：** VT 确保这 1 个物体只需要 1 个 Draw Call，而不是根据材质数量分裂成多个。

---

### 3. 量化实例 (大世界场景)

假设一个典型的开放世界远景视距（View Distance）包含以下内容：
*   **可视对象：** 2,000 棵树，500 栋建筑，1,000 个道具（石头、箱子等）。
*   **总对象数：** 3,500 个 Actor。
*   **材质多样性：** 100 种不同的材质实例。

#### 方案 A：仅使用 Instancing (无 HLOD)
虽然 GPU Instancing 可以合并相同 Mesh 的绘制，但在大世界中，物体种类繁多，变换矩阵不同，剔除开销大。
*   **Draw Calls:** 约 **1,000 - 2,000** (即使开启 Instancing，不同的 Mesh 和材质也会打断 Batch)。
*   **CPU 开销:** 极高（视锥剔除、遮挡剔除遍历 3500 个对象）。

#### 方案 B：HLOD + Atlas (无 VT)
将这 3,500 个对象按网格划分合并成 10 个 HLOD Cluster。但因为纹理不能无限合并，每个 Cluster 被拆分为 5 个材质段（Sub-sections）。
*   **Draw Calls:** $10 \text{ Clusters} \times 5 \text{ Materials} = \mathbf{50}$。
*   **缺点：** 纹理模糊，显存占用高，无法流式加载高精细节。

#### 方案 C：HLOD + VT (最佳实践)
将 3,500 个对象烘焙进 10 个 HLOD Cluster。所有纹理烘焙进一套 Runtime Virtual Texture (RVT)。
*   **Draw Calls:** $10 \text{ Clusters} \times 1 \text{ Material} = \mathbf{10}$。
*   **改善幅度：** 相比原始状态，Draw Call 降低了 **99% 以上**。

### 4. 深入技术细节与代价 (UE5 案例)

<details>
<summary><strong>点击展开：UE5 中的具体实现逻辑与 Shader 复杂度</strong></summary>

在 Unreal Engine 5 中，这套流程已经非常成熟（但在 Nanite 出现前是绝对主流，即使现在对非 Nanite 平台依然适用）：

1.  **HLOD Generation:**
    *   引擎使用 Simplygon 或内置工具，在构建时（Cook time）将指定半径内的 Static Mesh 变为一个 Static Mesh。
    *   **Proxy Mesh:** 面数大幅削减。

2.  **RVT Baking:**
    *   材质中包含 `Virtual Texture Sample` 节点。
    *   HLOD Actor 的材质被设置为专门的 "HLOD_RVT_Material"。
    *   这个材质不采样原始的 BaseColor/Normal 贴图，而是直接采样 VT 的 Page Table。

3.  **Draw Call 的本质:**
    *   当渲染远景 HLOD 时，GPU 只需要绑定一次 Vertex Buffer (合并后的大 Mesh) 和一次 Texture Resource (VT 的 Page Table)。
    *   **SetPass Call** 几乎降为 0。

4.  **Shader Complexity:**
    *   HLOD+VT 的 Shader 通常非常简单（Unlit 或者简单的 Default Lit），去掉了复杂的 Layered Material 逻辑，进一步降低了 GPU 耗时 (Pixel Shader Cost)。

### 5. 局限性与权衡

虽然对 Draw Call 的改善是革命性的，但并非没有代价：

1.  **磁盘占用 (Disk Size)：**
    HLOD Mesh 和烘焙好的 VT 纹理需要额外的存储空间。对于超大世界，这可能增加数 GB 的包体大小。
2.  **构建时间 (Bake Time)：**
    生成 HLOD 和烘焙 VT 非常耗时。修改了场景中的一棵树，可能需要重新烘焙整个分区的 HLOD。
3.  **流送压力 (Streaming)：**
    虽然降低了 Draw Call，但 VT 需要频繁地从磁盘流送 Tile 到显存。如果硬盘 IO 慢（如机械硬盘），会导致贴图模糊（Pop-in）。
4.  **过渡瑕疵：**
    从 HLOD 切换回真实模型时，如果 VT 的光照/颜色与原始模型有差异，会产生明显的视觉突变（Popping）。

### 总结

**HLOD + VT 对于大世界 Draw Call 的改善是决定性的。**

*   它将 Draw Call 的数量级从 **$O(N)$**（$N$ 为物体数量）降低到了 **$O(C)$**（$C$ 为空间划分的网格数量，通常极小）。
*   如果没有这项技术，现代主机和 PC 上的 3A 级开放世界（如《绝地求生》、《堡垒之夜》早期版本、《原神》远景处理等）在远景丰富度上根本无法达到现在的标准。
*   它将 CPU 从繁重的 "提交绘制命令" (Submit Draw Command) 中解放出来，使其能处理更多的逻辑、物理和 AI。

简单来说，HLOD 的本质就是一次**“空间换时间”和“以死（静态）换活（动态）”的交易**。

你总结的这个过程，在图形学里通常被称为 **"Imposter"（冒名顶替）** 或者 **"Proxy"（代理）** 生成。我们可以把你这两句话拆解开，看看它到底是怎么运作的，以及为什么说你抓住了重点。

### 1. Mesh 层面：从“散装”到“批发”

> **你的理解：** *"按cluster划分再生成Proxy"*

**完全命中。**

*   **原来的样子：** 一个村庄里有 100 栋房子，每栋房子有门、窗、屋顶、烟囱，全是独立的 Mesh。CPU 每次都要告诉 GPU：“画个门”、“画个窗”、“画个烟囱”……喊了几千次（几千个 Draw Call）。
*   **HLOD 的做法：**
    1.  **Cluster（聚类）：** 把这一整块区域（比如半径 100 米内）的所有东西圈起来。
    2.  **Merge（合并）：** 把这些乱七八糟的 Mesh 焊接到一起，变成一个大 Mesh。
    3.  **Simplify（简化/Remesh）：** **这是关键一步。** 仅仅合并还不够，因为面数没减。HLOD 生成器会重新拓扑（Remesh）或者减面，把原本复杂的门把手细节直接抹平，把原本是圆柱体的烟囱变成六棱柱甚至四棱柱。
    4.  **结果：** 几千个独立的精细物体，变成了一个面数很少的、大致轮廓相似的“大土块”（Proxy Mesh）。

### 2. Texture 层面：从“百家饭”到“大锅饭”

> **你的理解：** *"不管美术用了几千张texture，我全把他baking到几张2048x2048的tex上去"*

**这就是 Atlasing（图集化）的本质，也是 HLOD 材质合并的核心。**

*   **原来的样子：** 房子用砖块贴图，窗户用玻璃贴图，路灯用金属贴图。渲染时 GPU 需要不断切换贴图资源，打断流水线。
*   **HLOD 的做法：**
    1.  **重展 UV：** 既然所有物体合并成一个 Mesh 了，就需要给这个新 Mesh 重新生成一套 UV（通常是 UV2 或 UV3），确保它们互不重叠。
    2.  **Baking（烘焙）：** 这是一个“拍照”的过程。引擎会从各个角度“看”原始的那些高模和原始贴图，把它们表面的颜色、法线、粗糙度信息，“投影”并画到你说的那张新的 2048x2048 的大图上。
    3.  **结果：** 无论原来有多少种材质，现在对于这个 HLOD Proxy 来说，它只认这一张 2048 的图。**Draw Call 这里的 Material 切换瞬间归零。**

### 3. 你提到的“2048x2048”与 VT 的区别

这里有一个非常微妙但重要的技术分歧点，正好对应你上一个问题里的 **VT**。

**方案 A：传统 Atlas（你描述的 2048x2048）**
*   **做法：** 真的生成一张 2048 或 4096 的物理图片文件（Texture Atlas）。
*   **痛点：** 如果你合并了一个巨大的城市街区，包含几十栋楼，一张 2048 的图根本不够分。平均分摊下来，每栋楼可能只分到 100x100 的像素。
    *   **后果：** 远看还行，稍微拉近一点点，全是马赛克，糊成一团。这就是为什么很多老游戏的远景看起来特别“脏”和“糊”。

**方案 B：HLOD + VT（进阶版）**
*   **做法：** 逻辑上依然是把它们 Bake 到一张图上，但这张图不是物理上的 2048，而是一张**虚拟的**、甚至可以是 16384x16384 (16k) 甚至更大的超大图。
*   **优势：** 因为它是 VT，只有屏幕上看到的部分才会加载进显存。
*   **修正你的理解：** 所以在 VT 方案下，并不是“挤在几张 2048 上”，而是“**豪横地平铺在一张巨大的虚拟画布上**”，从而保证了即使是 HLOD，清晰度也可以做得比较高。

### 4. 为什么叫 "H" (Hierarchical)？

还有一个点你没提到，就是那个 **H**。这也是它强大的原因。

*   **LOD 0:** 原始模型（几万个物体）。
*   **HLOD 1 (近距离):** 把每 5 个房子合并成 1 个 Proxy。
*   **HLOD 2 (中距离):** 把每 20 个 HLOD 1 合并成 1 个更大的 Proxy。
*   **HLOD 3 (远距离):** 把整个城市合并成 1 个超大的 Proxy。

这就是一个**俄罗斯套娃**的逻辑。你离得越远，看到的那层壳（Proxy）就包得越大，内部被剔除的细节就越多。

### 总结

你的理解非常到位：

**HLOD = 暴力合并模型 (Mesh Combine & Simplify) + 暴力拼合贴图 (Texture Baking/Atlasing)**

它就是把美术原本精细制作的“手办柜”，拍成了一张“海报”。
*   近看，你需要看手办柜里的每一个手办（Draw Call 爆炸）。
*   远看，你只需要看墙上贴的那张海报（1 个 Draw Call）。

你说到了痛点。**“自动化生成完美的 Proxy Mesh”** 基本上是图形学里的一个圣杯问题，也是各大厂技术美术（TA）和引擎开发掉头发最多的地方。

这就是一个极端的**平衡游戏（Trade-off）**：
*   **网格太简：** 轮廓对不上，切换时画面跳变（Popping/Artifact），穿帮。
*   **网格太细：** 失去了 HLOD 节省性能的意义。
*   **Baking 太糊：** 远看像一坨马赛克。
*   **Baking 太慢：** 构建版本（Cook）需要几天几夜。

下面我拆解一下，生成这个“既要省性能又要不穿帮”的 Proxy 到底有哪些具体的技术难点，以及业界是怎么填坑的。

---

### 难点一：几何体生成的策略（是“包饺子”还是“糊灯笼”？）

面对一堆散乱的 Mesh（比如一个摆满杂物的集市摊位），生成 Proxy Mesh 主要有两种算法流派，各有各的坑：

#### 1. 表面重建 (Surface Reconstruction / Shrink Wrapping)
想象拿一块保鲜膜，紧紧裹住那一堆物体。
*   **做法：** 基于体素（Voxelization）或者 SDF（有向距离场）重建一个外壳。
*   **优点：** 能完美填补物体内部的空洞（剔除不可见的面），生成的拓扑结构比较干净。
*   **难点/Artifact：**
    *   **细节丢失：** 比如栏杆、电线杆这种细长的物体，如果体素精度不够，直接就**断掉**或者**消失**了。
    *   **融合粘连：** 两个离得很近但没挨着的物体，会被“粘”在一起，看着像融化的蜡像。

#### 2. 聚合简化 (Aggregation & Simplify)
把所有物体的 Mesh 合并，然后狂用减面算法（Decimation）。
*   **做法：** 也就是 Simplygon 等软件常用的方式。
*   **优点：** 细长物体（如栏杆）能保留下来。
*   **难点/Artifact：**
    *   **废面无法剔除：** 房子里面的桌子虽然看不见，但如果算法不聪明，可能把桌子也算进 Proxy 里了，浪费面数。
    *   **拓扑糟糕：** 减面减狠了会出现长条三角形（Sliver triangles），导致着色（Shading）出现黑斑或伪影。

---

### 难点二：法线与光影的烘焙（如何在一张纸上画出立体感？）

当你把一堆复杂的几何体变成一个平板或简单的凸包时，**法线（Normal）的匹配**是最大的穿帮来源。

*   **问题场景：** 原始物体是一个棱角分明的箱子，Proxy 是一个圆滑的土包。
*   **Artifact：** 
    *   如果你直接把箱子的法线 Bake 到土包上，在光照角度变化时，土包边缘会出现极不自然的黑边或反光断裂。
    *   **UV 接缝（Seams）：** 自动展开的 UV 往往支离破碎。如果 Padding（出血/留边）没留够，远看时 Mipmap 一开，背景色就会渗入纹理，导致物体边缘出现莫名其妙的白线或黑线。

**业界的解决办法：**
*   **Gutter Space Dilate:** 烘焙时疯狂向外涂抹边缘像素，防止 Mipmap 采样溢出。
*   **Bent Normals:** 烘焙遮蔽法线，让光照计算更符合原始的大致体积，而不是死扣细节。

---

### 难点三：半透明物体的噩梦（树叶与栅栏）

这是 HLOD 最大的死穴。

*   **问题：** 树叶是插片的（Alpha Test/Masked）。如果要合并几棵树做一个 Proxy：
    *   **如果是 Mesh 简化：** 减面算法很难处理插片，很容易把一片叶子减成一个点，或者把树冠减成一个实心的绿球。
    *   **如果是 Baking：** 你很难在一张不透明的图上正确表示“前后遮挡”的树叶关系。
*   **Artifact：**
    *   **体积感消失：** 远处的树看起来像纸片做的。
    *   **光照错误：** 原始树叶透光（Subsurface），Bake 出来的 Proxy 通常是不透光的，导致远处的树看起来特别黑，或者特别亮。

**这也是为什么《原神》、《塞尔达》或者《地平线》这种游戏，树木的 HLOD 通常是单独处理的（Imposter / Octahedral Imposter），不和房子石头混在一起 Bake。**

---

### 难点四：数据量与流送（DrawCall 降了，显存爆了）

你提到了 "baking 到几张 2048x2048"。

*   **两难境地：**
    *   如果 Cluster 划分得太大（比如整个山头），一张 2048 的贴图分给几千个物体，每个物体分不到几个像素，结果就是**“糊”**。
    *   如果 Cluster 划分得太小，HLOD 的数量就会变多，Draw Call 优化的效果就打折了。
*   **Artifact：**
    *   当玩家从 HLOD 切换回真实物体时，如果贴图精度差异过大，视觉上会有明显的**“变清晰”**的过程，这种突变很廉价。

### 总结：如何判断“好”的 HLOD？

现在的 3A 游戏（包括 UE5 的 World Partition 流程）通常依靠强大的中间件（如 **Simplygon** 或 **Houdini**）来自动化这个过程。

一个成功的 HLOD 生成流通常包含这些骚操作：

1.  **分层处理：** 
    *   **不透明物体（石头/墙）：** 激进地减面，Voxelization 处理。
    *   **半透明/Mask物体（树/栏杆）：** 单独提取，用公告板（Billboard）或特殊 Imposter 技术，最后再拼回去。
2.  **材质简化：**
    *   把 Metallic/Roughness/Specular 全部压在一张纹理的不同通道里（ORM图），极致压缩纹理数量。
3.  **过渡欺骗：**
    *   使用 **Dither Fade（抖动渐隐）**。在 HLOD 和真实物体切换时，不是瞬间变身，而是像噪点一样慢慢过渡，掩盖几何体不匹配带来的“跳动”。

**一句话总结：HLOD 的核心难点不在于原理，而在于对“丑”的容忍度控制。美术希望哪怕是一个像素的 Proxy 也要美若天仙，程序希望它最好只是一个只有 12 个面的立方体。TA 的工作就是在中间调停。**
