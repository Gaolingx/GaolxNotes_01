### 🤖 Assistant

这份后训练（Post-training）Pipeline 设计非常贴近目前业界最前沿（如 DeepSeek-R1, Qwen-2.5-Math, OpenAI o1-like）的 System 1（直觉响应）与 System 2（深度推理）分离的训练范式。

以下是对该 Pipeline 的亮点分析，以及全流程对基础设施（Infra）和数据（Data）提出的要求与挑战。

### 一、 训练 Pipeline 的可圈可点之处

1. **System 1 与 System 2 架构的解耦与专注**
   * **Instruct模型走短平快路线**：`Base -> Long ctx -> SFT -> DPO` 保证了模型作为日常对话和通用指令遵循（System 1）的低成本、高效率，避免了被复杂推理任务破坏其原本的语言风格和通用能力（即“对齐税”）。
   * **Reasoning模型走深度强化路线**：投入巨量资源进行两阶段 SFT 和混合 RL，专门榨取模型的逻辑、数学、代码和 Agent 能力（System 2）。
2. **教科书般的课程学习（Curriculum Learning）设计 (SFT 两阶段)**
   * **Stage 1 (High Quality, 基础能力)**：先用极高质量的小规模数据建立基本的指令遵循（IF）、格式控制和工具调用能力。这相当于给模型“立规矩”。
   * **Stage 2 (Wide, 领域拓展)**：在模型有了好习惯后，再灌入海量、宽泛的领域数据（Science, SWE, Terminal）。避免了早期直接引入复杂数据导致模型格式崩溃或损失收敛困难。
3. **混合强化学习（Mixed RL）与防遗忘机制**
   * **RLVR (RL with Verifiable Rewards)**：在数学和代码等领域使用基于规则验证的奖励，彻底解决了传统基于 RM（Reward Model）带来的 Reward Hacking（奖励作弊）问题。
   * **General SFT Sampling with Response Pool**：在 RL 阶段引入通用 SFT 数据进行经验回放（Replay）。这是极其关键的一步，纯 RLVR 极易导致模型在非推理话题上能力断崖式下跌（Catastrophic Forgetting）或语言退化（Language Degeneration）。Response Pool 保证了模型在变聪明的同时依然“会好好说话”。
4. **Context 扩展前置**
   * `Long ctx ft` 放在所有 SFT 和 RL 之前。这保证了后续的 Agent、SWE（软件工程）和 Terminal 操作等天然需要长上下文的任务，能够直接在长窗口下进行微调，无需在训练后期再做强行扩展。

### 二、 对 Infra（基础设施）的要求与挑战

MoE 加上长文本，再加上 RL，这是目前大模型 Infra 领域的“地狱难度”。

<details>
<summary><b>1. MoE 架构带来的通信与显存挑战</b></summary>

* **Expert Load Balancing（专家负载均衡）**：在 SFT 和特别是 RL 阶段，如果数据分布发生变化，极易导致某些专家被“饿死”，某些专家过载（Token Dropping）。Infra 需要支持动态的辅助损失函数（Auxiliary Loss）或无辅助损失的路由算法（如 DeepSeek-V3 的策略）。
* **All-to-All 通信瓶颈**：MoE 在节点间分配专家时，Forward 和 Backward 过程需要大量的 All-to-All 通信。对集群网络的 RDMA 带宽要求极高，必须实现计算与通信的极致重叠（Computation-Communication Overlap）。
</details>

<details>
<summary><b>2. 长上下文 (Long Context) 训练挑战</b></summary>

* **KV Cache 显存爆炸**：长文本会导致显存占用呈线性甚至二次方增长。需要 Infra 支持成熟的序列并行（Sequence Parallelism），如 `RingAttention` 或 `Ulysses`，将长序列切片分布到多张卡上。
* **计算效率**：在超长文本下，注意力机制的计算时间急剧上升。需要高度优化的 FlashAttention 算子来支持变长序列（Varlen）和长序列。
</details>

<details>
<summary><b>3. RL（强化学习）阶段的系统架构挑战</b></summary>

* **Actor-Critic-Reference-Reward 架构部署**：PPO/RLVR 流程通常需要同时驻留 4 个模型（Actor, Critic, Reference, Reward）。在 MoE 参数量巨大的情况下，如何利用 vLLM 等高性能推理引擎做 Actor 的 Generation，并与 Megatron/DeepSpeed 的 Training 引擎做快速的权重/梯度同步，是最大的工程难点。
* **在线验证环境（Verifiable Environments）的并发隔离**：RLVR 强调“Verifiable”。对于 SWE/Terminal Agent，需要 Infra 能够每秒启动和销毁成百上千个沙盒（Sandboxes, 如 Docker/Firecracker）来执行模型生成的代码并返回测试结果。这要求一个极其强大的分布式评测调度系统。
* **Response Pool 的 I/O 吞吐**：维护 SFT Replay Buffer 并从中实时采样，要求分布式文件系统拥有极高的随机读取吞吐量。
</details>

### 三、 对 Data（数据）的要求与挑战

后训练阶段，**数据质量决定了模型的上限**。该 Pipeline 对数据构造提出了严苛要求。

<details>
<summary><b>1. SFT 阶段：极端的质量与多样性控制</b></summary>

* **Stage 1 数据挑战（High Quality）**：这部分数据不在多，而在“精”。需要极其专业的数据标注员（甚至专家）手写指令和回复。任何格式错误、幻觉或逻辑漏洞都会在这一阶段被放大。
* **Stage 2 数据挑战（Wide/Domain Specific）**：
  * **SWE/Terminal 数据**：很难通过人类标注产生，需要利用更强的闭源模型（如 GPT-4o）结合具体的 Github 仓库历史来合成（Synthetic Data）。
  * **MoE 专家激活要求**：数据的领域分布必须足够均匀和宽泛，否则会导致 MoE 模型的专家坍塌（只激活少数几个专家）。
</details>

<details>
<summary><b>2. Mixed RL 阶段：Prompt 与 Verifier 的构造</b></summary>

* **RLVR Prompt 构造**：RL 不需要人类写答案，但需要人类/脚本提供海量的“有问题且带有确切测试用例”的 Prompt。例如：LeetCode 级别的代码题加 Unit Tests，或者过程复杂的数学题加最终答案提取逻辑。
* **Reward Shaping（奖励塑造）**：
  * 对于数学，仅通过最终答案判断对错（稀疏奖励）可能导致收敛极慢。需要数据工程层面构造 PRM（Process Reward Model）格式的数据，或者在验证器中加入中间步骤检查（Step-level Verifier）。
  * 对于 Agentic 任务，需要定义 Terminal/Web 操作的“部分成功状态”，否则模型很难在庞大的状态空间中探索出完整路径。
</details>

<details>
<summary><b>3. DPO 阶段：高质量偏好数据的获取</b></summary>

* **Reasoning 模型的 DPO 困境**：对于通用对话，人类很容易判断“哪个回复更好”。但对于动辄数千 Token 的深度推理过程或代码逻辑，人类很难快速分辨 $y_w$（Chosen）和 $y_l$（Rejected）。
* 应对挑战通常需要引入 **AI Feedback (RLAIF)**，利用更强的模型或者代码执行结果来辅助生成偏好对，或者对推理过程中的“风格”（如：是否清晰列出步骤、是否过度冗长）进行 DPO 优化，而非纯逻辑对错（逻辑对错已交由 RLVR 负责）。
</details>

### 总结

这条 Pipeline 是一条**重资产、重工程、重算力**的王者路线。它不仅要求极高的集群网络通信能力和分布式训练/推理框架的深度魔改，还要求团队具备极强的合成数据能力和在线代码沙盒调度能力。跑通这条流意味着该团队已经跨越了迈向 AGI 的“系统工程”门槛。

### 🧑‍💻 User

Frontier post-training is no longer a single SFT-to-RL recipe. The emerging pattern is a stack of methods, each solving a different bottleneck:
$$
\begin{aligned}
            &\text{SFT / offline distillation}
            \to
            \text{on-policy distillation / self-distillation} \\
            &\to
            \text{specialist RL}
            \to
            \text{multi-teacher OPD}
            \to
            \text{unified model}
            \end{aligned}
$$
SFT and offline distillation teach the model what good behavior looks like — fast and stable, but limited to trajectories the model didn't generate itself. On-Policy Distillation goes further by having the model generate its own trajectories, though its ceiling is still set by teacher capability. Self-distillation ( OPSD / SDPO) removes the external teacher entirely, letting the model supervise itself under richer context: either from a hint, a verified answer, or environment feedback. Applied Compute's RMSD extends this into production with a filtered loss mask for out-of-distribution tasks. improving data efficiency and training stability. But in all cases, the ceiling is determined by the data. 相比mixed rl这套流程优势在哪？

### 🤖 Assistant

你提到的这套全新 Pipeline（`SFT -> OPD/SD -> Specialist RL -> Multi-teacher OPD -> Unified Model`）代表了目前业界最前沿的**“分而治之与知识融合（Divide-and-Conquer & Fusion）”**后训练范式（类似于 Llama 3 晚期版本或各大厂探索探索下一代模型的内部路线）。

与上一轮提到的 **混合强化学习（Mixed RL）** 流程相比，这套新流程的核心思想是从**“端到端的联合优化”**转向了**“多阶段的蒸馏与专家融合”**。它的优势主要集中在突破能力上限、提升训练稳定性以及解决数据分布偏移上。

以下是这套新流程相比 Mixed RL 的核心优势：

---

<details open>
<summary><h3 style="display:inline-block; margin:0;">1. 打破“多任务学习”的梯度冲突，推高能力上限</h3></summary>

*   **Mixed RL 的瓶颈：** 在 Mixed RL 中，模型同时在做代码验证（RLVR）、数学推理和通用对话（SFT replay）。由于单一模型的容量有限，且不同任务的优化方向（Reward Landscape）经常相互垂直甚至冲突，模型很容易在某项能力上停滞不前，这被称为**“对齐税（Alignment Tax）”**。
*   **新流程的优势 (Specialist RL)：** 新流程允许先剥离出**领域专家（Specialist）**单独进行强化学习（比如单独训一个 Math RL 模型、一个 Coding RL 模型）。因为没有通用任务的羁绊，专家模型可以无所顾忌地通过 RLVR 探索到该领域的极限能力上限。

</details>

<details open>
<summary><h3 style="display:inline-block; margin:0;">2. 完美解决分布偏移 (Distribution Shift)，模型学得更自然</h3></summary>

*   **Mixed RL 的瓶颈：** SFT 和 Offline Distillation 的数据是外界（Teacher 或人类）生成的。这就好比让一个初中生去死记硬背大学生的解题步骤，虽然答案是对的，但这不符合初中生自身的“思维流（Trajectory）”，也就是偏离了模型自身的概率分布（Off-policy）。
*   **新流程的优势 (On-Policy Distillation, OPD)：** OPD 强制要求**使用模型自己生成的轨迹（Trajectories）**。模型先用自己的参数生成多个解答，然后由 Teacher 或环境来打分/筛选。这确保了用于优化的数据始终在模型自身的概率流形（Manifold）上。模型是在“纠正自己的错误”，而不是“模仿别人的完美”，这种学习效率更高，且极大减少了幻觉。

</details>

<details open>
<summary><h3 style="display:inline-block; margin:0;">3. 用多教师蒸馏 (Multi-teacher OPD) 替代混合采样，彻底避免灾难性遗忘</h3></summary>

*   **Mixed RL 的瓶颈：** 混合 RL 依赖 SFT 数据池（Response Pool）来防止模型遗忘对话能力。但这种强行混入静态数据的方法，在 RL 动态更新模型权重时，很难动态平衡 Loss 的权重。
*   **新流程的优势：** 新流程不直接对 Unified Model 做 RL。它将之前训练好的所有 Specialist RL 模型作为“教师委员会（Multi-teacher）”。当输入一个 Prompt 时，由对应的专家教师生成最完美的 On-policy 轨迹，再通过**蒸馏（监督学习）**的方式教给 Unified Model。由于最终的聚合阶段（Unified Model）本质上是稳定的**自回归监督学习（Cross-Entropy Loss）**，完全避开了混合 RL 中价值网络崩溃、Reward Hacking 等极端不稳定的情况。

</details>

<details open>
<summary><h3 style="display:inline-block; margin:0;">4. 通过 RMSD (过滤掩码) 解决 OOD 与训练稳定性问题</h3></summary>

*   **Mixed RL 的瓶颈：** 在 RL 阶段遇到分布外（Out-of-Distribution, OOD）的 Prompt 时，模型可能会生成完全无意义的内容，如果强行给予惩罚（Penalty），会导致梯度剧烈震荡，破坏已学到的泛化能力。
*   **新流程的优势 (RMSD Masking)：** 正如你提到的 Applied Compute 的 RMSD 技术，通过在 Self-distillation 阶段使用 Filtered Loss Mask。当模型在面对 OOD 任务生成的轨迹置信度极低、或者反馈极其嘈杂时，**直接在 Loss 计算时 Mask（遮蔽）掉这部分梯度**。这相当于一种“不知为不知”的安全机制，既保证了数据利用率（Data Efficiency），又极大提升了训练稳定性。

</details>

### 总结比较

| 维度 | Mixed RL Pipeline (上一代) | Distillation & Specialist Stack (新一代) |
| :--- | :--- | :--- |
| **优化方式** | 同一个模型上同时混合 RL 和 SFT 梯度 | 拆解为多个专家单独 RL，再统一蒸馏回主模型 |
| **分布匹配** | 静态 SFT (Off-policy) + 动态 RL | 高度依赖 On-policy 生成与自我蒸馏 |
| **训练稳定性** | **低**。Actor, Critic, 混合数据的平衡极难调 | **高**。将脆弱的 RL 限制在专家阶段，最终融合用稳定的监督蒸馏 |
| **能力上限** | 受限于多任务梯度冲突 (Capacity bottleneck) | 通过 Specialist RL 打破单域上限，再通过多教师集成实现 1+1>2 |
| **核心挑战** | RL Infra 的实时并发能力、Reward 权重调参 | **高质量打分/过滤机制**的建立、多教师推理的极高算力成本 |

新范式说明业界已经达成共识：**直接用 RL 训练全能基座模型太不可控了。** 未来的趋势是“把 RL 当作获取极限领域数据的手段（训练 Specialist）”，而“把蒸馏（OPD/SD）当作向主模型注入能力的稳定管道”。正如你最后引用的，“the ceiling is determined by the data” —— 新流程本质上是一套比 Mixed RL 更高效、更稳定地**自我生产高质量 On-policy 数据**的飞轮系统。
