# 默认右填充大模型需左填充

### 🧑‍💻 User

hf autotokenizer默认是left padding还是right？

---

### 🤖 Assistant

Hugging Face 的 `AutoTokenizer` 的默认 padding 方向（左侧还是右侧）**并不是全局固定的**，而是取决于你加载的**具体模型（Model Checkpoint）**的配置文件（`tokenizer_config.json`）。

但是，如果模型配置文件中没有显式指定，Hugging Face 底层 `PreTrainedTokenizer` 的**默认 fallback 值是 `right`（右侧填充）**。

不同的模型架构通常有不同的最佳实践和默认设置：

1. **Encoder-only 模型（如 BERT, RoBERTa）：**
   通常默认是 **`right`** padding。因为这些模型是双向注意力机制，且通常用于文本分类、NER等任务，右侧填充结合 Attention Mask 能够很好地工作。
2. **Decoder-only 模型（如 GPT-2, LLaMA, Qwen 等 Causal LMs）：**
   配置文件的默认值可能是 `right`，也可能是 `left`。**但是，如果你要进行文本生成（调用 `model.generate()`），强烈建议（有时是强制要求）使用 `left` padding。** 因为在自回归生成时，模型需要基于序列的最后一个 token 继续预测，如果右侧有 padding token，会干扰位置编码和生成的逻辑。

<details>
<summary><b>点击展开：如何查看和修改 Tokenizer 的 Padding 侧</b></summary>

你可以通过代码轻松查看当前加载的 tokenizer 的默认 padding 方向，也可以在需要时手动修改它：

```python
from transformers import AutoTokenizer

# 1. 加载 Tokenizer
tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased")

# 2. 查看当前的 padding side
print("当前 padding side:", tokenizer.padding_side) 
# 输出通常为: right

# 3. 修改 padding side 为 left (例如在做大模型生成任务时)
tokenizer.padding_side = "left"
print("修改后的 padding side:", tokenizer.padding_side)
```

**注意事项：**
如果你的模型没有默认的 padding token，在设置 padding side 之前，你需要先设置 `pad_token`。对于很多 Decoder 架构的模型（比如 LLaMA），通常会将 `eos_token` 设为 `pad_token`：

```python
if tokenizer.pad_token is None:
    tokenizer.pad_token = tokenizer.eos_token
tokenizer.padding_side = "left" # 为生成任务设置为左侧填充
```

</details>

**总结：**
* 默认值：通常是 **`right`**，但具体取决于加载的模型。
* 分类/理解等非生成任务：通常用 **`right`**。
* 大语言模型文本生成（Generation）：必须手动或确保设置为 **`left`**。

### 🧑‍💻 User

对于llama qwen等llm，这个参数在训练和推理时需要修改吗？

---

### 🤖 Assistant

对于 LLaMA、Qwen 等纯解码器（Decoder-only）架构的大语言模型（LLM），**在训练和推理（特别是批量生成）时，`padding_side` 是需要区分并修改的。**

业界的标准最佳实践是：**推理时用左侧填充（`left`），训练时用右侧填充（`right`）。**

以下是具体的原因和原理解析：

### 1. 推理阶段 (Inference / Generation)
**必须设置为 `left`（左侧填充）**

当你要对多个不同长度的 prompt 进行 **Batch 推理（调用 `model.generate`）** 时，必须使用左侧填充。

*   **原因：** LLM 采用自回归（Autoregressive）方式生成文本，即基于前面的 token 预测下一个 token，并追加到序列的最末尾。
*   **如果不改（用 right）：** 假设序列是 `[Token1, Token2, PAD, PAD]`。模型在计算生成下一个 token 时，会在最右侧的 `PAD` 后面继续生成，这不仅会导致位置编码（Position IDs）错乱，生成的文本也会是无意义的乱码。
*   **改为左侧填充后：** 序列变成 `[PAD, PAD, Token1, Token2]`。真实的输入 token 紧挨着要生成的新 token（位于序列最右侧），模型可以正确提取最后一个 token 的特征进行连续生成。

### 2. 训练阶段 (Training / Fine-tuning)
**通常设置为 `right`（右侧填充）**

在进行预训练（Pre-training）或指令微调（SFT）时，我们通常计算整段输入（或者 Mask 掉 prompt，只计算 response 部分）的交叉熵损失（Cross-Entropy Loss）。在这个阶段，标准做法是使用右侧填充。

*   **原因：** 数据通常被拼接为 `[Prompt, Response, EOS]` 的形式。如果序列长度不足最大长度 $L$，会在末尾补齐：`[Prompt, Response, EOS, PAD, PAD]`。
*   **为什么不用左侧填充：** 如果在左侧填充 `[PAD, PAD, Prompt, Response, EOS]`，虽然通过 Attention Mask 模型能够忽略前方的 PAD，但这种格式违背了自然语言从左到右的阅读和处理直觉，且在某些底层的算子优化（或特定 Loss 计算逻辑）中，右侧填充能够获得更好的内存连续性和兼容性。

**1. 代码中如何动态切换：**

在编写代码时，你可以根据当前的任务动态修改 Tokenizer 的属性：

```python
from transformers import AutoTokenizer

model_id = "Qwen/Qwen1.5-7B-Chat"
tokenizer = AutoTokenizer.from_pretrained(model_id)

# 确保有 pad_token
if tokenizer.pad_token is None:
    tokenizer.pad_token = tokenizer.eos_token

# -------- 推理阶段 --------
tokenizer.padding_side = "left"
# inputs = tokenizer(prompts, padding=True, return_tensors="pt")
# model.generate(**inputs)

# -------- 训练阶段 --------
tokenizer.padding_side = "right"
# dataset = dataset.map(lambda x: tokenizer(x["text"], padding="max_length", ...))
# trainer.train()
```

**2. 进阶：不需要 Padding 的情况 (Data Packing / Flash Attention)**

在实际的**大规模高效训练**中（比如使用 Flash Attention 2），很多时候我们根本**不使用 Padding**。
为了最大化 GPU 利用率，我们会将多个短序列用 `EOS` 拼接成一个长度正好为 $L$（如 4096）的长序列（即 **Data Packing** 或者 **Sequence Concatenation**）。在这种情况下，因为没有 PAD token，`padding_side` 参数也就失去了意义。

</details>

**总结：**
*   **训练模型（SFT / Pre-train）：** `tokenizer.padding_side = "right"`
*   **调用模型生成（Batch Generation）：** `tokenizer.padding_side = "left"`
