# 大模型训练时序设置

### 🧑‍💻 User

大模型训练中，model的triton compile/apply tp/apply fsdp2时序如何设置？

---

### 🤖 Assistant

在大模型训练（特别是基于 PyTorch 2.x 生态）中，结合 Tensor Parallelism (TP)、Fully Sharded Data Parallelism v2 (FSDP2) 和 `torch.compile` (Triton backend) 时的正确时序是非常关键的。

正确的时序设置应当是：

1. **实例化基础模型 (Init Model)**
2. **应用张量并行 (Apply TP)**
3. **应用全分片数据并行 (Apply FSDP2)**
4. **模型编译 (Triton Compile)**

即：**Model Init -> Apply TP -> Apply FSDP2 -> `torch.compile`**。

以下是详细的原因分析和标准的伪代码实现。

### 为什么是这个时序？

#### 1. 先 Apply TP，后 Apply FSDP2
* **依赖关系：** TP 是在单个设备组（通常是单机内的多卡）上对张量（如 Linear 层的权重）的物理维度进行切分。应用 TP 后，模型权重的局部形状（Local Shape）会发生改变。
* **FSDP2 的工作原理：** FSDP2 依赖于 `DTensor` 和 `DeviceMesh`，它需要在数据并行（DP）维度上对模型参数进行打散（Sharding）。FSDP2 必须知道经过 TP 切分后的准确张量大小，才能正确地在 DP 维度上分配显存并注册 Reduce-Scatter / All-Gather 等通信算子。因此，**TP 必须在 FSDP 之前应用**。

#### 2. 最后应用 `torch.compile`
* **图的完整性：** `torch.compile`（底层是 AOTAutograd 和 Inductor）通过追踪（Tracing）前向和反向传播来生成 Triton Kernel。为了实现全局最优（例如计算与通信的重叠、算子融合），**编译器必须看到包含所有分布式通信算子（TP 的 All-Reduce，FSDP2 的 All-Gather）的完整计算图**。
* 如果在应用分布式策略之前进行 compile，编译器只能看到单卡模型的图，后续注入的分布式钩子（Hooks）或张量替换会导致编译图失效或引发严重的正确性错误。

---

### 标准时序代码示例 (PyTorch 2.x 风格)

在 PyTorch 2.x 中，通常使用 2D `DeviceMesh` 来统一管理 TP 和 FSDP2。

```python
import torch
import torch.nn as nn
from torch.distributed.device_mesh import init_device_mesh
from torch.distributed.tensor.parallel import parallelize_module
from torch.distributed.fsdp import fully_shard # FSDP2 API

# 0. 初始化 2D DeviceMesh (DP x TP)
# 假设 8卡，2节点，dp_size=4, tp_size=2
mesh_2d = init_device_mesh("cuda", (4, 2), mesh_dim_names=("dp", "tp"))
dp_mesh = mesh_2d["dp"]
tp_mesh = mesh_2d["tp"]

# 1. 实例化模型 (推荐在 meta 设备上初始化以节省内存)
with torch.device("meta"):
    model = MyLargeLanguageModel(config)

# 2. Apply TP (张量并行)
# 将模型权重在 TP Mesh 上切分
tp_plan = get_my_tp_plan() # 自定义或使用预设的 Col/Row Parallel plan
parallelize_module(model, tp_mesh, tp_plan)

# 3. Apply FSDP2 (数据并行与状态分片)
# 在 DP Mesh 上对经过 TP 切分后的模型应用 FSDP2
# 注意：FSDP2 直接操作参数本身 (in-place 转换为 DTensor)，而不像 FSDP1 那样做复杂的 Module 包装
for module in model.modules():
    if isinstance(module, TransformerBlock):
        fully_shard(module, mesh=dp_mesh)
fully_shard(model, mesh=dp_mesh) # 对外层进行分片

# 初始化参数 (将 meta tensor 实例化为真实的 cuda tensor)
model.to_empty(device="cuda")
init_weights_function(model)

# 4. Triton Compile (模型编译)
# 此时模型已经是一个具备完整 2D 拓扑和通信逻辑的分布式模型
model = torch.compile(model, backend="inductor")

# 5. 训练循环...
```

<details>
<summary><b>点击展开：最佳实践与优化细节 (Best Practices)</b></summary>

1. **`device="meta"` 初始化**
   在大模型（如 Llama 70B）训练中，单卡通常无法装下完整的模型。必须首先在 `meta` device 上初始化模型（不占用实际显存），应用 TP 和 FSDP2 规划好显存分配后，再通过 `.to_empty(device="cuda")` 分配显存并加载/初始化权重。这对于 FSDP2 + TP 联合使用至关重要。

2. **FSDP2 与 `torch.compile` 的契合度**
   相比于 FSDP1（`FullyShardedDataParallel` 包装类），FSDP2 采用了基于 `DTensor` 的纯函数式设计。它不会修改 `nn.Module` 的层级结构，这使得 AOTAutograd 能够非常顺畅地追踪到前向和反向传播的图，大幅降低了 `torch.compile` 的图断裂（Graph Breaks）概率。

3. **通信与计算重叠 (Comm-Comp Overlap)**
   为了让 `torch.compile` 的 Inductor 后端更好地将计算与 FSDP2 的 All-Gather / Reduce-Scatter 以及 TP 的 All-Reduce 重叠，可以在编译前注入特定的配置选项：
   ```python
   # 开启 Inductor 对于分布式通信的重叠优化
   torch._inductor.config.overlap_comm = True
   torch._inductor.config.reorder_for_compute_comm_overlap = True
   ```

4. **Activation Checkpointing (GC) 的时序**
   如果你还需要应用激活重计算（Gradient Checkpointing），时序应当是：
   **Model Init -> Apply TP -> Apply GC -> Apply FSDP2 -> Compile**。
   GC 应该在 FSDP 之前应用，以确保 FSDP 能够正确追踪重计算阶段需要 re-prefetch 的权重。

</details>

在编写训练框架时为什么通常遵循以上顺序，如果有 triton kernel 的 autotune 一般放在哪里？

---

### 🤖 Assistant

这个包含 6 个步骤的时序是目前主流大模型训练框架（如 Torchtitan、PyTorch 官方示例等）在 PyTorch 2.x 生态下的**黄金标准**。

这个顺序的核心设计思想是：**先定义单卡/局部的物理结构变化，再定义跨卡的通信逻辑，接着分配真实显存，最后进行全局图编译与优化器绑定。**

以下是每一步必须严格遵循该顺序的深层原因，以及 Triton Kernel Autotune 的正确放置位置。

---

### 一、 为什么必须遵循这个 6 步顺序？

#### 1. `apply TP / EP / SP` (局部形状重塑)
* **原因**：张量并行 (TP)、专家并行 (EP) 和序列并行 (SP) 会直接改变模型的**局部张量形状 (Local Shape)**。例如，一个 $4096 \times 4096$ 的权重矩阵在 `tp_size=4` 时，会被物理切割成 $4096 \times 1024$。
* **时序要求**：必须在最前面执行。因为后续所有的分布式策略、显存分配和编译，都必须基于这个被切分后的真实物理形状来进行。

#### 2. `apply activation checkpoint` (计算图重组)
* **原因**：激活重计算 (AC) 会改变前向 (Forward) 和反向 (Backward) 传播的执行图，以时间换空间。
* **时序要求**：必须在 FSDP2 之前。FSDP2 在反向传播时需要触发 All-Gather 来预取 (Prefetch) 权重。如果先做 FSDP 再做 AC，FSDP 注册的通信 Hook 可能无法正确感知 AC 重算阶段的逻辑，导致权重预取时序错乱或死锁。

#### 3. `apply FSDP2 (fully_shard)` (数据维度分片)
* **原因**：FSDP2 负责在数据并行 (DP) 维度上对参数进行打散 (Sharding)。
* **时序要求**：此时模型已经完成了 TP 的形状切分，并且确定了 AC 的执行图边界。FSDP2 可以基于这些最终信息，精准地计算如何在 DP 组内注册 `DTensor`，并安排 All-Gather 和 Reduce-Scatter 通信算子。

#### 4. `move to device / DTensor init` (显存实例化)
* **原因**：为了防止大模型初始化时 CPU/GPU OOM，前 3 步通常是在 `device="meta"`（仅有形状，不占显存）上完成的。
* **时序要求**：在所有的切分策略 (TP/FSDP2) 都规划完毕后，模型变成了分布式的 `DTensor` 表达。此时调用 `.to_empty(device="cuda")` 并进行正态分布等权重初始化，**每张 GPU 就只会为其真正负责的那一小块参数 (Shard) 分配物理显存**，实现 $O(1)$ 的内存占用。

#### 5. `torch.compile` (全局计算图编译)
* **原因**：AOTAutograd 和 Inductor 需要追踪完整的计算和通信图。
* **时序要求**：必须在参数分配到真实设备 (`cuda`) 之后。因为编译器需要获取真实的硬件信息（如内存对齐、指针信息）。同时，它必须在所有的并行策略 (TP/FSDP2) 注入后执行，这样才能将通信算子（如 TP 的 All-Reduce）和计算算子（如 Matmul）进行**内核融合 (Kernel Fusion)** 和**通信计算重叠 (Comm-Comp Overlap)**。

#### 6. `optimizer` (优化器状态绑定)
* **原因**：优化器需要维护与模型参数一一对应的状态变量（如 Adam 的 $m$ 和 $v$）。
* **时序要求**：必须在 FSDP2 切分并移动到真实 GPU 之后初始化。如果提前初始化，优化器会试图为未切分的完整参数分配状态，导致瞬间 OOM。放在 `compile` 之后也是为了确保优化器绑定的是最终编译图中的参数对象。

---

### 二、 Triton Kernel 的 Autotune 应该放在哪里？

Triton 的 Autotune（无论是由 `torch.compile` 自动生成的，还是用户手写的自定义 Kernel）其本质是**在真实硬件上通过试运行 (Trial Runs) 来寻找最佳的 Block Size、Num Warps 和 Num Stages**。

因此，Autotune 需要真实的输入张量、真实的显存和触发执行。它不属于模型定义阶段，而是属于**执行阶段 (Execution Phase)**。

通常，Autotune 发生在这 6 步之后，具体的做法如下：

#### 7. Warmup / Dummy Step (触发 Autotune)
在第 6 步之后，正式的 `for epoch in epochs` 训练循环之前，框架通常会注入 1 到 3 步的 Warmup Step（使用真实的 dummy data）：

```python
# ... 完成 1 ~ 6 步 ...
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

# 7. 触发 Autotune 和编译 JIT 的 Dummy Step
# 这里是 Triton Autotune 真正发挥作用的地方
print("Starting warmup and autotune...")
for _ in range(3):
    dummy_input = generate_dummy_batch()
    optimizer.zero_grad()
  
    # 第一次 forward/backward 会触发：
    # 1. torch.compile 的 JIT 编译
    # 2. Inductor 生成的 Triton kernel 的 autotune (如 max-autotune 模式)
    # 3. 手写 custom kernel (如 FlashAttention) 的 @triton.autotune
    out = model(dummy_input)
    loss = out.mean()
    loss.backward()
    optimizer.step()

torch.cuda.synchronize()
print("Autotune completed. Ready for training.")

# 8. 正式训练循环
# ...
```

<details>
<summary><b>点击展开：关于 Triton Autotune 的进阶细节</b></summary>

1. **`torch.compile` 的内置 Autotune (Inductor)**
   如果你在编译时开启了 `torch.compile(mode="max-autotune")`，Inductor 会在**第一次执行（Forward/Backward）**时，为每一个生成的 Triton Kernel 生成多套配置，并在后台编译并运行它们以测量耗时，最后选出最优配置缓存下来。这会导致**第一个 Step 极其缓慢**（可能需要几十分钟），但这完全正常。

2. **手写自定义 Triton Kernel (Custom Kernel)**
   如果你在模型定义中使用 `@triton.autotune(configs=[...])` 编写了如自定义的 RMSNorm 或 RoPE。这些 Kernel 的 Autotune 也是在 **第一次被调用时 (First Invocation)** 动态触发的。

3. **Autotune 缓存 (`TRITON_CACHE_DIR`)**
   由于大模型的 Autotune 耗时极长，在分布式训练启动时，为了避免多卡同时编译导致文件系统 IO 爆炸，强烈建议设置 `TORCHINDUCTOR_CACHE_DIR` 和 `TRITON_CACHE_DIR`，或者通过某些机制让 Rank 0 先运行一个 Dummy Step 预热缓存，然后其他节点再读取缓存。

</details>
