# 更换LLM的Tokenizer成本极高

### 🧑‍💻 User

pretrain llm换tokenizer/scale vocab成本很高吗？

---

### 🤖 Assistant

结论是：**如果是在已经训练好的模型基础上换 Tokenizer 或扩充词表（Scale Vocab），成本非常高**；如果是**从零开始（From Scratch）预训练之前更换，核心成本在于数据处理，模型训练成本本身不受影响**。

通常业界问这个问题，大多是指**在已有的开源基座模型（如 LLaMA, Mistral）上扩充特定语言（如中文）的词表**。针对这种最常见的情况，以下是详细的成本拆解：

### 1. 模型参数与结构的破坏（高昂的对齐成本）

大语言模型的输入和输出层与词表大小 $V$ 直接绑定。
*   **Embedding Layer（输入层）**: 维度为 $V \times d$ （$d$ 为隐层维度）。
*   **LM Head（输出层）**: 维度为 $d \times V$。

当你将词表从 $V_{old}$ 扩充到 $V_{new}$ 时，你需要为新增的 $\Delta V$ 个 Token 增加全新的参数。
*   **随机初始化的灾难**：新增加的 Token 的 Embedding 通常只能随机初始化。如果直接用这些随机初始化的向量去激活模型内部经过千锤百炼的 Transformer 层，会产生巨大的噪声。
*   **计算成本**：为了让这批新的 Token “融入”原有的语义空间，你需要进行**继续预训练（Continual Pre-training, CPT）**。这通常需要消耗数百亿（Billion）级别的 Token 数据量和大量的 GPU 算力。例如，Chinese-LLaMA 为了扩充中文词表并对齐，消耗了大量的计算资源。

### 2. 灾难性遗忘的风险（算法与实验成本）

在重新训练新词表时，模型极易出现**灾难性遗忘（Catastrophic Forgetting）**。
*   如果你只用新语言（如纯中文）的数据去训练新 Token，模型会迅速遗忘原有的通用能力（如代码能力、英文逻辑能力）。
*   **混合数据的成本**：为了避免遗忘，你必须按精确的比例混合原有语料（如英文/代码）和新语料。这需要大量的消融实验（Ablation Study）来寻找最佳的数据配比（Data Mix），这背后的实验算力成本极高。

### 3. 数据重构成本（工程与 I/O 成本）

更换 Tokenizer 意味着你过去所有处理好的、缓存的预训练数据全部作废。
*   **重新 Tokenize**：面对 PB 级别的高质量语料，使用新的 Tokenizer 重新跑一遍切词流程，不仅耗费巨量的 CPU 时间和集群计算资源，还需要重新进行存储和分发。对于小团队来说，仅清洗和重新 Tokenize 数据就是一道门槛。

<summary><b>💡 深入技术细节：业界如何降低扩充词表的成本？（点击展开）</b></summary>

为了降低在现有模型上扩充词表的训练成本，业界通常采用以下“抢救”和优化策略：

1. **启发式初始化（Heuristic Initialization）**：
   不使用随机初始化，而是利用原有词表。例如，新增了一个词 `自然语言`，可以用旧词表中 `自然`、`语`、`言` 这几个 Subword 的 Embedding 取平均值，作为新词 `自然语言` 的初始 Embedding。这样可以大幅缩短收敛所需的步数。

   数学表示上，若新 Token $t_{new}$ 可由旧 Token 序列 $(t_1, t_2, ..., t_k)$ 组成，则其初始权重 $E(t_{new})$ 为：
   $$E(t_{new}) = \frac{1}{k} \sum_{i=1}^{k} E_{old}(t_i)$$

2. **冻结骨干网络（Freeze Backbone）**：
   在继续预训练的早期，**冻结** Transformer 的核心层（Attention 和 FFN），**只训练** Embedding 层和 LM Head。等新词的向量空间基本对齐后，再解冻全局参数进行全量微调（Full Fine-tuning）。这不仅节省了显存，还能有效防止灾难性遗忘。

3. **词表大小的权衡（Trade-off in $V$）**：
   词表越大，单句生成的 Token 数越少，推理速度越快；但词表过大（如超过 100k）会导致 Embedding 和 LM Head 占据模型过大的参数比例（在几十亿参数的小模型中尤为明显），这会导致训练极其困难。因此，并非词表越大越好。
</details>

### 总结

*   **如果不差钱/从零预训练**：换 Tokenizer 成本很低（只需花几个小时训练一个 SentencePiece 模型并重新处理数据）。
*   **如果在开源模型上二次开发**：**成本非常高**。你不仅要承担昂贵的 GPU 算力来进行 Continued Pre-training，还要投入大量的高级算法工程师精力去解决灾难性遗忘和权重初始化问题。这也是为什么很多垂直领域微调（SFT）宁可忍受原版 Tokenizer 在特定语言上分词效率低，也不愿轻易扩充词表的原因。
