# 算子融合降低显存和启动开销

### 🧑‍💻 User

fused kernel既降低激活显存又能减少kernel launch overhead吗？

---

### 🤖 Assistant

**是的，完全正确。** 算子融合（Fused Kernel）在深度学习推理和训练优化中，最核心的两个直接收益正是**降低激活显存（Activation Memory）**和**减少内核启动开销（Kernel Launch Overhead）**。

下面为您详细拆解它是如何做到这两点的：

### 1. 如何降低激活显存 (Reducing Activation Memory)

在没有算子融合的情况下，深度学习框架会按照计算图的节点依次执行 `kernel`。
*   **未融合前：** 假设有计算流程 `A -> B -> C`。当 GPU 计算完 `A` 后，必须将中间结果（激活值 / Activation）写回到 GPU 的全局显存（`HBM`）中。接着，`B` 再从 `HBM` 中读取这个中间结果进行计算，并再次将结果写回 `HBM`。如果是训练阶段，这些中间结果通常还需要保留在显存中，用于反向传播计算梯度。这会导致极大的显存占用。
*   **融合之后：** `A`、`B`、`C` 被编译成一个单独的 `Fused Kernel`。GPU 在计算完 `A` 后，结果直接保存在片上内存（如寄存器 `Registers` 或共享内存 `SRAM`）中，紧接着立刻被 `B` 拿来计算。中间结果根本不需要被写入全局显存 `HBM`，用完即焚（或者直接在寄存器中求导）。

**结论：** 避免了中间状态的显存分配，大幅降低了峰值显存占用（Peak Memory Footprint），这也是 FlashAttention 等融合算子能支持超长上下文的核心原因之一。

### 2. 如何减少 Kernel Launch Overhead

`Kernel launch overhead` 是指 CPU 向 GPU 发送指令以启动一个计算任务所花费的时间。这个过程涉及 CPU 侧的 API 调用、驱动层处理以及通过 PCIe 总线向 GPU 发送指令。
*   **未融合前：** 如果模型中存在大量计算量极小、执行时间极短的算子（例如 Element-wise 的加法、ReLU、LayerNorm 等），GPU 执行这些计算可能只需要几微秒，但 CPU 启动这个 `kernel` 可能就需要十几微秒。这会导致 **GPU Starvation（GPU 饥饿）**——GPU 大部分时间都在等待 CPU 的启动指令，处于闲置状态。启动 $N$ 个算子，就需要承担 $N$ 次 Overhead。
*   **融合之后：** 将多个小算子合并为一个复杂的 `kernel`。CPU 只需要下发 $1$ 次启动指令。

**结论：** 启动开销从 $O(N)$ 降低到了 $O(1)$，显著提升了 GPU 的实际利用率（Utilization），特别是在小 Batch Size 或小模型推理时效果极佳。

除了您提到的两点，算子融合还有其他关键影响：

#### 隐形的最大收益：突破内存墙（Memory Bound）
其实降低激活显存的副产品是**节省了显存带宽（Memory Bandwidth）**。
很多轻量级算子（如激活函数）是 Memory Bound（访存密集型）而非 Math Bound（计算密集型）。它们的主要时间花在从 `HBM` 读写数据上。通过融合算子将数据留在寄存器中，彻底省去了中间 tensor 从 `HBM` 的 `Read` 和 `Write` 过程，这通常能带来几倍甚至十几倍的执行速度提升。

#### 代码逻辑对比示例
```python
# 未融合（伪代码）：需要启动3次kernel，读写3次HBM
intermediate1 = cuda_add(x, y)      # Launch 1, Write HBM
intermediate2 = cuda_mul(intermediate1, z) # Launch 2, Read/Write HBM
output = cuda_relu(intermediate2)   # Launch 3, Read/Write HBM

# 融合后（伪代码）：只需启动1次kernel，只在开头读HBM，结尾写HBM
output = fused_add_mul_relu(x, y, z) # Launch 1
```

#### 算子融合的潜在挑战（为什么不能把整个网络融合成一个 Kernel？）
1.  **寄存器压力 (Register Pressure)：** GPU 上的寄存器数量是有限的。如果融合的算子逻辑过于复杂，需要的变量过多，会导致寄存器溢出（Register Spilling），数据被迫存入更慢的本地内存，反而严重拖慢速度。
2.  **开发成本极高：** 手写复杂的 CUDA 融合算子非常困难，需要极高水平的 GPU 架构知识。目前业界通常依赖编译器自动融合（如 `TorchInductor`, `XLA`, `TensorRT`）或使用更高级的语言（如 OpenAI 的 `Triton`）来降低编写融合算子的难度。
3.  **同步屏障 (Synchronization)：** 如果前一个算子需要全局数据聚合（如 Softmax 的分母需要所有元素参与），往往很难与后面的算子简单地进行 Element-wise 融合。

liger fused celoss kernel/liger rms/swiglu fused kernel/ligerfusedmoefunction(LigerExperts)是不是很好的体现了这点？

---

### 🤖 Assistant

**是的，非常精准！** 您提到的 `Liger Kernel`（目前在 LLM 训练圈非常火的轻量级高性能 Triton 算子库）中的这些核心算子，**简直是体现“降低激活显存”和“减少 Launch Overhead”的最佳教科书级案例。**

Liger Kernel 的核心设计哲学就是针对大语言模型（如 LLaMA, Qwen 等）中那些非矩阵乘法（Non-GEMM）的“胶水算子”和“瓶颈算子”进行深度融合。

我们逐一拆解这些算子是如何完美印证这两大优势的：

### 1. Liger Fused Cross Entropy Loss Kernel (融合交叉熵)
在 LLM 的训练中，交叉熵损失函数通常是**显存杀手**，因为它的计算涉及词表大小（Vocabulary Size, $V$），通常 $V$ 都在 32k 到 128k 甚至更大。

*   **传统做法（PyTorch原生）：**
    计算逻辑是 $Logits \rightarrow Softmax \rightarrow \log \rightarrow NLLLoss$。
    为了反向传播，PyTorch 必须在全局显存中实例化并保存 $B \times S \times V$ （Batch $\times$ SequenceLength $\times$ VocabSize）大小的 Logits 和 Softmax 概率矩阵。假设 $B=4, S=4096, V=128000$，这一个中间变量就能吃掉好几个 GB 的显存。
*   **Liger Fused CE Loss：**
    借鉴了 FlashAttention 的在线 Softmax 思想（Online log-sum-exp）。它在 Triton kernel 内部通过分块（Tiling）直接在前向计算 Loss，并在同一个 Kernel 里**直接把梯度（Gradients）算出来**。
    *   **显存收益：** **极其巨大**。因为直接输出了最终的 Loss 和输入 Logits 的梯度，彻底消灭了 $B \times S \times V$ 的中间激活状态，显存占用断崖式下降。
    *   **Overhead收益：** 将 Softmax、Log、NLL 等多个算子合并为 1 个。

### 2. Liger RMSNorm (融合均方根归一化)
RMSNorm 是 LLM 中使用最频繁的操作之一（每个 Transformer 层至少前后各用一次）。

*   **传统做法：**
    $x \rightarrow x^2 \rightarrow mean \rightarrow rsqrt \rightarrow x \cdot rsqrt \cdot weight$。这在 PyTorch 层面会触发至少 4~5 个基础算子的 Kernel Launch。同时，为了反向传播，需要保存大量的中间结果。
*   **Liger RMSNorm：**
    *   **Overhead收益：** 把上述 5 个小算子全部塞进一个 Triton Kernel 中，Launch Overhead 从 5 次降到 1 次。
    *   **带宽与显存收益：** 把数据从 HBM 读到 SRAM 后，在 SRAM 内完成求平方、求和、开根号、乘权重等所有动作，最后只把结果写回 HBM。不仅省了中间变量的激活显存，还彻底突破了 Memory Bound（显存带宽瓶颈）。

### 3. Liger SwiGLU Fused Kernel (融合激活函数)
SwiGLU 的数学表达式通常是 $Output = Swish(X_1) \otimes X_2$。

*   **传统做法：**
    计算过程中会产生 $Swish(X_1)$ 这样的庞大中间张量（大小与隐藏层维度一致，通常是 10k+ 级别），且必须保留在 HBM 中用于反向传播时计算梯度。
*   **Liger SwiGLU：**
    *   **显存收益：** 融合 Kernel 在反向传播时，**通过重计算（Recomputation）或直接推导解析解**的方式，不需要保存前向的中间激活值。它只需要最原始的输入就能在同一个 Kernel 内完成梯度的计算。
    *   **Overhead收益：** 将乘法、Sigmoid、点乘等组合操作合并，减少了 GPU 闲置等待 CPU 发送指令的时间。

### 4. Liger FusedMoEFunction (LigerExperts - 融合混合专家)
MoE 架构（如 Mixtral, Qwen-MoE）的痛点在于“碎”。

*   **传统做法：**
    一个 Token 进来，需要先被 Router 分配给不同的 Expert。如果有 8 个 Expert，传统的原生实现可能会循环调用 8 次小的矩阵乘法（或线性层），中间伴随大量的 `gather`（数据收集）和 `scatter`（数据分发）操作。不仅 Launch Overhead 极高（几十次启动），还会产生大量的数据搬移碎片。
*   **Liger Fused MoE：**
    采用了 Grouped GEMM（分组矩阵乘法）或深度融合的 Dispatch/Combine 逻辑。
    *   **Overhead收益：** 用一个超级 Kernel（或者极少量的 Kernel）搞定所有的路由分发、专家计算和结果聚合。极大地缓解了 GPU 饥饿问题。
    *   **显存收益：** 避免了因为动态路由而在 HBM 中频繁申请和释放不规则大小的临时张量。

<summary><b>💡 总结：为什么 Liger Kernel 能让训练 Batch Size 翻倍？（点击展开）</b></summary>

在 LLM **训练（Training）**场景下，显存的消耗大头从来不是模型权重本身，而是**反向传播所需的激活状态（Activation Memory）**。

Liger Kernel 通过上述这些 Fused Kernels 达到了一个核心目的：**“阅后即焚”**。
只要能用数学公式在一个 Kernel 内部把梯度直接推导出来，就坚决不把中间张量写到全局显存 HBM 里去。

这种对“激活显存”和“显存带宽”的极致压榨，使得在同样的硬件（如 A100 80G）上，原本只能跑 $BatchSize = 2$ 的训练，使用 Liger 后往往能跑起 $BatchSize = 4$ 或更大，同时吞吐量（Tokens/sec）也因为 Launch Overhead 和带宽占用的减少而显著提升。您的理解非常深刻且完全打中了当前 LLM 优化的核心命脉！
</details>
