# 无特殊标记时工具调用检测机制

### 🧑‍💻 User

当llm的tokenizer没专门为toolcall注册spec token时，推理引擎是怎么检测到模型发起toolcall的？

---

### 🤖 Assistant

当大语言模型（LLM）的 Tokenizer 中没有专门为 Tool Call（函数调用）注册 Special Tokens（如 `<tool_call>`、`<|function_start|>` 等）时，**模型本质上只是在做普通的文本补全（Text Generation）**。

在这种情况下，推理引擎（或上层的 Agent 框架，如 LangChain、LlamaIndex）检测并执行 Tool Call，主要依赖于**“格式约定 + 文本解析 + 停止词阻断”**的组合机制。

以下是推理引擎检测模型发起 Tool Call 的核心步骤和原理：

### 1. 强依赖 System Prompt 约定输出格式
推理引擎首先会在 System Prompt 中注入一段硬性规定，教导模型在需要调用工具时，必须输出**特定格式的纯文本**（通常是 JSON、XML 或特定的 Markdown 结构）。

最著名的例子就是 **ReAct (Reasoning and Acting)** 框架。引擎会告诉模型：
*"当你需要调用工具时，请严格按照以下格式输出："*
```text
Thought: [你的思考过程]
Action: [工具名称]
Action Input: [JSON格式的参数]
```

### 2. 使用“停止词”（Stop Sequences）阻断生成
这是最关键的一步。由于没有 Special Token 来告诉引擎“我已经生成完工具调用了”，引擎必须通过**停止词（Stop Words）**来及时打断模型。

在 ReAct 格式中，引擎通常会在推理时传入一个 `stop=["Observation:"]` 的参数。
*   模型输出 `Action Input: {...}` 后，习惯性地会想要自己生成 `Observation:`（观察结果）。
*   当引擎在输出流中检测到 `Observation:` 这个字符串（或对应的 Tokens）时，会**立即强制中断推理**。
*   中断后，引擎就知道：**“模型已经把工具名称和参数生成完毕了，现在该我接管了。”**

### 3. 实时文本解析与正则匹配（Parsing & Regex）
当生成被停止词打断后，推理引擎会对刚才生成的纯文本缓冲（Text Buffer）进行后处理。引擎会使用正则表达式（Regex）或字符串匹配来提取关键信息：

<details>

假设模型生成了以下文本：
```text
Thought: 我需要查一下北京的天气。
Action: get_weather
Action Input: {"location": "Beijing"}
```
推理引擎会使用类似下面的逻辑进行提取（伪代码）：
```python
import re

output_text = model.generate(...) # 生成被停止词打断

# 提取工具名称
action_match = re.search(r"Action:\s*(.*)", output_text)
tool_name = action_match.group(1).strip() if action_match else None

# 提取参数
input_match = re.search(r"Action Input:\s*(.*)", output_text, re.DOTALL)
tool_arguments = input_match.group(1).strip() if input_match else None

if tool_name and tool_arguments:
    # 成功解析到 Tool Call！
    execute_tool(tool_name, json.loads(tool_arguments))
```
</details>

### 4. 强制约束解码（Constrained Decoding / Guided Generation）
对于没有 Special Token 的模型，纯靠 Prompt 很容易出现格式幻觉（比如漏掉括号、JSON 格式错误导致解析失败）。现代推理引擎（如 vLLM, llama.cpp, TGI）通常会使用**约束解码（Constrained Decoding）**技术来保证模型必然输出合法的 Tool Call 格式。

*   **原理**：引擎将工具的参数结构（JSON Schema）转换为**有限状态机（FSM）**或**上下文无关文法（CFG）**。
*   **Logits 掩码（Logits Masking）**：在模型自回归生成下一个 Token 时，引擎会动态修改输出的概率分布（Logits）。如果不符合 JSON 语法的 Token，其概率会被强制设为 $-\infty$（或者极小值）。
*   **结果**：即使模型没有特殊的 Tool Call Token，它也被“按着头”强行生成了一段完美的 JSON 字符串，推理引擎只需直接 `json.loads()` 即可。

### 总结

当没有专属 Spec Token 时，工作流如下：

1.  **输入阶段**：引擎把工具列表转换成文本说明，塞进 Prompt。
2.  **生成阶段**：引擎设置 `stop` 参数（如 `\nObservation:`、`</tool>` 等）。引擎可能开启 JSON 约束解码。
3.  **阻断阶段**：模型输出完工具名和参数，触发了 `stop` 词，推理引擎掐断生成。
4.  **解析阶段**：引擎用正则或 JSON 解析器把文本切开，提取出 `tool_name` 和 `args`，这就是一次成功的 Tool Call 检测。

虽然这种纯文本约定的方式也能工作（早期的 GPT-3.5 和 GPT-4 都是这么做的），但它消耗了更多的上下文 Token（用于解释格式），且解析失败的概率相对较高。因此，现在的新模型（如 GPT-4o、Qwen2.5、Llama-3.1）都倾向于在 Tokenizer 层级引入原生的 `<tool_call>` 等 Special Tokens 来提高稳定性和效率。
