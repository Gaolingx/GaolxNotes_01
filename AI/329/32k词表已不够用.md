# 32k词表已不够用

### 🧑‍💻 User

llama 32k词表对于现在模型来说够用吗？

---

### 🤖 Assistant

简短的回答是：**在当前的大语言模型（LLM）发展趋势下，32k 的词表大小已经“不够用”了**。

尤其是对于多语言（特别是中文）、代码生成和长上下文场景，32k 的词表显得捉襟见肘。事实上，连 Meta 自己的 Llama 3 都已经将词表大幅扩充到了 128k。

以下是为什么 32k 词表在现在看来已经不够用的核心原因，以及当前业界的标准。

### 1. 多语言和中文编码效率极低
Llama 1 和 Llama 2 使用的 32k 词表主要是为**英语**设计的。在处理非英语（如中文）时，它通常会回退到字节级（Byte-level）编码。
* **低效的 Token 转化率 (Fertility Rate)：** 在 32k 词表中，一个汉字可能被拆分成 2 到 3 个 token，一个中文词语可能需要 4-6 个 token。而在专门优化过中文的词表中（如 Qwen 或 ChatGLM），一个中文词语通常只需要 1 个 token。
* **结果：** 表达同样的中文意思，32k 词表的模型消耗的 token 数量是其他模型的 $2$ 到 $3$ 倍。

### 2. 严重挤压上下文窗口 (Context Window)
现在模型动辄支持 128k 甚至 1M 的上下文长度。
假设一个模型支持 8k token 的上下文：
* 如果使用 100k+ 的大词表，这 8k token 可能能装下 **6000 个汉字**。
* 如果使用 Llama 的 32k 词表，这 8k token 可能只能装下 **2500 个汉字**。
词表越小，信息的“压缩率”就越低，导致实际能处理的文本长度大幅缩水。

### 3. 推理速度变慢，成本变高
大模型的生成是自回归的（Autoregressive），即一次生成一个 token。
* 如果生成一句中文问候，大词表模型只需预测 5 次（生成 5 个 token）；
* 32k 词表的 Llama 可能需要预测 15 次（生成 15 个 token）。
这直接导致**推理速度（Tokens per second）变慢**，且如果调用 API，用户需要为更多的 token 支付**更高的成本**。

---

### 当前主流模型的词表大小对比

现代模型为了追求更高的压缩率和多语言能力，普遍采用了极大的词表：

| 模型系列 | 词表大小 (Vocabulary Size) | 备注 |
| :--- | :--- | :--- |
| **Llama 1 / 2** | `32,000` | 偏向纯英文，中文编码效率极低 |
| **Yi 系列** | `64,000` | 中英双语优化，平衡了参数量与编码效率 |
| **GPT-4 (cl100k_base)**| `~100,000` | 现代多语言模型的基准线 |
| **Llama 3** | `128,256` | Meta 放弃了 32k，转向大词表以支持多语言和代码 |
| **Qwen (通义千问)** | `151,936` | 对中文、代码和多语言压缩率极高 |
| **Gemma 2** | `256,000` | 极大的词表，极高的文本压缩率 |

<summary><strong>💡 扩展阅读：既然大词表这么好，为什么一开始 Llama 要用 32k？（点击展开）</strong></summary>

尽管大词表在推理和压缩上有优势，但它并非没有代价，这也是早期模型（包括许多学术模型）选择 32k 的原因：

1. **参数占用问题：** 模型的 Embedding 层和最后的输出映射层（LM Head）大小与词表直接挂钩。假设隐藏层维度为 $d = 4096$，词表大小为 $V$。
   * 如果 $V = 32,000$，这部分参数量约为 $130M$（1.3亿）。
   * 如果 $V = 150,000$，这部分参数量将飙升至 $614M$（6.1亿）。
   对于一个 7B 的模型来说，大词表会占据近 10% 的参数量，这些参数原本可以用来加深模型的 Transformer 层数。
2. **长尾词语的训练难度：** 词表越大，意味着包含大量生僻词（Rare tokens）。在训练数据中，这些生僻词出现的频率极低，导致它们对应的 Embedding 向量往往训练不充分（Under-trained），反而可能影响模型表现。
3. **纯英语场景足够：** 如果只看纯英语（English-only）任务，32k 的 BPE 词表已经能对英文单词和子词进行非常高效的压缩，扩充词表带来的边际收益递减。

</details>

### 总结
对于**纯英文**、且**参数量极小**的特定领域微调模型，32k 词表勉强够用。
但对于**现在（2024年及以后）的通用大模型**，特别是需要处理中文、代码、多语言以及长文本上下文的模型来说，**32k 词表已经被淘汰**。100k 到 150k 的词表大小已经成为当前 SOTA (State-of-the-Art) 模型的标配。
