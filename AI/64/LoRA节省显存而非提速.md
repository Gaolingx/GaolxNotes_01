# LoRA节省显存而非提速

### 🧑‍💻 User

为什么lora不能大幅提高训练速度？

---

### 🤖 Assistant

这是一个非常经典且容易被误解的问题。很多人看到 LoRA 将“可训练参数量”降低了 90% 甚至 99% 以上，就会直觉地认为训练速度也应该成倍提升。

但实际上，**LoRA 的核心贡献是“大幅节省显存（Memory）”，而不是“大幅减少计算量（FLOPs）”**。因此，它不能按比例大幅提高单步训练的计算速度。

具体原因可以从以下几个维度来理解：

### 1. 前向传播（Forward Pass）计算量没有减少
在 LoRA 中，原始的预训练权重矩阵 $W_0$ 被冻结，旁边并联了两个低秩矩阵 $A$ 和 $B$。前向传播的公式为：

$$h = W_0 x + \Delta W x = W_0 x + B A x$$

*   **大矩阵乘法依然存在：** 尽管 $A$ 和 $B$ 的参数量极小，但模型依然需要计算 $W_0 x$。这意味着模型中所有原本庞大的矩阵乘法（计算密集型任务）一个都没有少。
*   **计算量反而微增：** 因为额外增加了 $B A x$ 的计算，前向传播的总浮点运算次数（FLOPs）实际上是**略微增加**的。

### 2. 反向传播（Backward Pass）依然需要穿过大矩阵
这是最核心的计算学原因。在反向传播中，我们需要计算梯度。
虽然我们不需要计算原始权重 $W_0$ 的梯度（即不需要算 $\frac{\partial L}{\partial W_0}$），但根据链式法则，为了把误差信号反向传递给上一层，我们**依然需要计算输入 $x$ 的梯度**：

$$\frac{\partial L}{\partial x} = W_0^T \frac{\partial L}{\partial h} + A^T B^T \frac{\partial L}{\partial h}$$

这就意味着，反向传播时，**依然需要用庞大的 $W_0^T$ 去乘以梯度矩阵**。
在标准的神经网络训练中，反向传播的计算量大约是前向传播的 2 倍。LoRA 仅仅省去了求权重梯度的部分（大约占反向传播总计算量的 1/3 到 1/2），但最耗时的误差回传计算并没有省去。

### 3. 参数更新（Optimizer Step）占总时间的比例很小
LoRA 确实大幅减少了需要更新的参数量。对于全量微调，优化器（如 AdamW）需要读取、更新并写回所有的权重、动量（Momentum）和方差（Variance），这会消耗大量时间。

但是，在庞大的 Transformer 模型中，**矩阵乘法（MatMul）才是消耗 GPU 计算时间的大头**。优化器的参数更新步骤虽然受限于显存带宽（Memory Bound），但其耗费的绝对时间远小于前向和反向传播中的矩阵乘法。因此，优化器步骤变快了，对总体训练速度的提升并不明显。

---

<details>
<summary><b>💡 既然不能大幅提速，为什么有时候感觉 LoRA 训练确实快了很多？</b></summary>
<br>
虽然 LoRA <b>单步计算时间（Step Time）</b>没有大幅缩短（通常只快 10%~25% 左右），但在实际工程应用中，我们经常会感觉 LoRA 训练效率高很多，这主要是因为“显存红利”带来的间接加速：

1. **可以成倍增加 Batch Size：**
   因为 LoRA 免去了保存全量优化器状态的显存开销（这通常是模型本身大小的 2-3 倍），省下来的显存可以用来容纳更大的 Batch Size。Batch Size 变大后，GPU 的并行计算利用率更高，**整体吞吐量（Tokens/second）**大幅提升，从而缩短了跑完一个 Epoch 的总时间。
2. **多卡通信开销大幅降低：**
   在多卡分布式训练（如 DDP）中，每一步都需要在 GPU 之间同步梯度（AllReduce）。全量微调需要同步几百 GB 的梯度数据，而 LoRA 只需要同步几十 MB。通信时间的急剧减少，使得多卡训练的效率近乎线性增长。
3. **避免了频繁的显存交换（Offloading）：**
   如果没有 LoRA，全量微调可能因为显存不足而不得不使用 CPU Offloading（把部分数据扔给慢速的系统内存）。这种 PCIe 数据传输极其缓慢。LoRA 使得模型能完整塞进显存，避免了这种极其拖慢速度的情况。

</details>

**总结：**
LoRA 的本质是用“微小的额外计算”换取“巨大的显存节省”。参数量减少了 99%，只意味着**“存储需求”**减少了 99%，并不意味着**“数学计算量”**减少了 99%。真正的计算瓶颈依然在于原始庞大权重的矩阵乘法。

### 🧑‍💻 User

当我们使用peft情况下，可以关掉梯度检查点加速训练吗？

---

### 🤖 Assistant

**一句话结论：是的，关掉梯度检查点（设置 `gradient_checkpointing=False`）确实可以加速训练，但前提是你的显卡显存（VRAM）足够大，否则会直接 OOM（显存溢出）。**

这是一个经典的**“用空间（显存）换时间（计算速度）”**的取舍问题。为了让你更好地做出决定，我们需要理清 PEFT（如 LoRA）和梯度检查点（Gradient Checkpointing）在显存管理上的不同分工。

### 1. 为什么关掉梯度检查点会变快？
梯度检查点（Gradient Checkpointing, 简称 GC）的作用是**节省显存**。
在正常的前向传播中，神经网络会把每一层的中间激活值（Activations）都保存在显存里，以便在反向传播时用来计算梯度。对于深层的大模型，这些激活值会占用极其恐怖的显存。

开启 GC 后，系统会丢弃掉大部分中间激活值，只保留少量的“检查点”。当反向传播需要用到这些被丢弃的激活值时，**模型会重新进行一次前向计算来临时生成它们**。
*   **开启 GC：** 显存大幅降低，但因为要重复计算前向传播，训练速度通常会**变慢 20% ~ 30%**。
*   **关闭 GC：** 不再需要重复计算，反向传播直接读取显存里的激活值，**训练速度显著提升**。

因此，单从计算速度上讲，关掉它是能加速的。

### 2. 为什么在 PEFT/LoRA 下，我们往往还是必须开启 GC？
这里有一个极易踩坑的误区：**“既然 LoRA 已经省了那么多显存了，我应该就有显存关掉 GC 了吧？”**

答案通常是：**并没有。** 因为 LoRA 和 GC 节省的是**两块完全不同的显存**：

*   **LoRA 节省的是“模型状态显存”：** 它免去了庞大的全量梯度（Gradients）和优化器状态（Optimizer States，如 AdamW 的动量和方差）。
*   **GC 节省的是“激活值显存”（Activation Memory）：** 激活值的大小与模型的参数量关系不大，而是与 **批次大小（Batch Size）** 和 **序列长度（Sequence Length, $L$）** 强相关。

在使用 PEFT 微调 LLM 时，由于我们通常会输入很长的文本（比如 $L = 2048$ 或 $4096$），前向传播产生的激活值依然非常庞大。**LoRA 无法减少这部分激活值的显存占用。**

如果你关掉 GC，长序列带来的激活值会瞬间撑爆显存。

---

<details>
<summary><b>🛠️ 实战建议：我到底该开还是关？</b></summary>
<br>
这完全取决于你的硬件条件和训练配置。你可以按照以下步骤来决定：

1. **先尝试关闭（追求最大速度）：**
   在你的训练脚本中设置 `gradient_checkpointing=False`，然后启动训练。
2. **观察显存占用：**
   使用 `watch -n 1 nvidia-smi` 紧盯显存。
   * 如果模型跑起来了，而且显存还有富裕，恭喜你！保持关闭，享受 20%~30% 的速度提升。
   * 如果程序直接崩溃报 `RuntimeError: CUDA out of memory`，说明你的显存吃不消完整的激活值。
3. **妥协方案（如果 OOM）：**
   * **方案 A（推荐）：** 乖乖把 `gradient_checkpointing=True` 打开。虽然慢了一点，但至少能跑。
   * **方案 B：** 保持关闭 GC，但调小 `batch_size` 或截断 `max_seq_length`。但这通常不划算，因为减小 Batch Size 会降低 GPU 并行效率，反而可能让整体训练耗时更长；减小序列长度则可能损害模型的上下文学习能力。

</details>

**总结：**
如果你有 80G A100 这样的顶级显卡，且微调的序列长度不长（例如 $L=512$），你完全可以关掉梯度检查点来加速 PEFT 训练。但如果在消费级显卡（如 24G 的 4090/3090）上微调长文本，**PEFT + 梯度检查点** 依然是保证模型能跑起来的“黄金搭档”。
