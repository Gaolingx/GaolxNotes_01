# 判断LoRA适配器容量是否耗尽

### 🧑‍💻 User

在针对llms进行sft微调的时候，通常使用高rank+all linear进行lora peft，如何根据metrics判断适配器容量是否耗尽？

---

### 🤖 Assistant

在使用 LoRA 进行 SFT 时，判断适配器容量是否“耗尽”是一个关键的工程问题。当使用高 `rank` 且目标模块设为 `all linear`（或 `all modules`）时，你需要通过以下**三类指标**构建判断体系：

不要仅依赖 Loss，当 Loss 还在下降但其他指标已饱和时，容量可能已满。

### 1. 核心判断指标：信息增益与奇异值分布

这是从数学本质上判断适配器是否“学会了该学的”。

-   **指标 A：奇异值热力衰减（SVD Analysis）**
    -   **判断逻辑**：对 LoRA 的 $A$ 和 $B$ 矩阵的乘积 $\Delta W = BA$ 进行奇异值分解。如果适配器容量充足，奇异值谱会呈现长尾分布（有大量非零的显著奇异值）。如果容量耗尽，奇异值谱会迅速衰减。
    -   **操作**：
        1. 分别取训练 1K steps 和 10K steps 的 checkpoint。
        2. 计算 `U, S, V = torch.svd(delta_w)`。
        3. 观察 `S`：如果训练后期，有效奇异值的数量不再增加，或新增的奇异值趋近于 0，说明**容量的物理维度已满**。
    -   **代码片段**：
        ```python
        # 计算解释方差比
        explained_variance = (S**2) / torch.sum(S**2)
        cumulative_var = torch.cumsum(explained_variance, dim=0)
        # 如果 rank=128，但前 32 个奇异值解释了 99% 的方差，说明容量过剩
        # 如果 rank=128，需要 100+ 个奇异值才能解释 99%，且 loss 还在降，说明容量即将耗尽
        ```

-   **指标 B：适配器更新量范数（$||\Delta W||_F$）**
    -   **判断逻辑**：衡量引入 LoRA 后对原始权重的扰动大小。如果容量耗尽，模型为了拟合新数据，$\Delta W$ 的范数会急剧膨胀（意味着强行扭曲原始空间）。
    -   **阈值参考**：监控 $||\Delta W||_F / ||W||_F$ 的比值。如果该比值在训练后期呈现**指数级增长**或超过 $1$ - $10\%$，可能是适配器在挣扎着解决其容量无法解决的问题。

### 2. 性能饱和指标：Loss 与 Eval 的背离

这是最直观但容易误判的指标。

-   **指标 C：梯度相似度（Gradient Similarity / SNR）**
    -   **判断逻辑**：在容量充足时，不同 Batch 之间的梯度方向在适配器核心子空间内应高度一致。当容量耗尽，模型为了兼顾所有样本，梯度方向开始打架。
    -   **操作**：每隔 N 步计算当前 Batch 梯度与历史平均梯度在投影空间内的余弦相似度。
    -   **判断**：如果相似度从 $>0.8$ 骤降至 $<0.3$ 并持续震荡，说明适配器已无法找到同时满足所有数据的低秩空间。

-   **指标 D：Train Loss 降速与 Eval Perplexity 平台期**
    -   **判断逻辑**：高 Rank 通常容易过拟合，但容量耗尽则不同。容量耗尽表现为训练 Loss 仍顽固高于理想基线（如 1.0），而 Eval Loss 已不再下降。
    -   **关键动作**：对比 Full Fine-tuning 或大 Rank 实验的收敛下限。
    -   **判断**：如果你把 Rank 从 128 提升到 512，Eval 分数毫无变化，说明 128 的容量不是瓶颈；反之则说明 128 已耗尽。

### 3. 参数空间诊断：Intrinsic Dimension（本征维度）

这是最严谨的“容量”测量方法。

-   **方法论**：
    1. 在训练完成后，计算当前数据在 LoRA 子空间内的本征维度。
    2. 如果你使用的是 Rank 128，但该任务的本征维度只需要 40，说明容量过剩（可通过 SVD 压缩）。
    3. 如果本征维度计算出来大于 128（例如 150），说明**容量被硬截断**，适配器已耗尽。
-   **简易替代法**：
    -   使用 LoRA-XS 或类似思想，引入一个小的秩探测头。如果在训练结束时，该探测头的核心维数始终维持在 `rank` 的上限边界，说明容量不足。

### 4. 针对 "All Linear" 的模块级诊断

当你对 **All Linear** 做微调时，绝不能只看全局 Loss。不同层的适配器容量需求差异巨大。

-   **层间方差分析（Inter-layer Variance）**：
    -   **判断逻辑**：如果容量耗尽，某些层（通常是 Attention 的 $V$ 矩阵或 FFN 的 Down 层）的 $\Delta W$ 会出现极端的离群值。
    -   **操作**：绘制所有线性模块的 $||\Delta W||_F$ 柱状图。
    -   **判断**：如果后几层适配器更新量接近于 0，而前几层更新量爆表，说明**总容量虽大，但分配不均导致局部耗尽**。此时不应加总 Rank，而应调整 `target_modules` 分配策略。

### 总结：决策流程图

你可以按照以下逻辑链来判断：

1.  **看分布**（SVD）：如果累积方差达到 $95\%$ 所需的奇异值数量 $ k \ll \text{rank}$，说明**容量过剩**。
2.  **看趋势**：如果 $k \approx \text{rank}$，且训练 Loss 仍在稳步下降，说明 **容量接近极限**。
3.  **看出力**：如果 $k \approx \text{rank}$，且 Eval 先降后升（过拟合）或纹丝不动，而 Train Loss 还在降，说明**容量耗尽且正在死记硬背**。

**一个常被忽略的真相**：对于高 Rank + All Linear 组合，适配器容量耗尽通常不是 LoRA 的矩阵大小不够，而是优化器（AdamW）的**二阶动量无法在受限的低秩空间内找到最优点**。如果发现容量耗尽，尝试将 LoRA 的学习率提升 2-3 倍，往往比单纯加大 Rank 更有效。
