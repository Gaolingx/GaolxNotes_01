# 显存够用时不建议用梯度检查点

### 🧑‍💻 User

多卡训练中，为什么能优先使用fsdp/deepspeed zero2/3的场景，不建议使用gradient checkpoint？

---

### 🤖 Assistant

在多卡分布式训练中，如果仅靠 `FSDP`（Fully Sharded Data Parallel）或 `DeepSpeed ZeRO-2/3` 就已经能够将模型装入显存（即不发生 OOM），那么**强烈不建议**再开启 `gradient checkpoint`（梯度检查点，或称 Activation Checkpointing）。

核心原因可以用一句话概括：**Gradient Checkpointing 是一种“以时间换空间”的技术，如果在 ZeRO/FSDP 的加持下显存已经够用，开启它不仅毫无收益，还会带来严重的计算和通信开销，导致训练速度大幅下降。**

具体原因如下：

### 1. 纯计算开销的增加（Time Penalty）
`gradient checkpoint` 的原理是在前向传播（Forward Pass）时不保存所有的中间激活值（Activations），而只保存少量的“检查点”。在反向传播（Backward Pass）计算梯度时，对于没有保存的激活值，需要**重新进行一次前向计算**。
*   **计算冗余：** 这会导致整个训练过程增加大约 $20\% \sim 30\%$ 的额外前向计算量。
*   **资源浪费：** 如果显存本来就够用，清空这些激活值只是让显存“空着”，但代价却是让 GPU 花费额外的时间去重新计算，直接拉低了整体的训练吞吐量（Tokens/sec）。

### 2. 致命的额外通信开销（针对 FSDP / ZeRO-3）
这是在多卡 sharding 场景下最关键的一点。`FSDP` 和 `ZeRO-3` 的核心机制是**参数切片（Parameter Sharding）**。
*   **正常情况：** 前向传播时，单卡通过 `All-Gather` 收集完整的权重，计算完后立刻丢弃（或保留给反向传播）。
*   **开启 GC 的情况：** 当反向传播进行到某个被释放的层时，`gradient checkpoint` 会触发**重计算（Recomputation）**以恢复激活值。但是，此时该卡的完整权重已经被释放了！
*   **后果：** 为了完成这次重计算，GPU 必须**再次发起一次 `All-Gather` 通信**来拉取切片权重。这意味着，原本前向和反向各一次的通信开销，变成了前向一次、反向重计算一次、反向算梯度一次。这会引入海量的跨节点/跨卡通信延迟，直接让通信带宽成为瓶颈。

### 3. 降低模型算力利用率（MFU / TFLOPS）
评估大模型训练效率的一个重要指标是 MFU（Model Flops Utilization）。
*   使用 `FSDP / ZeRO` 的目的是在多卡上高效分配显存，使得我们可以跑更大的 Batch Size，从而提升矩阵乘法（GEMM）的效率，推高 GPU 利用率。
*   如果强行加入 `gradient checkpoint`，由于计算流被打断、引入了额外的重计算逻辑和通信等待，GPU 会出现更多的空闲时间（Idle Time），导致实际的 MFU 显著下降。

<details>
<summary><strong>💡 扩展补充：什么时候应该同时使用 ZeRO/FSDP 和 Gradient Checkpoint？</strong></summary>

虽然在显存够用的情况下不建议使用，但在以下**极限场景**中，我们不得不将它们组合使用：

1.  **超大模型训练（如 70B+ 模型）：** 即使使用了 `ZeRO-3` 并在上百张卡上切片，模型状态（参数、梯度、优化器状态）依然占据了大量显存，留给 Activation 的空间极小，一跑前向就会 OOM。
2.  **超长上下文（Long Context）训练：** 激活值（Activations）的显存占用通常与序列长度的平方 $O(L^2)$ 或线性 $O(L)$ 成正比。当训练 $32k$、$128k$ 甚至更长的 Context Window 时，激活值的显存占用会爆炸，远超模型权重本身。此时必须用 `gradient checkpoint` 来压榨显存。
3.  **单卡显存太小（如 24GB 消费级显卡）：** 在算力相对较弱、显存极小且多卡互联带宽较低的环境下，需要穷尽所有省显存的方法才能跑起来，此时只能牺牲时间换取可运行性。

</details>

### 总结建议

在配置训练参数时，**显存优化的优先级**应该遵循以下原则：

1. 优先使用并行策略（`DDP` -> `ZeRO-2` -> `ZeRO-3 / FSDP`）来解决显存问题。
2. 尽可能增大 `Batch Size` 直到把显存填满（保持约 $90\%$ 的显存占用率），以实现最大吞吐。
3. **只有当**使用了最强级别的切片策略（如 `ZeRO-3`），并且把 `Batch Size` 降到了 $1$（微批次），依然出现 `CUDA Out of Memory` 时，**才去开启** `gradient checkpoint`。
