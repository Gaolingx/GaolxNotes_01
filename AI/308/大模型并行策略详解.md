# 大模型并行策略详解

### 🧑‍💻 User

大模型训练中，TP(SP)/PP/CP/EP/DP(DDP/FSDP2/HSDP)这几种并行策略分别是什么意思，哪种并行策略实现难度最大？

---

### 🤖 Assistant

在大模型（LLM）训练中，由于模型参数量、激活值和优化器状态占用的显存远超单张GPU的物理上限，必须采用分布式的**并行策略（Parallelism）**。

以下是主流并行策略的详细解释，以及关于实现难度的分析。

---

### 一、 各种并行策略详解

#### 1. DP (Data Parallelism) 数据并行及其进阶
数据并行的核心思想是：**模型不动，数据分割**。每个 GPU 上都有相同的模型，但处理不同的数据批次（Batch）。
*   **DDP (Distributed Data Parallel):** 最基础的数据并行。每张显卡保留完整的模型参数、梯度和优化器状态。前向传播后，各 GPU 通过 `All-Reduce` 同步梯度，然后各自更新参数。缺点是显存浪费严重，无法训练超大模型。
*   **FSDP / FSDP2 (Fully Sharded Data Parallel):** 基于 ZeRO（零冗余优化器）思想。它将模型参数、梯度和优化器状态**切片（Shard）**到不同的 GPU 上。计算某一层时，通过 `All-Gather` 临时获取完整参数，计算完立刻释放。**FSDP2** 是 PyTorch 推出的新一代架构，相比第一代，它解耦了切片与 Module 的绑定，采用基于张量大小的 2D 切片策略，通信和计算的重叠（Overlap）做得更好。
*   **HSDP (Hybrid Sharded Data Parallel):** 混合切片数据并行。在节点（Node，通常是单台拥有 8 张 GPU 的机器）内部使用 FSDP 切片以节省显存，在节点与节点之间使用 DDP 复制以减少跨节点的网络通信开销。

#### 2. TP (Tensor Parallelism) 与 SP (Sequence Parallelism)
核心思想：**切分矩阵乘法和张量**。通常在单节点内部（NVLink 环境下）使用。
*   **TP (张量并行):** 以 Transformer 中的线性层（Linear）为例，计算式为 $Y = XW$。TP 会将权重矩阵 $W$ 按行或按列切分到多个 GPU 上，各 GPU 计算出部分结果后，通过 `All-Reduce` 或 `All-Gather` 聚合成最终结果（即著名的 Megatron-LM 策略）。
*   **SP (序列并行):** 配合 TP 使用。在 Transformer 模型中，LayerNorm 和 Dropout 等非线性操作不需要特征维度的交互。SP 将这些操作沿着**序列长度（Sequence Length）**维度进行切分，从而进一步降低激活值（Activations）的显存占用，并减少 TP 带来的部分通信冗余。

#### 3. PP (Pipeline Parallelism) 流水线并行
核心思想：**按层纵向切分模型**。
*   将模型的不同层（Layers）分配到不同的 GPU 上。例如，GPU 0 计算第 1-10 层，GPU 1 计算第 11-20 层。
*   为了提高 GPU 利用率，需要将一个 Batch 进一步切分为多个 Micro-batch（微批次）。GPU 像工厂流水线一样工作。经典调度算法包括 GPipe、1F1B（One Forward One Backward）以及 Interleaved 1F1B。

#### 4. CP (Context Parallelism) 上下文并行
核心思想：**专门针对超长上下文（Long Context）的序列切分**。
*   当输入序列长度 $L$ 达到 100K 甚至 1M 时，注意力机制的计算复杂度为 $O(L^2)$，单卡即使使用了 SP 也无法承受。
*   CP 会在**注意力机制（Attention）计算的内部**对序列进行切分。各个 GPU 只负责计算 Query/Key/Value 的一部分，通过特殊的通信机制（如 Ring-Attention 或 DeepSpeed Ulysses）在计算过程中动态传递 Key 和 Value，从而实现几乎无限的上下文长度扩展。

#### 5. EP (Expert Parallelism) 专家并行
核心思想：**专用于 MoE（Mixture of Experts）架构**。
*   在 MoE 模型中，前馈神经网络（FFN）被替换为多个“专家”。EP 将不同的专家放置在不同的 GPU 上。
*   当输入 Token 经过路由（Router）时，会被通过 `All-to-All` 通信发送到对应的专家所在的 GPU 上进行计算，计算完成后再通过 `All-to-All` 发送回原来的 GPU。

---

### 二、 哪种并行策略实现难度最大？

从工程实现和优化的角度来看，**EP (专家并行)** 和 **PP (流水线并行)** 的实现难度最大，其次是 CP。

#### 🏆 难度并列第一：EP (专家并行)
EP 的难点在于**动态路由与负载均衡**：
1.  **负载不均 (Load Imbalance):** Token 路由是动态的，某些专家可能被“撑死”（分配了大量 Token，导致 OOM 或计算瓶颈），而其他专家“饿死”。实现复杂的辅助损失函数（Auxiliary Loss）或无损负载均衡算法极具挑战。
2.  **通信瓶颈:** EP 高度依赖 `All-to-All` 通信，这是一种极其密集的全局通信模式。如何让 `All-to-All` 通信与专家的计算完美重叠（Overlap），是底层系统优化的噩梦。
3.  **显存波动:** 由于各个 GPU 接收到的 Token 数量在训练过程中是动态变化的，显存的分配和管理变得极难预测。

#### 🏆 难度并列第一：PP (流水线并行)
PP 的难点在于**调度逻辑与气泡（Bubble）消除**：
1.  **流水线气泡:** 由于各个 Stage 之间存在依赖关系，GPU 会出现空闲等待的时间（即气泡）。优化这些气泡需要编写极其复杂的调度器（如 Interleaved 1F1B 或 Zero-Bubble 调度）。
2.  **死锁问题 (Deadlocks):** P2P (Peer-to-Peer) 通信如果 Send 和 Recv 的时序或微批次（Micro-batch）状态管理稍微出错，就会导致整个集群死锁卡死。
3.  **显存生命周期管理:** 必须精确管理每个 Micro-batch 在前向传播产生的激活值，并一直保存到反向传播使用，内存池的设计极为复杂。

#### 难度第二梯队：CP (上下文并行)
CP（如 Ring-Attention）需要深入修改底层 CUDA Kernel（如 FlashAttention）。要在硬件层面手动控制块（Block）的加载、片上内存（SRAM）的复用以及 GPU 之间的点对点环形通信，对开发者的底层算子编写能力要求极高。

#### 难度相对较低：TP、DP、FSDP
这三种策略的计算图相对静态。目前 PyTorch 的 `DTensor` (Device Tensor) 和成熟的开源框架（如 Megatron-LM, DeepSpeed）已经将 TP 和 FSDP 抽象得非常好，开发者只需添加几行配置或包装代码即可实现。

---

<details>
<summary><strong>💡 补充信息：3D/4D 混合并行（Hybrid Parallelism）</strong></summary>

在实际训练万亿参数的巨型模型（如 GPT-4 或 Gemini）时，往往不能单独使用某一种策略，而是必须将它们结合起来，这被称为 **3D 或 4D 并行**。

典型的 3D 并行拓扑结构：
*   **TP** 部署在同一个节点内的 GPU 之间（利用极高的 NVLink 带宽）。
*   **PP** 部署在不同节点之间（容忍相对较高的网络延迟，只传递边界激活值）。
*   **DP / FSDP** 部署在由 TP+PP 组成的副本组（Replica Groups）之间。

如果模型是 MoE 且处理长文本，则会叠加 **EP** 和 **CP**。将这 5 种并行策略在一个集群中无bug地融合在一起，并实现全局的通信计算重叠，是目前 AI 系统工程界的最高难点。

### 🤖 Assistant

在分布式训练中，`DeviceMesh`（设备网格）是将底层物理 GPU 集群抽象为多维逻辑拓扑的核心机制。通过将不同的并行策略映射到 `DeviceMesh` 的不同维度，我们可以灵活地组合出复杂的 2D、3D 甚至 4D 并行策略。

根据物理 GPU 总数 $N_{\text{total}}$，其数学关系始终满足：
$$N_{\text{total}} = N_{\text{dp}} \times N_{\text{ep}} \times N_{\text{pp}} \times N_{\text{cp}} \times N_{\text{tp}}$$

以下结合实际模型规模和硬件资源，从单节点到万卡级别，分析 `DeviceMesh` 的设置与并行策略组合。

---

### 一、 单节点 (8 GPUs)：百亿参数以下 Dense 模型
**适用案例：** Llama-3-8B, Qwen-2-7B Dense 模型。
**物理环境：** 单台 8x A100/H100，节点内具备 NVLink 高速互联。

在单节点规模下，通常不需要复杂的 3D 并行，显存瓶颈是首要解决的问题。

*   **并行策略组合：** 纯 **FSDP2** (Fully Sharded Data Parallel) 或者 **TP** (张量并行)。
*   **DeviceMesh 设置：** 一维网格 (1D Mesh)。

```python
from torch.distributed.device_mesh import init_device_mesh

# 方案 A: 纯 FSDP2 (推荐，实现最简单，吞吐量高)
# Mesh 形状: (8,)
mesh_fsdp = init_device_mesh("cuda", (8,), mesh_dim_names=("fsdp",))

# 方案 B: 纯 TP (适合对 latency 要求极高的场景)
# Mesh 形状: (8,)
mesh_tp = init_device_mesh("cuda", (8,), mesh_dim_names=("tp",))
```

---

### 二、 中型集群 (64 - 256 GPUs)：数百亿参数长文本模型
**适用案例：** Llama-3-70B Dense，处理 128K 超长上下文。
**物理环境：** 8 - 32 个节点，节点间通过 InfiniBand (IB) 网络连接。

当模型达到 70B 且涉及长文本时，单卡显存无法容纳，且长文本会导致注意力计算爆炸，此时必须结合 **TP**（切分参数）和 **CP**（切分序列），并使用 **DP/FSDP** 来横向扩展 Batch Size。

*   **并行策略组合：** **TP** (节点内) + **CP** (长文本序列并行) + **FSDP / DDP** (数据并行)。
*   **DeviceMesh 设置：** 假设使用 128 张卡（16 个节点），构建 3D Mesh。

```python
# 总卡数 N = 128。设置 TP=8 (占满单节点 NVLink), CP=4 (跨节点切分上下文), DP=4 (数据并行)
# 128 = 4 (DP) * 4 (CP) * 8 (TP)
mesh_3d = init_device_mesh(
    "cuda", 
    (4, 4, 8), 
    mesh_dim_names=("dp", "cp", "tp")
)
```
**分析：** 将 `tp` 放在最后一个维度，是因为 TP 需要极其密集的 `All-Reduce`，必须在节点内 (NVLink) 解决。`cp` 和 `dp` 维度跨节点分布。

---

### 三、 大型集群 (1024 - 4000 GPUs)：千亿参数 MoE 模型
**适用案例：** Mixtral 8x22B, Grok-1 等包含大量专家的 MoE 模型。
**物理环境：** 128 - 500 个节点。

MoE 模型的特点是**总参数量极大，但激活参数量相对较小**。这里的核心难点在于专家并行（EP）的通信开销。通常将 EP 维度与 DP 维度重叠，或者采用 FSDP 结合 EP（即 FSDP 切分非专家层，EP 切分专家层）。

*   **并行策略组合：** **TP** (非专家层) + **EP** (专家层) + **FSDP** (非专家层数据并行，通常与 EP 共享通信组或解耦)。
*   **DeviceMesh 设置：** 假设使用 1024 张卡，模型有 16 个专家。

```python
# 总卡数 N = 1024。设置 TP=4 (节点内), EP=16 (专家并行), DP=16 (数据并行)
# 1024 = 16 (DP) * 16 (EP) * 4 (TP)
moe_mesh = init_device_mesh(
    "cuda", 
    (16, 16, 4), 
    mesh_dim_names=("dp", "ep", "tp")
)
```
**分析：** 专家层使用 `ep` 和 `tp` 组成的 sub-mesh 进行计算。路由（Router）决定 Token 走向后，通过 `All-to-All` 在 `ep` 维度上的 16 台设备间传输 Token。

---

### 四、 万卡集群 (10,000+ GPUs)：万亿参数巨无霸模型 (Dense/MoE)
**适用案例：** GPT-4 级别（假设为 1.8T 参数，多专家 MoE），Gemini Ultra 级别。
**物理环境：** 1250+ 个节点，超大规模无阻塞 IB 胖树网络。

在万卡规模下，网络通信延迟和容错成为了最大的敌人。**流水线并行 (PP)** 变得不可或缺，因为只有 PP 能够将万张卡的全局密集通信，降级为相邻 Stage 之间少量的点对点 (P2P) 通信。

*   **并行策略组合：** **DP** (结合 FSDP 的参数切分) + **EP** (针对 MoE) + **PP** (跨多个节点) + **TP** (节点内)。这是典型的 4D 或 5D 并行。
*   **DeviceMesh 设置：** 总卡数 $N_{\text{total}} = 16384$。

```python
# 假设架构：万亿参数，16个专家。
# 配置：DP=16, EP=16, PP=8, TP=8
# 校验：16 * 16 * 8 * 8 = 16384
frontier_mesh = init_device_mesh(
    "cuda", 
    (16, 16, 8, 8), 
    mesh_dim_names=("dp", "ep", "pp", "tp")
)
```
**万卡策略执行流：**
1.  **节点内计算：** 模型被 `tp` 切分成 8 份，单台机器利用 NVLink 飞速计算。
2.  **流水线传递：** 计算完当前层后，结果顺着 `pp` 维度发送给下一个节点组合（P2P 通信，隐藏在计算中）。
3.  **专家路由：** 遇到 MoE 层时，Token 在 `ep` 维度的 16 个节点间进行全局 `All-to-All` 交换。
4.  **梯度同步：** 反向传播结束后，只在 `dp` 维度的 16 个副本组内进行相对小规模的 `Reduce-Scatter` 和 `All-Gather`，极大降低了万卡同步梯度的爆炸性网络冲击。

<details>
<summary><strong>⚙️ 进阶拓展：DeviceMesh 设计的黄金法则 (点击展开)</strong></summary>

在设计 `DeviceMesh` 并分配各个并行的数值时，需遵循以下物理限制法则：

1.  **TP 永远在最内层 (Inner-most dimension)：** 
    张量并行的通信（`All-Reduce`）极其频繁，数据量等同于激活值。因此，$N_{\text{tp}}$ 的大小**绝不应该超过单台机器的 GPU 数量（通常是 8）**，必须将其约束在 NVLink 或 NVSwitch 的高带宽域内。
2.  **PP 适合放在最外层或次外层：**
    流水线并行只在 Stage 的边界传递激活值张量，通信量很小（$O(\text{Batch} \times \text{SeqLen} \times \text{Hidden})$），对带宽要求最低，但对延迟敏感。可以将其映射到跨机架（Rack）或跨交换机的慢速网络连接上。
3.  **EP 的大小必须能整除专家数量：**
    如果模型有 $E$ 个专家，$N_{\text{ep}}$ 最好满足 $E \pmod{N_{\text{ep}}} = 0$。如果 $E=8$ 而 $N_{\text{ep}}=16$，会导致一半的 GPU 闲置；如果不能整除，会导致严重的负载不均（Load Imbalance）。
4.  **HSDP (混合分片) 代替纯 FSDP：**
    在万卡级别，纯 FSDP 会导致跨域数千个节点的全局 `All-Gather`，效率极低。通常将 DDP 拆分为两维：`("dp_replicate", "dp_shard")`，在同一个机架内的几十张卡做切分（shard），在跨机架级别做复制（replicate），这就是 HSDP 策略。

</details>
