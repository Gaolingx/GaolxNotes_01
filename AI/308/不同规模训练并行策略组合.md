# 不同规模训练并行策略组合

### 🧑‍💻 User

pytorch2.4+中引入了hsdp策略可以方便的划分devicemesh组合tp/dp/pp/sp等，实际不同规模训练中如何组合并行策略？（例如tp+fsdp2/tp+ep+fsdp2）

---

### 🤖 Assistant

在 PyTorch 2.4 及更高版本中，基于 `DeviceMesh` 的分布式架构得到了全面重构和增强，特别是 FSDP2（底层完全基于分布式张量 `DTensor`）和 HSDP（Hybrid Sharded Data Parallel）的引入，使得组合多种并行策略变得前所未有地直观。

在实际不同规模的训练中，组合并行策略的核心逻辑是**在显存容量、计算效率（MFU）和通信带宽（NVLink vs. InfiniBand）之间寻找最优解**。

以下是针对不同训练规模和模型结构的实际组合策略指南：

### 1. 核心概念：基于 DeviceMesh 的 N 维拓扑

在 PyTorch 2.4+ 中，所有组合并行的基础都是 `init_device_mesh`。它将物理 GPU 划分为多维逻辑网格。

例如，对于一个 8 节点、每节点 8 张 GPU（共 64 卡）的集群，我们可以定义一个二维网格：
```python
from torch.distributed.device_mesh import init_device_mesh

# 维度 0: 数据并行组 (跨节点通信, size=8)
# 维度 1: 张量并行组 (节点内通信, size=8)
mesh = init_device_mesh("cuda", (8, 8), mesh_dim_names=("dp", "tp"))
```

---

### 2. 不同规模与场景下的并行组合策略

#### 场景 A：中等规模集群，百亿参数模型 (例如 Llama3-8B, 16~64 GPUs)
**推荐策略：纯 FSDP2 或 HSDP (FSDP2 + DDP)**

对于 $10B$ 以下级别的模型，通常不需要引入复杂的 TP（张量并行）。
*   **纯 FSDP2:** 将模型参数、梯度和优化器状态打散到所有 GPU 上。
*   **HSDP:** 如果跨节点网络带宽较差（例如只有普通以太网，没有 IB 网络），使用 HSDP。在节点内（8卡）做 FSDP 共享参数，在节点间做 DDP 复制参数。

<details>
<summary>点击查看 HSDP 的 DeviceMesh 配置代码</summary>

```python
# HSDP 网格配置: node 维度用于 DDP (复制), gpu 维度用于 FSDP (分片)
hsdp_mesh = init_device_mesh("cuda", (num_nodes, gpus_per_node), mesh_dim_names=("dp_replicate", "dp_shard"))

# 应用 FSDP2
from torch.distributed.fsdp import fully_shard

# 使用 fully_shard 替代原来的 FSDP class
fully_shard(model, mesh=hsdp_mesh)
```
</details>

#### 场景 B：大规模集群，千亿参数模型 (例如 Llama3-70B, 128~1024 GPUs)
**推荐策略：TP + SP + FSDP2**

当模型增大到单节点 FSDP 通信开销过高，或者需要极大的 Sequence Length 时，引入 TP 和 SP（序列并行）。
*   **TP (Tensor Parallel):** 限制在**节点内**（利用高带宽 NVLink）。
*   **SP (Sequence Parallel):** 与 TP 强绑定，沿着序列维度对 Activation 进行切分，大幅降低超长上下文训练时的显存。
*   **FSDP2 (Data Parallel):** 跨节点进行，处理被 TP/SP 切分后的副本。

**DeviceMesh 划分:**
```python
mesh = init_device_mesh("cuda", (dp_size, tp_size), mesh_dim_names=("dp", "tp"))
dp_mesh = mesh["dp"]
tp_mesh = mesh["tp"]

# 伪代码：先对模型应用 TP
parallelize_module(model, tp_mesh, tp_plan)
# 再应用 FSDP2 到 DP mesh 上
fully_shard(model, mesh=dp_mesh)
```

#### 场景 C：超大规模集群，万亿参数模型 (1000+ GPUs)
**推荐策略：3D 并行 (TP + PP + FSDP2 / DP)**

对于超大模型，即使用了 TP+FSDP2，跨节点的 All-Gather 和 Reduce-Scatter 通信量依然会阻塞网络。此时必须引入 PP（流水线并行）。
*   **TP:** 节点内（$N=8$）。
*   **PP (Pipeline Parallel):** 跨节点，将模型的不同 Transformer 层分配到不同的阶段（Stages）。PP 的通信仅限于相邻 Stage 传输 Activation 和 Gradient，对带宽要求较低。
*   **FSDP2 / DDP:** 在相同的 PP Stage 和不同的 TP 组之间进行数据并行。

<details>
<summary>点击查看 3D 并行的拓扑解析</summary>

一个 `(pp_size, dp_size, tp_size)` 的 3D Mesh：
1. `tp_size` 通常为 4 或 8（单节点内）。
2. 根据模型层数决定 `pp_size`。
3. 剩余的卡数作为 `dp_size`。
PyTorch 2.4 中的 `PiPPy` 或 `torch.distributed.pipelining` 可以与 `DeviceMesh` 无缝结合。
</details>

#### 场景 D：MoE 架构模型 (例如 Mixtral 8x7B / 8x22B)
**推荐策略：TP + EP + FSDP2**

MoE（混合专家）模型的特点是参数量极大，但单次 Forward/Backward 的活跃参数（Active Parameters）很少。
*   **EP (Expert Parallel):** 将不同的专家（Experts）分配到不同的 GPU 上。通常在跨节点进行。
*   **FSDP2:** 用于非专家层（如 Attention 层）的参数切分。
*   **TP:** 如果单个专家的计算量或 Attention 计算量巨大，可结合 TP。

在 PyTorch 2.4 中，你可以为专家层和非专家层指定不同的 `DeviceMesh`：
```python
# Attention 层使用 FSDP 打散
fully_shard(model.attention, mesh=dp_mesh)

# MoE 路由层使用 EP (通过 DTensor 按专家维度切分)
# 专家参数按 ep_mesh 分布
```

### 3. 并行策略显存与通信数学模型

在选择策略时，我们需要评估显存的消耗。假设模型总参数量为 $\Phi$，优化器状态占用的系数为 $K$ (AdamW 通常 $K=2$ 对应动量和方差，总共占用 $12$ bytes/param)，微批次大小为 $B$，序列长度为 $S$，隐藏层维度为 $H$。

#### 纯数据并行 (DDP) 显存占用：
$$ M_{DDP} = \Phi \cdot (2 + K \cdot 4) + O(B \cdot S \cdot H) $$
*这里每张卡都需要保存完整的模型和优化器，显存极易溢出。*

#### FSDP 显存占用 (切分度为 $D$)：
$$ M_{FSDP} = \frac{\Phi \cdot (2 + K \cdot 4)}{D} + O(B \cdot S \cdot H) $$

#### TP + SP + FSDP 显存占用 (TP大小为 $T$，DP大小为 $D$)：
加入序列并行后，Activation 也会被切分：
$$ M_{3D} = \frac{\Phi \cdot (2 + K \cdot 4)}{D \cdot T} + \frac{O(B \cdot S \cdot H)}{T} $$

*(注：以上公式中，常量乘数表示 FP16/BF16 训练下的字节数估算)*

### 4. 总结与实践建议表

在 PyTorch 2.4 时代，建议放弃旧版杂乱的配置方式，**全面拥抱基于 `DeviceMesh` 的声明式 API**。

| 模型规模 | 集群硬件条件 | 推荐并行组合 | `DeviceMesh` 维度设计 |
| :--- | :--- | :--- | :--- |
| **< 10B** | 单节点 / 万兆以太网多节点 | **FSDP2** 或 **HSDP** | 1D `(fsdp,)` 或 2D `(ddp, fsdp)` |
| **10B - 70B** | 多节点，具备 InfiniBand | **TP + FSDP2** (结合 SP) | 2D `(fsdp, tp)` |
| **> 100B** | 大型超算集群 | **PP + TP + FSDP2** | 3D `(pp, fsdp, tp)` |
| **MoE 模型** | 多节点集群 | **EP + FSDP2** (按需加 TP) | 2D `(ep, fsdp)` |

**最佳实践步骤：**
1. 优先使用 **纯 FSDP2**。只要显存不爆、吞吐量能接受，纯 FSDP2 是代码最简单、最鲁棒的。
2. 如果 Activation 显存成为瓶颈（通常是因为 $S$ 极大），引入 **TP + SP**。确保 TP Mesh 局限在单台机器内。
3. 只有当模型极深、参数极大（千亿级别），或者网络带宽严重受限时，才考虑引入调试难度较高的 **PP（流水线并行）**。

### 🧑‍💻 User

相比纯fsdp，这种多维并行对于scaling有哪些好处？

---

### 🤖 Assistant

相比纯粹的全局 FSDP（一维切分），引入基于 `DeviceMesh` 的多维并行策略（如 HSDP、TP+SP+FSDP、3D 并行），其核心目的在于打破**“通信墙”**和**“显存墙（特别是激活值显存）”**。

当集群规模扩展到数百乃至数千张 GPU，或者模型参数量极大、上下文极长时，纯 FSDP 会暴露出严重的物理瓶颈。多维并行的优势主要体现在以下四个关键方面：

### 1. 匹配底层硬件拓扑，极大降低跨节点通信开销

纯 FSDP 将所有 GPU 视为一个扁平的集群（1D Mesh）。在每次前向（Forward）和反向（Backward）传播中，都需要在**全局所有 GPU 之间**执行 All-Gather（收集参数）和 Reduce-Scatter（聚合梯度）。

*   **纯 FSDP 的痛点**：跨节点网络（如 InfiniBand 或 Ethernet）的带宽远远低于单机节点内带宽（如 NVLink，通常相差 10 倍以上）。全局通信会导致严重的网络拥塞和延迟，使得计算单元处于等待状态，MFU（模型计算算力利用率）暴跌。
*   **多维并行的优势（以 HSDP 为例）**：
    HSDP (FSDP2 + DDP) 通过 2D Mesh 完美契合了硬件物理拓扑。在节点内（利用高带宽 NVLink）使用 FSDP 共享参数，在节点间使用 DDP 同步梯度。
    *通信量对比：DDP 仅在反向传播结束时进行一次梯度的 All-Reduce，相比 FSDP 频繁的跨节点 All-Gather / Reduce-Scatter，大幅降低了对跨机网络的带宽要求。*

### 2. 打破“激活值显存墙”，支持超长上下文 (Long Context)

纯 FSDP 能够完美切分**模型权重、梯度和优化器状态**，但它**不能切分单个样本的激活值 (Activations)**。

*   **纯 FSDP 的痛点**：在训练长文本（例如 Llama-3 的 128K 上下文）时，根据公式，注意力机制的激活值显存与序列长度 $S$ 呈二次方甚至线性正相关：$O(B \cdot S \cdot H)$。此时，即便模型参数本身的显存全部分摊，单卡的显存也会瞬间被激活值撑爆（OOM）。
*   **多维并行的优势（引入 TP + SP）**：
    张量并行（TP）结合序列并行（SP），能够沿着序列维度对激活值进行切分。如果节点内引入规模为 $T$ 的 TP/SP 组，单卡的激活值显存消耗直接降为原来的 $\frac{1}{T}$。这是目前训练 100K+ 上下文大模型的唯一成熟方案。

### 3. 减小集合通信组的规模 (Group Size)，降低延迟

在分布式训练中，集合通信（Collective Communication，如 All-Gather）的延迟与参与通信的设备数量 $N$ 正相关。

$$ \text{Latency} \propto \log(N) \quad \text{or} \quad O(N) \text{ depending on topology} $$

*   如果集群有 1024 张卡，纯 FSDP 的通信环（Ring）或树（Tree）非常庞大，极易因为某一张卡的“微小卡顿（Straggler）”导致全局阻塞。
*   如果使用 `TP(8) + FSDP(128)`，FSDP 的通信组规模从 1024 骤降到 128，极大地提高了通信的稳定性和执行效率，也更容易通过代码实现计算与通信的掩盖（Overlap）。

### 4. 利用流水线并行 (PP) 跨越单节点通信极限

当模型规模达到千亿、万亿级别时，即使参数被 FSDP 打散，单层前向传播所需的完整参数量在跨节点传输时，也会造成网络瘫痪。

<details>
<summary>点击查看 PP (流水线并行) 的独特优势</summary>

*   **点对点通信 vs. 集合通信**：FSDP 依赖的是全局集合通信（All-Gather）。而流水线并行（PP）将模型按层切分到不同节点，节点之间只需通过点对点（Point-to-Point, `send`/`recv`）通信传递隐藏层状态（Hidden States）。
*   **极低的带宽占用**：PP 传输的数据量仅为 $O(B \cdot S \cdot H)$，完全不涉及庞大模型权重的传输，极其适合跨机房、跨非对称网络的超大规模集群训练。
</details>

### 总结对比表

| 特性 | 纯 FSDP | 多维并行 (TP/PP/HSDP+FSDP) |
| :--- | :--- | :--- |
| **拓扑感知** | 差 (扁平化，无视节点边界) | 强 (精准匹配 NVLink 与 IB 网络) |
| **激活值切分** | 否 (不支持序列切分) | 是 (通过 SP 极大地降低激活值显存) |
| **长上下文支持**| 弱 (容易 OOM) | 强 (支持 100K+ Tokens 序列) |
| **超大模型扩展**| 受限于跨节点网络带宽 | 通过 PP 和 TP 完美扩展到万亿参数 |
| **代码复杂度** | **极低** | 较高 (但在 PyTorch 2.4 DeviceMesh 下已大幅降低) |
