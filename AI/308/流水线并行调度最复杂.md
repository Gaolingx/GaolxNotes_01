# 流水线并行调度最复杂

### 🧑‍💻 User

模型并行策略中，dp/tp/pp/ep哪种pipeline设计起来复杂度更高？

---

### 🤖 Assistant

在模型并行的这四种策略（DP, TP, PP, EP）中，**Pipeline Parallelism (PP，流水线并行)** 在**调度与执行流水线（Pipeline/Schedule）的设计复杂度上是最高的**。

这四者的侧重点不同，导致它们在工程实现上的复杂点也不一样。如果单论“时间线上的调度控制流（Control Flow）”和“流水线设计”，PP 无疑是最复杂的。以下是详细的对比分析：

### 1. 🥇 流水线并行 (PP, Pipeline Parallelism) - 复杂度：极高

PP 的核心思想是按层（Layers）将模型切分到不同的 GPU 上。为了让所有 GPU 都能运转起来（避免前面的 GPU 算完干等），必须引入**微批次（Micro-batching）**，这就带来了极其复杂的调度设计。

*   **调度算法复杂 (Schedule Design)：** 你不能像单卡那样按顺序执行前向（Forward）和反向（Backward）。你需要设计复杂的调度策略，例如：
    *   **GPipe (Fill and Drain)：** 容易产生极大的气泡（Bubble，即 GPU 闲置时间）。
    *   **1F1B (One Forward One Backward)：** 为了降低激活显存占用，必须交替调度前向和反向计算。
    *   **Interleaved 1F1B (交错 1F1B)：** 进一步切分模型块（Virtual Pipeline Stages）来压缩气泡，调度逻辑堪比操作系统的指令流水线。
*   **状态管理与显存屏障 (State & Memory Management)：** 前向计算的激活值（Activations）必须被缓存（Stash），直到对应的微批次反向传播到来时才能释放。
*   **死锁风险 (Deadlock Risks)：** GPU 之间存在大量的点对点（P2P, Point-to-Point）通信。如果发送（Send）和接收（Recv）的顺序、微批次的依赖关系在不同阶段没有严格对齐，极易引发分布式死锁。
*   **气泡优化 (Bubble Optimization)：** 理论上流水线的闲置时间比例大约为 $\frac{p-1}{m}$（其中 $p$ 是流水线阶段数，$m$ 是微批次数量），如何通过调度压榨这部分时间是无止境的工程难题（如 Zero Bubble Pipeline）。

### 2. 🥈 专家并行 (EP, Expert Parallelism) - 复杂度：高

EP 通常用于 MoE（Mixture of Experts）架构，它的复杂度主要体现在**数据路由（Routing）和负载均衡**上，而非时间流水线的调度。

*   **动态路由与 All-to-All 通信：** 每个 Token 在运行时才决定去哪个 GPU 上的专家。这需要复杂的 All-to-All 通信，且通信量和目标节点是动态变化的。
*   **负载不均衡 (Load Imbalance)：** 某些专家可能被分配了过多的 Token，导致该 GPU OOM（内存溢出）或计算瓶颈，而其他 GPU 闲置。必须设计复杂的丢弃策略（Token Dropping）、容量限制（Capacity Factor）或辅助损失函数（Auxiliary Loss）。
*   **底层通信重叠：** 为了提高效率，往往需要深入 CUDA 层面，将 All-to-All 通信与计算进行重叠（Overlap）。

### 3. 🥉 张量并行 (TP, Tensor Parallelism) - 复杂度：中等

TP 的核心是将矩阵乘法（如 Linear 层、Attention）切分到不同的 GPU 上（以 Megatron-LM 为代表）。

*   **数学切分：** 需要推导前向和反向的数学等价性（例如行切分、列切分）。
*   **同步要求极高：** 它的执行是**完全同步的**。在每一层计算中，GPU 算完自己的部分后，必须通过 `All-Gather` 或 `Reduce-Scatter` 聚合数据才能进行下一步。
*   **流水线复杂度低：** 虽然算子底层的通信实现很复杂，但在宏观的“训练循环（Training Loop）”来看，它和单卡训练的控制流是一模一样的：顺序执行 Forward -> Backward -> Step。**它没有 PP 那种微批次交错的复杂调度。**

### 4. 🏅 数据并行 (DP, Data Parallelism) - 复杂度：低

DP（包括衍生的 FSDP / ZeRO 等显存优化技术）是业界最成熟、最标准的方案。

*   **执行逻辑简单：** 每个 GPU 拥有完整的模型（或逻辑上完整），处理不同的数据。前向算完，反向算完，最后做一次梯度同步（All-Reduce）。
*   **框架支持完善：** PyTorch 的 `DistributedDataParallel` (DDP) 让开发者几乎可以像写单机代码一样写分布式代码。即使是 ZeRO/FSDP，其复杂度也主要是框架底层对显存（Parameter/Gradient/Optimizer State）的管理和 Hook 拦截，对上层的流水线控制流几乎没有影响。

### 总结

如果以**“流水线控制流设计（Pipeline Schedule Design）”**为标准来排名，复杂度从高到低依次为：

**PP (流水线并行) >> EP (专家并行) > TP (张量并行) > DP (数据并行)**

*   **PP** 的难点在于 **时间与空间（微批次）的交错调度**，极易死锁。
*   **EP** 的难点在于 **动态数据的路由与负载均衡**。
*   **TP** 的难点在于 **底层算子切分与极致的通信优化**。
*   **DP** 的难点在于 **显存切分（如 ZeRO）与通信带宽的利用**，但控制流最简单。
