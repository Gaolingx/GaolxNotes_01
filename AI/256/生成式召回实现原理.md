# 生成式召回实现原理

### 🧑‍💻 User

搜广推除了双塔召回，现在还有一种Transformer和RQ-VAE训打分取topk的那种像分类器是怎么实现的？

---

### 🤖 Assistant

你提到的这种架构，在搜广推（尤其是推荐和检索领域）中被称为**生成式召回（Generative Retrieval）**或**生成式推荐（Generative Recommendation）**。这是近两年打破传统双塔模型（Two-Tower）范式的前沿技术。

这种方案的核心思想是：**放弃为每个Item分配独立的无语义ID（如UUID），而是用RQ-VAE将Item压缩成一段有语义的“离散Token序列”（Semantic ID），然后用Transformer像做文本生成（Language Modeling）一样，预测用户下一个会点击的Item的Token序列。**

因为它在预测下一个Token时使用的是Softmax分类，所以你会有“像分类器”的直觉。以下是这种方案的具体实现全链路解析：

### 核心架构：两阶段框架

这种实现通常分为两个相互独立的阶段：**语义ID生成阶段**（RQ-VAE）和**序列生成与召回阶段**（Transformer）。

#### 1. 阶段一：用 RQ-VAE 生成语义 ID (Item Tokenization)
传统推荐系统中，Item ID 是极度稀疏的，模型很难学到生僻 Item 的特征。RQ-VAE（残差量化变分自编码器）的作用，就是将 Item 的多模态特征“翻译”成一个简短的、有语义的离散词表。

*   **输入：** Item 的内容特征（如标题文本的BERT向量、图片的ViT向量，甚至是传统的协同过滤Embedding）。记为 $x \in \mathbb{R}^d$。
*   **残差量化过程：**
    1. RQ-VAE 有多个层级的离散密码本（Codebook），假设有 $M$ 层，每层有 $V$ 个词（Token）。
    2. 第1层：寻找与 $x$ 最接近的 Codebook 向量 $e_1$，记录其索引 $c_1$。计算残差 $r_1 = x - e_1$。
    3. 第2层：寻找与残差 $r_1$ 最接近的 Codebook 向量 $e_2$，记录索引 $c_2$。计算残差 $r_2 = r_1 - e_2$。
    4. 以此类推，直到第 $M$ 层。
*   **输出：** 最终，原来的 Item 被表示为了一个长度为 $M$ 的离散 Token 序列 $(c_1, c_2, ..., c_M)$。
*   **效果：** 相似的 Item 会具有相同的前缀 Token。例如，“苹果手机”可能是 `[12, 45, 89]`，而“华为手机”可能是 `[12, 45, 102]`。这就赋予了 ID 结构化和语义化的能力。

#### 2. 阶段二：用 Transformer 训练序列“分类/生成”模型
有了语义 ID 后，用户的历史行为就变成了一串 Token 序列。此时，召回任务就被彻底转化为了一个**自回归生成（Autoregressive Generation）**任务。

*   **输入构建：** 假设用户历史点击了 Item A `[12, 45, 89]` 和 Item B `[34, 11, 09]`。我们将这些 Token 拼接成一维序列作为 Transformer 的输入（可以加入 User Profile 的特征作为 Prompt）。
*   **模型结构：** 通常使用仅包含 Decoder 的架构（类似 GPT 或 LLaMA），或者 Encoder-Decoder 架构（类似 T5）。
*   **“像分类器”的训练机制：**
    Transformer 在每一个时间步，都会输出一个维度为 $V$（Codebook 大小）的 Logits 向量。经过 Softmax 后，这就是一个针对 $V$ 个类别的**多分类任务**。
    训练损失使用的是标准的交叉熵损失（Cross-Entropy Loss）：
    $$ \mathcal{L} = - \sum_{j=1}^{M} \log P(c_j | \text{User History}, c_{1:j-1}) $$
    *这里可以看出，它本质上就是在各个层级上做 $V$ 分类。*

#### 3. 推理阶段：如何打分并取 Top-K 召回？
这是生成式召回与普通文本生成最大的区别。如果是普通的文本生成，模型可能会“幻觉”出一个根本不存在的 Token 组合（比如 `[12, 99, 99]` 这个 ID 对应的库里没有商品）。为了保证召回的都是真实存在的 Item，并取出得分最高的 Top-K，需要使用 **前缀树（Trie） + 束搜索（Beam Search）**。

*   **构建前缀树（Trie）：** 在离线阶段，将全量商品库中所有 Item 的语义 ID（Token 序列）存入一棵 Trie 树中。这棵树定义了所有“合法”的生成路径。
*   **受限波束搜索（Constrained Beam Search）：**
    1. **第一步（预测 $c_1$）：** Transformer 给出第一个 Token 的概率分布。由于受 Trie 树限制，只保留树中第一层存在的 Token 的概率，选出概率最高的 $K$ 个分支。
    2. **第二步（预测 $c_2$）：** 沿着刚才的 $K$ 个分支继续预测。在算力允许的情况下，每一步都只在 Trie 树允许的子节点中展开，计算累计概率：
       $$ \text{Score}(Item) = P(c_1) \times P(c_2 | c_1) \times \dots \times P(c_M | c_{1:M-1}) $$
    3. **输出 Top-K：** 当生成到第 $M$ 步（序列结束）时，我们就会得到概率（得分）最高的 $K$ 条完整且合法的序列。
    4. **ID 映射：** 将这 $K$ 个语义 ID 直接映射回对应的真实商品，即完成了 Top-K 召回。

<summary><b>点击展开：为什么这种方案比传统双塔（Two-Tower）更强？</b></summary>

1. **打破了内积的表达瓶颈：** 双塔模型强行用两个向量的内积（Dot Product）来衡量用户和物品的匹配度，表达能力有限。Transformer 直接让用户历史和目标 Item 在深层网络中进行交叉（Self-Attention），属于早期交叉（Early Crossing），表达能力上限极高。
2. **彻底解决了冷启动问题：** 新 Item 只要有文本或图片，RQ-VAE 就能给它分配一个合法的语义 ID（前缀必然和相似老物品相同）。Transformer 即使没见过这个新 ID，也能通过预测前缀将其召回。
3. **极大的词表压缩：** 传统 Embedding table 需要存储亿级别的 Item ID，参数量巨大且稀疏。使用 $M=3, V=256$ 的 RQ-VAE，总容量为 $256^3 \approx 1600万$，但模型实际只需要维护 $3 \times 256 = 768$ 个 Token 的 Embedding，参数量锐减。
</details>

### 总结
你所说的“像分类器一样打分取 Top-K”的直觉非常准确。它的本质就是**把亿级别的 Item 检索任务，降维成了短序列（长度约 3~4）的、小词表（几百到几千）的连续多分类任务**。利用 Transformer 强大的序列概率预测能力，配合 Trie 树限制查找空间，从而直接计算出用户最可能点击的 Top-K 物品。
