# DPO易崩因数据噪声直接冲击模型

### 🧑‍💻 User

为什么dpo特别容易训崩，尤其是打分数据质量不好情况下

---

### 🤖 Assistant

Direct Preference Optimization (DPO) 是一种非常优雅的对齐算法，它通过数学推导将奖励模型（Reward Model）和策略模型（Policy Model）合二为一。但正是这种**“端到端”的直接性**，导致它在面对低质量、高噪声的偏好数据时，比传统的基于 PPO 的 RLHF 更加脆弱，极易“训崩”（如生成乱码、陷入重复、模型能力退化）。

主要原因可以归结为以下几个核心机制：

### 1. 缺乏“奖励模型（RM）”作为缓冲层
在传统的 RLHF 中，训练分为两步：先训 RM，再用 PPO 训大模型。
*   **RLHF 的缓冲效应：** 奖励模型在拟合偏好数据时，会学习到数据的**整体分布和一般规律**。如果数据中有 15% 是标错的（噪声），RM 通常能泛化出那 85% 正确的规律，从而“平滑”掉噪声。PPO 强化学习阶段面对的是 RM 输出的平滑标量，而不是直接面对错误数据。
*   **DPO 的直接冲击：** DPO 移除了 RM 这个“减震器”。大模型直接基于偏好数据对 $\pi_\theta$ 计算交叉熵损失。如果打分数据质量差（比如把差的回答标成了 Chosen，好的标成了 Rejected），模型会**直接且毫无保留地吃下这些有害梯度**，导致原有的语言能力受到严重破坏。

### 2. 损失函数的“无限拉扯”与梯度爆炸
我们可以看一下 DPO 的核心梯度/损失逻辑。DPO 旨在最大化 Chosen ($y_w$) 和 Rejected ($y_l$) 之间的隐式奖励差：

$$ \hat{r}(x, y) = \beta \log \frac{\pi_\theta(y|x)}{\pi_{ref}(y|x)} $$

DPO 倾向于不断推高 $y_w$ 的生成概率，并拼命压低 $y_l$ 的生成概率。
*   **好数据下：** 模型学会提高高质量文本的概率。
*   **坏数据下：** 如果 $y_l$ 其实是逻辑严密的好回答，而 $y_w$ 是有语病或乱码的差回答，DPO 会强行**惩罚模型生成好词的概率**，并**鼓励模型生成坏词**。
*   由于对数概率 $\log \pi$ 是无界的，在极端错误数据的引导下，模型为了最大化这个差值，会将某些 Token 的概率推向 0 或 1，导致概率分布崩塌（Mode Collapse），最终表现就是**模型只会不断重复某几句话，或者彻底生成乱码**。

### 3. 对 KL 散度约束的破坏
DPO 公式中的 $\beta$ 参数是用来控制模型偏离参考模型（$\pi_{ref}$，即 SFT 模型）的程度（KL 散度）。
当数据质量极差（存在大量矛盾标签，比如同一个问题，A 数据对偏好长回答，B 数据对偏好短回答）时，模型在优化过程中会感到“无所适从”。为了强行降低 Loss，模型会产生极大的梯度更新，这会轻易冲破 $\beta$ 设定的软约束。一旦模型严重偏离了 $\pi_{ref}$ 的基础语言分布，它就“忘了怎么正常说话”，也就是我们常说的“遗忘灾难（Catastrophic Forgetting）”。

### 4. 极易学到“伪特征”（Shortcut Learning）
当打分数据质量不好（通常表现为标注者偷懒或缺乏专业知识）时，数据往往会包含一些表面特征偏差。最典型的就是**长度偏见（Length Bias）**和**排版偏见**。
*   如果标注者只是无脑给字数多的回答打高分。
*   DPO 没有 RM 来综合评估语义质量，它会极其敏锐地捕捉到“只要字数多，Loss 就低”这个捷径。
*   结果：模型训完之后变成了“废话生成器”，为了凑字数而疯狂产生幻觉或重复句式。

<summary>💡 <b>展开查看：如何缓解 DPO 训崩的问题？</b></summary>

如果你在使用 DPO 时遇到了崩溃，可以尝试以下几个维度的改进：

1.  **清洗数据（最关键的一步）：**
    *   过滤掉长度差异极大的偏好对（消除 Length Bias）。
    *   使用强模型（如 GPT-4 或 Claude 3.5）对你的偏好数据进行重新打分（AI-as-a-Judge），剔除置信度低、两者质量相近或明显标错的对子。
    *   **宁缺毋滥：** 1000 条高质量的 DPO 数据远比 10000 条充满噪声的数据效果好。

2.  **调整超参数：**
    *   **降低学习率（Learning Rate）：** DPO 对学习率非常敏感，通常应该比 SFT 阶段低一个数量级（例如 $5e-7$ 或更低）。
    *   **增大 $\beta$ 值：** $\beta$ 越大，模型受参考模型的约束越强。如果模型容易乱码，尝试调高 $\beta$（例如从 0.1 调到 0.2 或 0.5）。

3.  **尝试更鲁棒的算法变体：**
    *   **IPO (Identity Preference Optimization)：** 添加了正则项，防止过度惩罚 Rejected 样本，对噪声数据容忍度更高。
    *   **KTO (Kahneman-Tversky Optimization)：** 不需要成对的偏好数据，只需标记某段文本是好是坏，在某些噪声场景下表现更好。
    *   **ORPO (Odds Ratio Preference Optimization)：** 将 SFT 和对齐合并为一步，不需要参考模型，有时能带来更稳定的训练过程。

4.  **在 SFT 阶段打好基础：**
    *   确保用作 $\pi_{ref}$ 的 SFT 模型本身具有足够好的指令遵循能力。如果 SFT 模型本来就很弱，DPO 也很难把它的上限提上去，反而更容易崩溃。

</details>

### 🧑‍💻 User

grpo 同样也移除了 critic 这个模型，为什么它的鲁棒性稍微好一些？

---

### 🤖 Assistant

这是一个非常敏锐且专业的问题！要回答这个问题，我们首先需要**澄清一个概念上的微妙差异**：

DPO 和 GRPO 虽然在架构上都做到了“减负”，但它们移除的**并不是同一个东西**。
*   **DPO 移除的是“奖励模型 (Reward Model, RM)”**：大模型直接吃偏好数据（人工标注的 A 比 B 好）。
*   **GRPO (Group Relative Policy Optimization) 移除的是 PPO 中的“评论家模型 (Critic Model / Value Model)”**：但它**依然保留了奖励信号 (Reward)**（通常是基于规则的判分，或者独立的 RM）。

正是因为这种机制上的根本差异，使得 GRPO 的鲁棒性远好于 DPO，也比传统的 PPO 更容易训练。具体原因可以归结为以下四点：

### 1. 组内相对优势（替换 Critic），消除了估计误差
在传统的 PPO 中，Critic 模型的作用是预测当前状态的“基线价值（Baseline）”，用来计算**优势函数 (Advantage)**：
$$ A = Reward - Critic(State) $$
Critic 也是一个神经网络，它在训练中经常会**预测不准**。如果 Critic 瞎指挥，大模型的梯度更新就会跟着错，导致 PPO 极易崩溃。

**GRPO 的绝妙之处在于，它用一个极其简单的数学统计，完美替换了玄学的 Critic：**
对于同一个问题，GRPO 让模型生成一组（比如 $G=4$ 或 $8$ 个）回答。然后对这组回答打分，计算这组分数的均值 $\mu$ 和标准差 $\sigma$。每个回答的优势函数变成了：
$$ A_i = \frac{r_i - \mu}{\sigma} $$
*   **为什么更鲁棒？** 这个基线（均值）是**绝对真实、当下计算出来的**，而不是靠另一个神经网络猜出来的。它彻底消除了 Critic 训练滞后或预测误差带来的系统性风险。

### 2. 依然存在“缓冲层”或“绝对正确的奖励”
正如开头所说，GRPO 并没有放弃获取 Reward 这一步：
*   **在逻辑/数学/代码场景（如 DeepSeek-R1）：** GRPO 使用的是**基于规则的奖励（Rule-based Reward）**。结果对就是对，错就是错，**毫无噪声**。因为打分数据质量是 100% 完美的，模型自然不容易训崩。
*   **在通用对话场景：** 即使 GRPO 使用 RM 打分，RM 依然充当了前文提到的**“缓冲层”**，平滑了人类偏好数据中的噪声。

### 3. PPO 家族的截断机制（Clipping）防止了梯度爆炸
DPO 在极端数据下会为了拉大好坏样本的差距，产生无限大的拉扯（导致概率归零或拉满）。
而 GRPO 继承了 PPO 家族的核心保护机制——**Clip 截断**。其目标函数中包含：
$$ \min \left( \frac{\pi_\theta}{\pi_{old}} A_i, \text{clip} \left( \frac{\pi_\theta}{\pi_{old}}, 1-\epsilon, 1+\epsilon \right) A_i \right) $$
*   这个公式硬性规定了：无论某个回答的优势有多大（即使是满分），**模型在单步更新中，概率变化的幅度也不能超过 $\epsilon$**（通常设为 0.2）。
*   这种“步步为营”的机制，使得 GRPO 即使偶尔遇到质量不好的打分，也不会像 DPO 那样“一脚油门冲下悬崖”。

### 4. 显式的 KL 散度惩罚
DPO 虽然在推导时考虑了 KL 散度，但在实际的 Loss 函数中，它是将其隐式化了。
而 GRPO 通常会在目标函数中加入**显式、实时的 Token 级别 KL 惩罚**：
$$ - \beta D_{KL}(\pi_\theta \| \pi_{ref}) $$
在每生成一个词时，如果模型偏离参考模型 $\pi_{ref}$ 太远，就会直接扣分。这种实时的硬性约束，极大地防止了模型能力的退化和乱码的产生。

<summary>📊 <b>展开查看：DPO, PPO, GRPO 架构依赖对比总结</b></summary>

| 算法 | 策略模型 (Policy) | 奖励模型 (Reward) | 评论家模型 (Critic/Value) | 鲁棒性 / 崩溃风险 | 主要崩溃原因 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **DPO** | 必须有 | **❌ 移除** (隐式) | **❌ 无** | 极易崩溃 | 噪声数据直接冲击模型；概率拉扯无上限 |
| **PPO** | 必须有 | 必须有 | **✅ 必须有** (常与Policy同规模) | 较易崩溃 | Critic 估计不准导致 Advantage 计算错误 |
| **GRPO** | 必须有 | 必须有 (或用规则) | **❌ 移除** (用组内统计替代) | **相对稳定** | 结合了 PPO 的截断保护，又去除了 Critic 的不稳定性 |

**总结：**
GRPO 丢掉的是会带来不确定性的“预测基线”（Critic），保留了起到缓冲/导航作用的“奖励”（Reward），并且带着安全带（Clip 机制）。而 DPO 丢掉了“缓冲层”（Reward Model），且没有速度限制（无 Clip），因此对路况（数据质量）的要求极为苛刻。

</details>
