# 预训练不足导致SFT越训越傻

### 🧑‍💻 User

为什么基模第一轮sft越训越傻一般是预训练没做好导致的，尤其是见过大量pretrain ood外的sample，表现就是val loss快速下降然后越训越高

---

### 🤖 Assistant

这是一个非常经典且在开源社区和企业自研大模型过程中经常遇到的现象。业界有一句名言（源自 LIMA 论文）：**“大模型的知识来自于预训练，SFT 只负责激发和对齐格式。”**

当你发现基座模型（Base Model）在第一轮 SFT 时“越训越傻”，且验证集 Loss（Val Loss）呈现先快速下降后迅速反弹升高的“U型”曲线，这确实高度指向**预训练阶段的缺陷（特别是数据多样性和泛化能力的不足）**。

以下是导致这种现象的深层机制分析：

### 1. 核心矛盾：用 SFT 的方式强行注入 OOD 知识
预训练的目标是学习世界知识和底层逻辑（拟合数据分布 $P(X)$），而 SFT 的目标是学习人类交互格式和指令遵循（拟合条件概率 $P(Y|X)$）。

如果 SFT 数据中包含了大量在预训练阶段从未见过的 Out-of-Distribution (OOD) 样本（例如预训练全是中文网文，SFT 突然让它做复杂的英文数学推理），模型会面临灾难性的困境：
*   **无法“激发”**：因为预训练的权重参数 $\theta$ 中根本没有相关的知识储备，模型无法通过微调建立 $X$ 和 $Y$ 之间的有效映射。
*   **被迫“死记硬背”**：为了降低训练集的 Loss，模型只能动用极其有限的 SFT 步数和数据量去**强行记忆（Memorization）**这些 OOD 样本。
*   **结果（越训越傻）**：强行记忆会导致模型破坏预训练阶段建立的脆弱的通用特征表示（Representation Collapse）。它不仅没学会新知识的泛化，反而把原来预训练学到的通用常识（Generalization）给遗忘了，这就是经典的**灾难性遗忘（Catastrophic Forgetting）**。

### 2. 解释 Loss 曲线：为什么 Val Loss 先骤降后狂飙？

你观察到的 Val Loss 曲线是这种现象的完美数学体现：

*   **第一阶段：Val Loss 快速下降（格式学习期）**
    在 SFT 的前几百步，模型会非常迅速地掌握指令微调的**表面模板（Surface-level Formatting）**。例如学会了 `<|user|>` 和 `<|assistant|>` 的对话结构，学会了用特定的语气开头。因为格式学习的难度很低，这会导致 Loss 出现快速下降。
*   **第二阶段：Val Loss 快速反弹升高（表征崩溃期）**
    当格式学完后，梯度开始逼迫模型去拟合 SFT 数据中的 **OOD 核心语义/知识**。
    由于基模在预训练时没有构建好相关的流形空间（Manifold），梯度更新 $\nabla_\theta L_{SFT}$ 会导致参数发生剧烈且不规则的偏移。
    $$ \theta_{new} = \theta_{old} - \alpha \nabla_\theta L_{SFT}(X_{OOD}, Y_{OOD}) $$
    这种剧烈的更新破坏了模型原本的泛化能力，导致模型在验证集（通常包含各种类型的测试用例）上的表现断崖式下跌，Loss 越训越高。

### 3. 为什么说“归根结底是预训练没做好”？

如果 SFT 数据成了 OOD，说明预训练数据的**覆盖面（Coverage）**和**高质量数据的密度（Density）**不够。

1.  **数据配比失衡**：如果在预训练时，代码、数学、多语言、逻辑推理类的高质量语料占比太低，基模的隐空间（Latent Space）就没有为这些高级能力预留位置。
2.  **模型容量未被有效激活**：如果预训练 Loss 没有降到足够低，或者预训练数据太脏，模型的参数并没有收敛到一个平滑且鲁棒的局部最优解。此时的基模就像一个“根基不稳的建筑”，SFT 数据稍微一用力推（OOD 数据带来的大梯度），整个建筑（通用能力）就塌了。

<summary><b>👉 诊断与解决方案（点击展开）</b></summary>

如果你目前卡在这个阶段，可以尝试以下策略来抢救或验证问题：

**1. SFT 阶段的抢救方案（治标）：**
*   **降低学习率 (Learning Rate)**：对于不够鲁棒的基模，SFT 的 LR 需要设置得比正常值低（例如降到 $5 \times 10^{-6}$ 或更低），防止梯度爆炸破坏预训练权重。
*   **混合预训练数据 (Pre-train Data Mixing)**：在 SFT 数据中混入 10% - 20% 的高质量预训练数据。这种正则化手段可以像锚点一样，防止模型在拟合 OOD 数据时发生灾难性遗忘。
*   **清洗 SFT 数据**：过滤掉那些明显超出基模当前能力的超困难样本（Hard OOD）。强迫一个只有小学水平的基模去做微积分 SFT，只会让它连加减法都忘记。
*   **提前停止 (Early Stopping)**：在 Val Loss 达到最低点（即刚学完格式，还没开始破坏知识时）直接停止训练。

**2. 预训练阶段的根本解决（治本）：**
*   **提升预训练数据多样性**：如果 SFT 要做特定垂直领域（如医疗、法律）或特定能力（如代码），必须**退回预训练阶段**，进行 Continual Pre-training (CPT)，注入海量该领域的无标签文本，构建好底层概率分布后，再进行 SFT。
*   **退火训练 (Annealing)**：在预训练的最后 5%-10% 阶段，使用极高质量、多样化的指令或书籍数据进行退火，拉高基模的通用泛化上限。

哪些类型的sft数据可能会被认为是“ood”需要剔除？

---

### 🤖 Assistant

在 SFT 阶段，所谓的“OOD（分布外）”数据并不是绝对的，而是**相对基座模型（Base Model）的预训练数据分布而言的**。

如果 SFT 数据中包含了基模在预训练时**完全没见过、或者见得极少**的知识、逻辑或语言，这些数据就会成为“毒药”。对于一个预训练不够充分的基模，以下几类 SFT 数据通常会被视为 OOD，在第一轮微调时需要特别谨慎甚至剔除：

### 1. 深度垂直领域的专业知识（Knowledge OOD）
如果你的预训练语料主要是通用网页（Common Crawl）、新闻和百科，那么以下 SFT 数据绝对是 OOD：
*   **高难度医疗诊断**：例如包含复杂医学术语、生化指标分析的病历推理。
*   **严肃法律咨询**：需要引用特定法条、撰写格式严谨的法律合同。
*   **前沿学术文献解析**：量子物理、高级材料学等包含大量专业名词和推导的问答。
*   **原因**：基模的词表和隐空间里根本没有这些低频词汇及其关联关系的表征（Representation）。强行 SFT 会导致模型用通用词汇去生搬硬套专业词汇，破坏原有的语言连贯性。

### 2. 复杂的逻辑与符号推理（Reasoning OOD）
这是导致模型“越训越傻”的重灾区。大模型的推理能力高度依赖预训练阶段的代码和数学语料。
*   **复杂数学题（如 MATH 数据集）**：包含多步推导、复杂代数公式、微积分的数据。如果在预训练时没有见过大量的 $\LaTeX$ 公式和解题步骤，SFT 时直接让它做解答题，它只会死记硬背最终答案或胡言乱语。
*   **未见过的编程语言或复杂代码生成**：如果基模只见过少量的 Python 基础代码，你却在 SFT 中塞入大量 C++ 并发编程、Rust 内存管理或复杂的 SQL 查询语句。
*   **原因**：逻辑链条（Chain-of-Thought）不是通过 SFT 凭空产生的，而是预训练阶段逻辑模式的激发。没有底层逻辑，强行拟合多步推理会导致严重的灾难性遗忘。

### 3. 未覆盖的语种（Language OOD）
*   **小语种指令**：如果基模是纯中英双语预训练的，SFT 数据中却包含了阿拉伯语、韩语或俄语的问答对。
*   **多语言翻译对**：在未掌握目标语言语法的情况下，要求模型进行中日、中法的高质量互译。
*   **原因**：模型为了降低这些小语种样本的 Loss，会剧烈调整底层注意力权重，直接导致原本流利的中英文能力退化。

### 4. 严苛的格式控制与超长上下文（Structural OOD）
*   **复杂的结构化输出（如纯 JSON/XML）**：如果指令要求“仅输出符合特定 Schema 的 JSON，不包含任何解释”，而基模在预训练时没怎么见过纯净的 JSON 文本。
*   **超出预训练长度的推理**：预训练时的 Context Window 只有 4K，SFT 数据中却塞入了大量长度达到 8K 甚至 16K 的“大海捞针”或长文档摘要任务。
*   **原因**：超越位置编码（Positional Encoding）泛化上限的数据，或极端的格式约束，会让模型产生困惑，表现为输出乱码或陷入无限重复。

<details>
<summary><b>🛠️ 实践技巧：如何科学地找出并剔除这些 OOD 数据？（点击展开）</b></summary>

在实际炼丹中，不能仅凭直觉剔除数据。可以通过以下量化方法找出 SFT 训练集中的 OOD 样本：

**1. 基模 Perplexity (PPL) 过滤法**
这是最有效的方法。在 SFT 开始前，让基模在 Zero-shot 状态下对 SFT 数据集进行推理，计算基模对每一条数据目标回复 $Y$ 的困惑度（Perplexity）或交叉熵损失（Cross-Entropy Loss）：
$$ Loss_{base} = -\frac{1}{N} \sum_{i=1}^{N} \log P_{base}(y_i | X, y_{<i}) $$
*   **操作**：按照 $Loss_{base}$ 对所有 SFT 样本进行排序。
*   **剔除**：将 Loss 极高（例如排名前 5%-10%）的样本视为 OOD 予以剔除。这些就是基模“看都看不懂”的数据。

**2. 梯度范数（Gradient Norm）监控**
如果不想提前过滤，可以在 SFT 训练的前几十步，监控每个 Batch 的梯度。
*   如果某个 Batch 导致梯度范数 $\|\nabla_\theta L\|$ 突然飙升，说明这个 Batch 中包含严重的 OOD 样本。可以记录下这些样本并将其移出训练集。

**3. 循序渐进的 Curriculum Learning (课程学习)**
不要一开始就把最难的 SFT 数据丢给模型。
*   **Phase 1**：先用基模非常熟悉的数据（如简单的闲聊、基础问答）进行几百步的格式微调。
*   **Phase 2**：逐渐混入中等难度的推理数据。如果在这个阶段加入某类数据导致 Val Loss 剧烈反弹，就立刻定位到该类数据，说明基模的能力还没准备好，需要退回预训练阶段去“补课” (Continual Pre-training)。

</details>
