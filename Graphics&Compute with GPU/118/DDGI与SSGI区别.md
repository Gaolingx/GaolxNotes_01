# DDGI与SSGI区别

### 🧑‍💻 User

ddgi和常规的ssgi有什么区别？

---

### 🤖 Assistant

**DDGI** (Dynamic Diffuse Global Illumination，动态漫反射全局光照) 和 **SSGI** (Screen Space Global Illumination，屏幕空间全局光照) 是现代实时渲染中用于计算间接光照（全局光照）的两种完全不同的技术路线。

它们最核心的区别在于**数据来源**和**空间维度**：SSGI 仅依赖当前屏幕上可见的像素信息，而 DDGI 则依赖真实的三维世界空间和光线追踪。

以下是两者的详细对比：

### 1. 核心原理对比

*   **SSGI (屏幕空间全局光照)**
    SSGI 是一种基于后处理（Post-processing）的技术。它利用渲染管线生成的 `G-Buffer`（如深度图、法线图、颜色图），在屏幕的二维像素空间内进行光线步进（`Ray Marching`）。如果一条在屏幕空间射出的光线碰撞到了屏幕上的另一个像素，它就会将那个像素的颜色和亮度作为反弹光照亮起点像素。
*   **DDGI (动态漫反射全局光照)**
    DDGI（通常与 NVIDIA 的 RTXGI 技术紧密相关）是一种基于真实 3D 空间的技术。它在场景中放置一个或多个 3D 探针网格（`Probe Grids`）。利用硬件光线追踪（Hardware Ray Tracing），探针向四周发射光线，收集场景中的光照信息，并将其存储为球面调和函数或八面体映射（Octahedral Mapping）。渲染物体时，物体会从周围最近的探针中采样光照。

---

### 2. 关键特性差异表

| 特性 | SSGI (常规屏幕空间) | DDGI (动态探针网格) |
| :--- | :--- | :--- |
| **空间感知** | **局限于屏幕内**（无法感知摄像机背后或被遮挡的物体）。 | **真实 3D 空间感知**（可以反射屏幕外、被遮挡物体的光照）。 |
| **视觉稳定性** | 差。随着镜头转动，屏幕边缘的间接光会突然消失（Popping）。 | 极佳。光照信息存储在世界空间的探针中，与摄像机视角无关。 |
| **硬件要求** | 极低。依赖传统的栅格化着色器计算。 | 较高。通常需要支持硬件光线追踪（DXR/Vulkan RT）的显卡。 |
| **性能消耗** | 固定开销，与屏幕分辨率成正比 $O(Pixels)$。 | 开销受探针密度、发射光线数量影响，且通常带有时间性累积（Temporal Accumulation）。 |
| **多次反弹 (Multi-bounce)**| 极难实现，通常只有一次反弹。 | 容易实现无限次反弹（通过探针之间的相互采样更新）。 |

---

### 3. 技术难点与数学原理

#### DDGI 的防漏光机制 (Chebyshev Visibility)
DDGI 虽然在 3D 空间工作，但由于探针是离散放置的，很容易出现“漏光”现象（例如墙外的探针照亮了墙内的地板）。为了解决这个问题，DDGI 不仅记录光照颜色，还记录光线击中表面的距离（深度）及其平方。
在着色时，DDGI 使用切比雪夫不等式（Chebyshev's Inequality）来计算可见性概率 $P$。当采样点深度 $d$ 大于探针记录的平均深度 $\mu$ 时，可见性上限公式如下：

$$ P(x > d) \le \frac{\sigma^2}{\sigma^2 + (d - \mu)^2} $$

*其中 $\sigma^2$ 是深度的方差。通过这个数学模型，DDGI 可以有效地判断采样点是否被遮挡，从而消除大部分漏光。*

#### SSGI 的厚度假设
由于 SSGI 只有屏幕表面的深度信息（本质上是一个 2.5D 高程图），它不知道物体的背面在哪里。在进行 `Ray Marching` 时，它必须人为假设物体有一个“固定厚度”。如果光线步进的深度穿过了这个假设的厚度，算法就会出现误判，导致光影错误。

### 4. 优缺点详情 (展开查看)

<details>
<summary><strong>🟢 SSGI 的优缺点</strong></summary>

*   **优点 (Pros):**
    *   **性能极佳：** 适合低端硬件、移动端或对帧率要求极高的竞技游戏。
    *   **易于集成：** 只要引擎有标准的延迟渲染（Deferred Shading）管线，就可以轻松加入 SSGI，不需要修改场景资产。
    *   **高频细节好：** 在屏幕空间内，能够很好地处理微小物体之间的接触间接光照（类似于 SSDO）。
*   **缺点 (Cons):**
    *   **画外信息丢失：** 最大的硬伤。一个发红光的物体只要移出屏幕边缘，它对场景照亮的红光就会瞬间消失。
    *   **视角依赖伪影：** 当你低头看地板时，天花板的光照贡献会变为 0。
</details>

<details>
<summary><strong>🔵 DDGI 的优缺点</strong></summary>

*   **优点 (Pros):**
    *   **物理正确性：** 提供了非常接近离线渲染的漫反射全局光照质量。
    *   **时间/空间稳定性：** 无论摄像机怎么移动，光照都是稳定的。
    *   **性能可伸缩：** 可以通过调整探针的更新频率（例如每帧只更新 10% 的探针）来摊薄光线追踪的性能开销。
*   **缺点 (Cons):**
    *   **参数调试繁琐：** 开发者需要在场景中手动或半自动地摆放 `Probe Volumes`，并调整密度以避免漏光。
    *   **光照延迟：** 由于采用了时间累积（Temporal Reprojection）和摊薄更新，当场景光源发生剧烈变化（如突然开灯）或快速移动时，间接光照可能需要几十毫秒才能“追上”变化。
</details>

### 总结：如何选择？

*   如果你在开发一款**主打写实画质的 3A 游戏**（例如《赛博朋克 2077》、《控制》），且目标平台是次世代主机（PS5/XSX）或中高端 PC，**DDGI**（或类似的硬件光追 GI）是必选项，因为它能提供真实的沉浸感。
*   如果你在开发一款**跨平台游戏、手游**，或者场景多为室外、对光影的物理准确性要求不高，那么 **SSGI** 配合传统的烘焙光照图（Lightmaps）和环境光探针（Cubemaps）是性价比最高的成熟方案。目前很多引擎（如 Unreal Engine 的 Lumen 在低画质下，或 Unity 的 URP）都会使用 SSGI 作为兜底方案。

### 🧑‍💻 User

为什么ue5 lumen是两者都结合了？

---

### 🤖 Assistant

Unreal Engine 5 的 **Lumen** 是一套极其精妙的混合（Hybrid）全局光照系统。它之所以将屏幕空间技术（SSGI）和真实 3D 空间技术（类似 DDGI 的硬件/软件光追 + 表面缓存）结合在一起，是为了追求**“既要极致画质，又要兼顾性能”**。

简单来说，Lumen 采用这套混合方案的核心原因可以总结为四个字：**取长补短**。

以下是 Lumen 结合两者的具体原因和底层逻辑：

### 1. 弥补 3D 空间 GI 在“微观细节”上的不足 (高频细节)
无论是 DDGI 的探针网格，还是 Lumen 自己的 `Surface Cache` (表面缓存) 和 `Radiance Cache` (辐射度缓存)，它们在 3D 空间中的数据精度都是**相对较低的（低频信息）**。
*   如果完全依靠 3D 光追，为了计算两本叠在一起的书之间的微小漫反射和接触阴影（Contact GI），你需要极高的射线采样率和极高分辨率的 3D 缓存，这会瞬间吃光显存和 GPU 算力。
*   **结合 SSGI 的优势：** 屏幕空间包含了 1:1 的像素级精准几何与材质信息。Lumen 首先在屏幕空间发射光线，这样就能完美且廉价地捕捉到极微小物体之间的光线反弹（高频细节）。

### 2. 极致的性能优化 (光线追踪非常昂贵)
在真实的 3D 空间中进行光线追踪（无论是遍历 BVH 树的硬件光追，还是 Lumen 基于距离场 SDF 的软件光追），计算开销 $O(\log N)$（$N$ 为场景复杂度）加上内存带宽读取，都是非常昂贵的。
*   **屏幕空间的廉价性：** 在屏幕二维空间（`G-Buffer`）中进行光线步进（Ray Marching）是非常廉价的矩阵运算，开销极小。
*   **结合策略：** Lumen 遵循**“能省则省”**的原则。如果一条光线在屏幕空间内就能命中物体，Lumen 就会直接使用屏幕空间的数据；只有当光线飞出屏幕，或者被屏幕遮挡时，才会去调用昂贵的 3D 空间光追。

### 3. 完美解决 SSGI 的“画外消失”问题
正如前面提到的，纯 SSGI 最大的痛点是视口边缘的伪影（当发光物体移出屏幕时，光照突然消失）。
*   Lumen 利用 3D 光追作为 SSGI 的**“兜底方案（Fallback）”**。当屏幕空间光线没有命中任何东西（比如光线射向了摄像机背后），Lumen 就会无缝切换到 3D 场景表达（SDF 或硬件光追）去寻找碰撞点。这样玩家既享受了 SSGI 的高性能和高细节，又不会看到任何光照闪烁或断层。

---

### Lumen 的混合追踪管线是如何工作的？(展开查看)

Lumen 发射一条光线去寻找间接光照时，实际上经历了一个层层递进的**降级/回退（Fallback）机制**。它的优先级如下：

<details>
<summary><strong>第一步：屏幕空间追踪 (Screen Space Tracing) —— 最快、最精细</strong></summary>

*   光线首先在当前的屏幕深度图和颜色图中进行追踪。
*   **结果：** 如果命中了屏幕上可见的像素，直接读取该像素的光照和颜色，追踪结束。这解决了微小几何体的接触 GI。
</details>

<details>
<summary><strong>第二步：软件/硬件光线追踪 (Software/Hardware Ray Tracing) —— 中等成本、解决画外信息</strong></summary>

*   如果第一步光线飞出了屏幕，或者被遮挡（Screen Space Trace Miss）。
*   Lumen 会在 3D 空间中继续追踪这条光线。
    *   在未开启硬件光追时，Lumen 会追踪场景的 **SDF (Signed Distance Fields，有向距离场)**。
    *   在开启硬件光追时，Lumen 会利用显卡的 RT Core 追踪真实的三角面（BVH）。
*   **结果：** 光线命中 3D 模型后，Lumen 会从它的核心创新技术 `Surface Cache` (一张包含了场景材质和光照的低分辨率 3D 贴图集) 中读取光照颜色。
</details>

<details>
<summary><strong>第三步：辐射度缓存与天空 (Radiance Cache & Sky) —— 解决多次反弹</strong></summary>

*   对于更远处的背景，或者为了计算无限次的光线反弹（Multi-bounce GI）。
*   Lumen 使用了类似于 DDGI 探针理念的 `Radiance Cache`。它在空间中稀疏地分布着探针来记录宏观的光照环境。
</details>

### 总结

UE5 的 Lumen 并不是简单地“同时开启两个效果”，而是构建了一个**优雅的分层架构**：

1.  用 **SSGI** 负责近处的、像素级的**高频细节**，节约性能。
2.  用 **3D 光追 (结合 Surface Cache)** 负责屏幕外和宏观空间的**低频信息**，保证物理正确性。

通过这种混合方式，Lumen 成功地在当代硬件（如 PS5 级别的机能）上，实现了不需要极高算力就能跑出接近离线渲染级别的全动态全局光照。
