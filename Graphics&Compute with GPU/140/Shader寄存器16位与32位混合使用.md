# Shader寄存器16位与32位混合使用

### 🧑‍💻 User

在实时渲染中，shader一般使用16还是32位寄存器？

---

### 🤖 Assistant

在现代实时渲染中，Shader **通常是 16 位和 32 位寄存器混合使用的**，而不是绝对地只用其中一种。

具体使用哪种精度，主要取决于**变量的物理意义**、**目标平台（移动端 vs PC/主机）**以及**性能优化的需求**。在着色器语言中，32 位通常用 `float` 或 `highp` 表示，而 16 位通常用 `half` 或 `mediump` 表示。

以下是具体的应用场景和选择标准：

### 1. 必须使用 32 位（FP32 / `float` / `highp`）的场景
32 位浮点数提供了高精度，适用于对微小数值变化敏感，或者跨度范围极大的数据。如果这些数据降级到 16 位，会导致明显的视觉瑕疵（如画面抖动、模型撕裂、闪烁）。

*   **顶点位置与世界坐标 (World Position)：** 尤其是在大世界地图中，16 位的精度不足以表示远离原点的坐标，会导致模型在移动时发生剧烈抖动。
*   **纹理坐标 (UV Coordinates)：** 当使用很大的纹理图集（Texture Atlas）或者高频率平铺（Tiling）时，16 位会导致采样偏移，产生马赛克或像素扭曲。
*   **深度值 (Depth)：** 深度测试对精度要求极高，精度不足会导致严重的 Z-Fighting（闪烁穿插）问题。
*   **时间变量 (Time)：** 用于动画（如水面波浪、风吹草动）的时间累加器，如果用 16 位，运行一段时间后精度会耗尽，导致动画卡顿或停止。

### 2. 推荐使用 16 位（FP16 / `half` / `mediump`）的场景
16 位浮点数可以显著减少 GPU 寄存器的占用压力（提高 Warp/Wavefront 的并行度），降低显存带宽消耗，并降低功耗。

*   **颜色和光照 (Colors & Lighting)：** 绝大多数颜色数据（如 Albedo、Specular、光照强度）不需要极高的精度。即使是 HDR 颜色，16 位通常也足够表示了。
*   **法线和向量 (Normals & Directions)：** 大多数法线贴图采样出来的数据以及光照方向向量，在归一化后处于 $[-1, 1]$ 之间，16 位精度完全可以满足视觉需求。
*   **后处理特效 (Post-processing)：** 比如 Bloom（泛光）、模糊（Blur）、颜色分级（Color Grading），中间计算过程使用 16 位可以大幅提升性能。

不同平台的 GPU 架构对 16 位和 32 位的处理方式有所不同：

1.  **移动端（手机 / 平板 - OpenGL ES, Vulkan, Metal）：**
    *   移动端对功耗和发热极其敏感。移动 GPU（如 Mali, Adreno, Apple GPU）在硬件底层原生支持并极度依赖 16 位（`mediump`）优化。
    *   **黄金法则：** 在移动端写 Shader，除了位置、UV 等关键数据，其他应尽可能全部使用 16 位。

2.  **现代 PC / 主机端（DirectX 12, Vulkan）：**
    *   **历史背景：** 早期的 PC GPU 实际上没有真正的 16 位 ALU，即使在 Shader 里写了 `half`，硬件也会自动转成 `float` (32位) 来计算，因此以前在 PC 上优化精度意义不大。
    *   **现代硬件（AMD RDNA, NVIDIA Turing/Ampere/Ada）：** 现代桌面 GPU 引入了对 FP16 的原生加速支持（例如 AMD 的 Rapid Packed Math 和 NVIDIA 的 Tensor Cores/FP16 单元）。现代 GPU 可以在一个 32 位的 ALU 周期内同时执行两个 16 位的计算（Packed Math）。
    *   **趋势：** 即使在 3A 大作中，为了榨干性能（尤其是配合光线追踪和 AI 缩放技术），开发者也越来越倾向于在 PC 上积极使用 16 位优化。
</details>

<details>
<summary><b>🛠️ 性能与寄存器压力 (Register Pressure)（展开查看）</b></summary>

在 GPU 架构中，寄存器是极其宝贵的资源。

*   GPU 通过将大量线程（Threads）打包成组（Warp 或 Wavefront）来并行执行。
*   如果一个 Shader 写得太复杂，使用了过多的 32 位变量，就会导致**寄存器溢出 (Register Spilling)**，或者**降低占用率 (Occupancy)**（即 GPU 无法同时调度足够多的线程组来掩盖内存延迟）。
*   将变量从 32 位降级到 16 位，可以直接节省一半的寄存器空间，从而允许 GPU 并发执行更多的顶点或像素，带来直接的帧率提升。
</details>

### 总结建议

作为技术美术或图形程序员，通用的最佳实践是：**“默认使用 16 位，只有在出现视觉瑕疵（Artifacts）或物理意义必须要求高精度时，才提升到 32 位。”** 这种混合精度（Mixed Precision）策略是目前各大游戏引擎（如 Unreal Engine, Unity）的默认标准。

### 🧑‍💻 User

那fma操作是否很普遍（exp pow add multi），硬件是怎么完成的？

---

### 🤖 Assistant

在实时渲染和 GPU 架构中，**FMA（Fused Multiply-Add，融合乘加）操作极其普遍**。

不过，首先需要纠正一个概念分类：`add`（加法）和 `multi`（乘法）可以组合成 FMA 操作，但 `exp`（指数）和 `pow`（幂）**不属于** FMA，它们在 GPU 硬件中是由另一种完全不同的计算单元来处理的。

以下是关于这些操作在 Shader 中有多普遍，以及硬件是如何完成它们的详细解析。

### 1. FMA 操作有多普遍？

FMA 的数学表达式为 $A \times B + C$。在现代 GPU 中，FMA 是**最基础、执行最频繁的指令，没有之一**。

*   **无处不在的矩阵与向量计算：** 渲染中充满了坐标变换（矩阵乘法）和光照计算（点乘 Dot Product）。比如计算两个三维向量的点乘 $(x_1x_2 + y_1y_2 + z_1z_2)$，编译器会将其转化为一系列的 FMA 指令。
*   **线性插值 (Lerp / Mix)：** Shader 中极其常用的 `lerp(a, b, t)`，其数学本质是 $a \cdot (1 - t) + b \cdot t$，编译器通常会将其优化为 $t \cdot (b - a) + a$，这完美契合 FMA 操作。
*   **多项式逼近：** 许多复杂的物理光照公式（BRDF）在底层会被简化为多项式，而计算多项式（如秦九韶算法/霍纳法则）全都是 FMA 操作。

**编译器优化：** 当你在 Shader 中写出 `float d = a * b + c;` 时，编译器绝不会生成一条乘法指令和一条加法指令，而是**强制**将其编译为一条硬件原生的 `fma` 指令。

### 2. 硬件是如何完成这些操作的？

GPU 的流式多处理器（Streaming Multiprocessor, SM）内部有不同类型的执行单元，主要分为 **标准 ALU** 和 **SFU (Special Function Unit，特殊函数单元)**。

#### A. 标准 ALU 处理 FMA (以及 Add, Multi)

现代 GPU 将 FMA 设计为**单周期指令**。相比于“先乘后加”，FMA 硬件有两大优势：**速度快**（一条指令完成两步操作）和**精度高**。

*   **传统方式（非 FMA）：** 计算 $A \times B$ $\rightarrow$ 截断/舍入（损失一次精度） $\rightarrow$ 加上 $C$ $\rightarrow$ 再次截断/舍入（损失第二次精度）。
*   **FMA 硬件电路：**
    1.  硬件直接将 $A$ 和 $B$ 送入**乘法器树（如 Wallace Tree）**，保留完整的、未舍入的中间积（比如两个 32 位浮点数相乘，内部会保留完整的尾数精度）。
    2.  同时，将 $C$ 的尾数进行对齐。
    3.  通过一个**进位保留加法器 (Carry-Save Adder, CSA)** 将乘积与 $C$ 相加。
    4.  最后只进行**一次舍入 (Single Rounding)**。
  
这种硬件设计使得 $A \times B + C$ 在硅片上的执行速度与单纯的 $A + B$ 几乎一样快，且精度更高。

#### B. SFU (特殊函数单元) 处理 `exp`, `pow`, `sin`, `rcp`

`exp` 和 `pow` 在 Shader 中非常常见（例如计算高光 Shininess，或者 Gamma 校正），但它们在硬件层面上**非常昂贵**。标准的 ALU 算不了这些，必须交给专门的 **SFU** 来完成。

<details>
<summary><b>🛠️ 展开查看：硬件如何计算 pow 和 exp？</b></summary>

在硬件底层，**没有任何电路可以直接计算** $x^y$ **(pow)**。
GPU 的 SFU 通常只原生支持基数为 2 的操作，即 `exp2` ($2^x$) 和 `log2` ($\log_2(x)$)。

1.  **指数 `exp2` 和对数 `log2` 的硬件实现：**
    SFU 通常不使用复杂的泰勒展开，而是使用 **硬件查找表 (LUT, Look-Up Table)** 结合 **二次/三次多项式插值 (Minimax Polynomial)** 来逼近结果。
    这个多项式逼近的过程中，依然需要调用 ALU 的 **FMA** 单元来计算。

2.  **`pow(x, y)` 的硬件实现：**
    当你在 Shader 里写 `pow(x, y)` 时，编译器会利用数学恒等式 $x^y = 2^{y \cdot \log_2(x)}$ 将其拆解成三条底层指令：
    *   指令 1：SFU 计算 $t_1 = \log_2(x)$
    *   指令 2：ALU 计算 $t_2 = y \times t_1$ (乘法)
    *   指令 3：SFU 计算最终结果 $2^{t_2}$
  
    这意味着，一个简单的 `pow` 操作，实际上占用了多个计算周期，并且跨越了 SFU 和 ALU。

3.  **倒数 `rcp` (1/x) 和平方根倒数 `rsqrt` (1/$\sqrt{x}$)：**
    同样由 SFU 处理，通常使用牛顿-拉弗森迭代法 (Newton-Raphson iteration) 的硬件实现，配合查找表提供一个初始猜测值，然后迭代几次得到高精度结果。
</details>

### 总结与性能优化启示

1.  **FMA 是免费的午餐：** 在写 Shader 时，尽量把数学公式整理成 $A \times B + C$ 的形式，硬件会以最高效的方式处理它。
2.  **小心 SFU 瓶颈：** `pow`、`exp`、`sin`、`cos` 需要动用 SFU，而 GPU 中 SFU 的数量通常远少于标准 ALU（例如，比例可能是 1:4 或 1:8）。过多使用超越函数会导致 ALU 等待 SFU，降低性能。
3.  **优化技巧：** 如果遇到 `pow(x, 2.0)` 或 `pow(x, 4.0)`，**绝对不要**写成 `pow`，而应该写成 `x * x`（编译为普通乘法）或者连续乘法，这样可以直接使用又多又快的 ALU 而非拥挤的 SFU。

### 🧑‍💻 User

为什么这类操作一般不使用mma加速单元？

---

### 🤖 Assistant

这是一个非常深刻且直击现代 GPU 架构核心的问题。

你提到的 **MMA (Matrix-Multiply-Accumulate，矩阵乘加) 单元**，也就是我们常说的 **NVIDIA Tensor Cores (张量核心)** 或 **AMD Matrix Cores**。它们拥有极其恐怖的浮点运算能力，专为 AI 和深度学习设计。

既然 Shader 里有大量的 FMA 乘加操作，为什么不能直接扔给这些算力爆炸的 MMA 单元去加速呢？核心原因在于**数据结构、线程模型和数学本质的极度不匹配**。

以下是具体的解析：

### 1. 数据形状不匹配：标量/小向量 vs 密集矩阵
*   **Shader 的运算对象：** 在图形渲染中，Shader 的主要数据类型是标量（`float`）和小向量（`vec2`, `vec3`, `vec4`），用于表示坐标 $(x, y, z)$ 或颜色 $(r, g, b, a)$。
*   **MMA 的运算对象：** MMA 单元在硬件电路上是**写死的**，它只能接受特定维度的**二维矩阵块**。例如，早期的 Tensor Core 一次操作必须是 $4 \times 4$ 矩阵的相乘，而现代架构通常要求 $16 \times 16$ 甚至更大的矩阵块。

如果强行用 MMA 计算 Shader 里的 `vec3` 点乘或 FMA，你需要用大量的零去“填充”（Padding）一个 $16 \times 16$ 的矩阵。这不仅浪费了 MMA 99% 的算力，而且将零散的寄存器数据拼装成矩阵所消耗的时间（内存和寄存器搬运开销），远比直接用标准 ALU 算要慢得多。

### 2. 数学本质的局限性：纯线性代数
*   **MMA 只能做一件事：** 它的硬件电路只为 $$D = A \times B + C$$ （其中 $A, B, C, D$ 皆为矩阵）这一种纯线性代数操作而生。
*   **Shader 逻辑非常复杂：** 像你提到的 `exp`, `pow`, `sin`, `rcp`（倒数）等超越函数，属于非线性操作。MMA 单元的电路里根本没有查找表（LUT）或插值器，它在物理层面上**完全无法**计算这些操作，只能依赖 SFU。哪怕是 Shader 里的向量叉乘（Cross Product），放进矩阵乘法里也很难高效表达。

### 3. 执行模型的冲突：独立线程 vs 强制协同

<details>
<summary><b>🛠️ 展开查看：SIMT 线程模型与 MMA 的冲突</b></summary>

GPU 采用 **SIMT (单指令多线程)** 架构，通常 32 个线程（一个 Warp）或 64 个线程（一个 Wavefront）绑定在一起执行。

*   **常规 ALU 渲染（像素独立）：**
    在 Pixel Shader 中，一个线程对应屏幕上的一个像素。32 个像素虽然在一起执行同一条指令，但它们的**数据是独立的**，并且可能发生**分支发散 (Divergence)**。比如，像素 A 碰到了金属材质执行代码分支 1，像素 B 碰到了水面执行代码分支 2。标准 ALU 可以完美处理这种独立性。

*   **MMA 单元的要求（强制协同）：**
    调用一次 MMA 指令，**不是**让 1 个线程去算一个矩阵，而是**强制要求整个 Warp（32 个线程）协同工作**，共同凑齐矩阵 $A$ 和矩阵 $B$ 的数据，送给 MMA 算完后，再把结果矩阵 $D$ 拆分发还给这 32 个线程。
  
    在图形 Shader 中，你极难保证这 32 个像素刚好需要执行相同的、可以拼成一个大矩阵的数学运算。一旦有条件分支（`if-else`），矩阵计算就会直接崩溃。
</details>

### 总结：MMA 在图形渲染中的真正用途

综上所述，图形管线中传统的 Vertex/Pixel/Compute Shader 中的数学运算，仍然是由**标准 ALU (处理 FMA) + SFU (处理 exp/pow)** 来完成的。

但这并不意味着 MMA 在渲染中没用。现代 GPU 巧妙地将 MMA 应用在了**后处理和特定的计算阶段**，在这些阶段，数据可以被完美组织成大矩阵：

1.  **AI 超分辨率提升 (DLSS / XeSS)：** 渲染出低分辨率画面后，将整个画面作为 Tensor（张量/多维数组）送入神经网络。此时，Tensor Cores (MMA) 全力开动，进行疯狂的矩阵乘法，预测并生成高分辨率图像。
2.  **光线追踪降噪 (Ray Tracing Denoising)：** 类似于 DLSS，使用基于 AI 的降噪器，将包含噪点的 G-Buffer 数据转换为矩阵交由 MMA 处理。
