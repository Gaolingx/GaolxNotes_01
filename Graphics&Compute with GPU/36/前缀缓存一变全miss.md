# 前缀缓存一变全miss

### 🧑‍💻 User

vLLM/SGLang Automatic Prefix Cache是不是只要prefill时候一个token变了后面全miss？

---

### 🤖 Assistant

**是的，完全正确。**

在 `vLLM` 和 `SGLang` 的 Automatic Prefix Cache (APC) 机制中，**只要 prefill 阶段从左到右依次匹配时遇到第一个不同的 `token`，从这个变动的 `token` 开始，后面的所有内容都会发生 Cache Miss（缓存未命中），必须重新进行计算。**

这是由**前缀匹配（Prefix Matching）**的底层逻辑和**因果语言模型（Causal Language Model）**的数学性质共同决定的。

---

### 为什么会这样？（原理解析）

#### 1. 因果注意力机制（Causal Attention）的限制
在 GPT、LLaMA 等自回归模型中，每一个 `token` 的 KV Cache（Key 和 Value 向量）都包含了它前面所有 `token` 的上下文信息。

假设我们有两个序列：
*   **序列 A (已缓存):** $t_1, t_2, t_3, t_4, t_5$
*   **序列 B (新请求):** $t_1, t_2, t_{new}, t_4, t_5$

当计算 $t_4$ 的注意力时：
$$ Attention(Q_4, K_{\le 4}, V_{\le 4}) $$

在序列 A 中，$t_4$ 注意的是 $[t_1, t_2, t_3, t_4]$。
在序列 B 中，$t_4$ 注意的是 $[t_1, t_2, t_{new}, t_4]$。

因为上下文变了，所以序列 B 中 $t_4$ 和 $t_5$ 的隐藏状态（Hidden States）以及生成的 KV 向量，在数学上与序列 A 中的**完全不同**。因此，即使字面上的文本（$t_4, t_5$）一模一样，底层的数据也变了，无法复用之前的缓存。

#### 2. 基数树（Radix Tree）的匹配逻辑
`SGLang` 提出了 RadixAttention（随后被 `vLLM` 借鉴吸收），它使用**基数树**来管理 KV Cache。
*   匹配过程是从树的根节点（即 prompt 的开头）开始的。
*   一旦遇到一个不匹配的 `token`（即树的分支发生了分叉），前缀匹配就立刻中断。
*   系统会将匹配成功的部分作为 Cache Hit 提取出来，剩下的部分全部作为新的输入扔进 GPU 进行 prefill 计算。

<details>
<summary><b>💡 核心启示：如何优化 Prompt 以提高 Cache 命中率？ (点击展开)</b></summary>

既然“一旦改变，后面全 miss”，我们在实际编写 Prompt 时，必须遵循**“静态在前，动态在后”**的原则。

#### ❌ 错误示范（低命中率）
如果你把动态变量（如时间、用户名字、随机 ID）放在 Prompt 的开头：
```text
今天是 2023-10-27。（动态：明天这里就变了）
系统提示词：你是一个精通物理的 AI 助手，请遵循以下 1000 字的规则...（静态的巨长文本）
用户输入：请解释量子纠缠。（动态）
```
**结果：** 因为开头的日期每天/每次都在变，导致开头的 `token` 变了，后面那 1000 字的系统规则每次都会 **Cache Miss**，浪费大量算力。

#### ✅ 正确示范（高命中率）
将所有固定不变的内容顶格放在最前面，把动态内容全挪到最后：
```text
系统提示词：你是一个精通物理的 AI 助手，请遵循以下 1000 字的规则...（静态的巨长文本，永远 Hit）
今天是 2023-10-27。（动态）
用户输入：请解释量子纠缠。（动态）
```
**结果：** 前面 1000 字的规则完美 **Cache Hit**，模型只需要 prefill 最后极短的日期和用户输入。

#### 典型的高效应用场景
1.  **System Prompt 缓存：** 将几千字的复杂 System Prompt 或 Few-shot examples 放在最前面，多个用户并发请求时共享这部分 KV Cache。
2.  **多轮对话（Multi-turn Chat）：** 多轮对话天然是不断在末尾追加 `token` 的（Append-only），前面的历史记录完美契合 Prefix Cache 的命中逻辑。
3.  **RAG（检索增强生成）：** 结构应为：`[固定指令] + [检索到的长文档] + [用户查询]`。如果检索的文档在多个问题中被重复使用，也能被有效缓存。

<details>
<summary><b>🛠️ 进阶：有办法只替换中间的 Token 而不丢失后面的缓存吗？ (点击展开)</b></summary>

目前在标准的 `vLLM` / `SGLang` 的 Causal LM 架构下：**做不到。**

但是，学术界和工业界有一些前沿研究在尝试解决“中间内容修改导致缓存失效”的问题：

1.  **Prompt Compression / KV Cache Editing:** 一些非官方的魔改模型尝试通过特殊操作只更新局部 KV，但这通常会带来精度损失（因为破坏了严格的因果注意力机制）。
2.  **Cascade/Hierarchical Caching:** 针对特定场景（比如中间插入的文本是相互独立的模块），通过修改 Attention Mask 强制让后面的 token 不去注意中间变动的 token。这需要深度定制模型和推理引擎，目前 `vLLM` 的标准功能不支持。
3.  **模块化 Prompt:** 如果你有多个长文档 A 和 B，用户有时问 A，有时问 B。不要写成 `[System] + [A] + [B] + [Query]` 然后频繁调换 A 和 B 的顺序。固定它们的绝对顺序，或者针对不同的组合建立独立的分支。

</details>
