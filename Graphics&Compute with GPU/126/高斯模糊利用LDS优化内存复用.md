# 高斯模糊利用LDS优化内存复用

### 🧑‍💻 User

为什么高斯模糊可以用lds优化？

---

### 🤖 Assistant

高斯模糊（Gaussian Blur）之所以能够（且非常适合）使用 **LDS（Local Data Share，局部数据共享，NVIDIA 体系中称为 Shared Memory / 共享内存）** 进行优化，核心原因在于高斯模糊的计算特性导致了**极高的全局内存重复读取率**，而 LDS 正好可以用来消除这种内存带宽瓶颈。

以下是详细的原因和优化原理：

### 1. 核心痛点：相邻像素的重复读取 (Memory Bound)

高斯模糊本质上是一个卷积（Convolution）操作。为了计算某一个像素模糊后的值，线程需要读取该像素及其周围半径为 $R$ 的所有相邻像素。

假设我们使用分离的高斯模糊（先进行水平 1D 模糊，再进行垂直 1D 模糊），卷积核大小为 $K = 2R + 1$。
* 在未优化的朴素实现中，**每个线程**计算一个目标像素，需要从缓慢的全局内存（Global Memory / VRAM）中读取 $K$ 个像素。
* 对于相邻的两个线程（例如计算像素 `[x, y]` 和 `[x+1, y]`），它们需要读取的 $K$ 个像素中有 $K-1$ 个是**完全重合的**。

这就导致了海量的冗余全局内存读取。GPU 的计算核心很快就会被闲置，只能干等着缓慢的内存传输，程序成为典型的 **内存带宽瓶颈 (Memory Bound)**。

### 2. LDS 的优势：高速、低延迟的片上缓存

LDS 是位于 GPU 计算单元（Compute Unit / Streaming Multiprocessor）内部的一块极高速、低延迟的内存，由同一个工作组（Workgroup / Thread Block）内的所有线程共享。

将 LDS 引入高斯模糊，核心思想是：**“按块读取，多次复用” (Block Tiling)**。

### 3. LDS 优化高斯模糊的具体机制

使用 LDS 优化的流程如下：

1.  **协作加载 (Cooperative Fetch)：** 工作组内的所有线程协同工作，每个人负责从全局内存中读取少量的像素数据，并将其写入 LDS。
2.  **处理“裙边” (Halo / Apron)：** 由于边缘的像素在模糊时需要用到块外的像素，所以加载到 LDS 中的数据块（Tile）必须比实际计算输出的块稍微大一圈，这多出来的一圈被称为裙边（大小通常等于模糊半径 $R$）。
3.  **同步 (Barrier Synchronization)：** 所有线程等待（例如调用 `barrier()` 或 `__syncthreads()`），确保整个数据块和裙边都已经完整地加载到 LDS 中。
4.  **高速计算 (Compute from LDS)：** 线程不再访问全局内存，而是直接从极高速的 LDS 中读取所需的 $K$ 个像素进行加权求和，最后将结果写回全局内存（或寄存器进入下一轮计算）。

通过这种方式，原本对全局内存的 $O(K \times N)$ 次读取，被大幅降低到了接近 $O(N)$ 次读取（$N$ 为像素总数），彻底释放了内存带宽的压力。

<summary><b>展开查看：高斯模糊优化的“终极形态” (LDS + 分离卷积)</b></summary>

在实际的高性能图形学或计算摄影中，高斯模糊通常会结合**二维可分离性 (Separability)** 和 **LDS** 一起优化。

**二维高斯核的数学拆分：**
二维的高斯分布矩阵可以拆分为一个垂直一维向量和一个水平一维向量的外积：
$$ G_{2D}(x, y) = G_{1D}(x) * G_{1D}(y) $$

**优化步骤：**
1.  **水平 Pass：** 工作组按行划分。线程协同将一行加上左右“裙边”的图像加载到 LDS 中。从 LDS 中读取数据进行 1D 水平模糊，结果写入中间全局内存（或暂存）。
    *   *复杂度从 $O(K^2)$ 降到了 $O(K)$。*
2.  **垂直 Pass：** 工作组按列划分（通常需要对图像做一次内存转置以保证合并访存，或者精细设计 LDS 的步长）。将列数据及上下“裙边”加载到 LDS，进行 1D 垂直模糊。

如果不使用 LDS，即使做了 1D 分离，水平 Pass 依然会被全局内存的重复读取拖垮。**LDS + 1D 分离卷积** 是目前 GPU 上实现大半径模糊的最标准且最高效的方案。
</details>

<details>
<summary><b>展开查看：LDS 优化中的注意事项 (Bank Conflict)</b></summary>

虽然 LDS 速度极快，但在编写 Shader / Compute Kernel 时需要注意**存储体冲突 (Bank Conflict)**。
* LDS 通常被划分为多个 Bank（例如 32 个）。
* 如果一个工作组（Wavefront / Warp）内的多个线程同时访问同一个 Bank 中的不同地址，LDS 的访问就会被串行化，导致性能下降。
* 在高斯模糊中，由于相邻线程通常访问相邻的像素，一般能较好地避免严重的 Bank Conflict，但在处理垂直模糊（按列访问）时，如果没有巧妙设置 LDS 数组的维度（例如 Padding），就极其容易触发 Bank Conflict。
</details>

### 总结

高斯模糊能用 LDS 优化，是因为其**局部性极强，相邻像素的复用率极高**。LDS 充当了用户可控的高速 L1 缓存，通过**“一次加载，多次读取”**的策略，大幅度砍掉了昂贵的全局内存带宽消耗，从而让 GPU 发挥出最大的浮点运算性能。
